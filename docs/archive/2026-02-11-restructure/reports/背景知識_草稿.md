# 第二章 背景知識

## 2.1 機器人作業系統 ROS2 Humble

### 2.1.1 ROS2 簡介

ROS2（Robot Operating System 2）是目前機器人領域最廣泛使用的開源軟體框架，由 Open Robotics 組織維護開發。相較於前一代 ROS1，ROS2 在以下方面進行了重大改進：

1. 即時性支援：ROS1 不支援即時作業系統，而 ROS2 原生支援 RTOS，可滿足對延遲敏感的應用場景。
2. 多機器人協作：ROS1 需額外配置才能實現多機器人通訊，ROS2 則原生支援分散式架構。
3. 安全性：ROS2 採用 DDS（Data Distribution Service）中介軟體，提供加密與存取控制機制。
4. 平台支援：ROS1 主要支援 Linux，ROS2 則可運行於 Linux、Windows 與 macOS。
5. Python 版本：ROS2 採用 Python 3，告別 ROS1 的 Python 2。

本專題選擇 ROS2 Humble Hawksbill（2022 年 5 月發布）作為開發平台，這是目前最新的 LTS（Long Term Support）版本，官方支援至 2027 年。

### 2.1.2 ROS2 核心概念

ROS2 採用發布-訂閱（Publish-Subscribe）架構進行節點間通訊，以下為本專題使用的核心概念：

1. 節點（Node）

節點是 ROS2 的基本執行單元，代表一個獨立的程式或功能模組。本專題使用的主要節點包括 go2_driver_node（Unitree Go2 機器狗驅動程式）、snapshot_service（相機截圖服務）、以及 rosbridge_websocket（WebSocket 通訊橋接器）。

2. 話題（Topic）

話題是節點間傳遞資料的通道，採用非同步通訊模式。本專題使用的關鍵話題包括：/camera/image_raw 用於相機原始影像傳輸，採用 sensor_msgs/Image 訊息類型；/cmd_vel 用於機器狗速度控制，採用 geometry_msgs/Twist 類型；/scan 用於 LiDAR 雷射掃描資料，採用 sensor_msgs/LaserScan 類型；/odom 用於里程計資料，採用 nav_msgs/Odometry 類型。

3. 服務（Service）

服務採用請求-回應（Request-Response）模式，適用於需要確認執行結果的操作。本專題自行開發的服務包括：/capture_snapshot 用於擷取當前相機畫面並儲存為 JPEG；/move_for_duration 用於在指定時間內以指定速度移動；/stop_movement 用於緊急停止所有移動。

4. 動作（Action）

動作是服務的延伸，適用於長時間執行的任務，支援進度回報與取消功能。Nav2 導航框架即透過動作介面控制機器人移動至目標點。

### 2.1.3 本專題使用的 ROS2 套件

本專題使用的主要套件包括：go2_robot_sdk 為自行開發的 Unitree Go2 驅動程式；rosbridge_server 用於 WebSocket 橋接，供 MCP 連線使用；slam_toolbox 用於即時 SLAM 建圖；nav2 用於自主導航（本次 Demo 保留未啟用）；tf2 用於座標轉換框架。

### 2.1.4 為何選擇 ROS2

本專題選擇 ROS2 而非其他機器人框架（如 YARP、MOOS-IvP）的原因如下：首先，ROS2 是目前機器人產業最廣泛採用的業界標準，Unitree Go2 原廠即提供 ROS2 支援；其次，ROS2 擁有豐富生態系，包含完整的 SLAM、導航、視覺處理套件；第三，開源的 ros-mcp-server 專案提供 ROS2 與 MCP 協定的橋接，便於整合；最後，ROS2 具備跨平台支援，可在 Ubuntu VM、Windows WSL2 等多種環境運行。

## 2.2 Unitree Go2 四足機器人平台

### 2.2.1 Go2 簡介

Unitree Go2 是由中國宇樹科技（Unitree Robotics）於 2023 年發布的消費級四足機器人，定位為「人人可負擔的機器狗」。相較於 Boston Dynamics Spot（售價約 75,000 美元），Go2 以約 1,600 美元的價格提供類似的基礎功能，大幅降低四足機器人的研究門檻。

### 2.2.2 硬體規格

Go2 提供三種版本：Go2 Air、Go2 Pro 與 Go2 EDU。

Go2 Air 售價約 1,600 美元，不配備 LiDAR 與 4G 模組，僅提供基礎 APP 控制，連線方式為 Wi-Fi。

Go2 Pro 售價約 2,800 美元，配備 Hesai XT16 LiDAR 與 4G 模組，同樣僅提供基礎 APP 控制，連線方式為 Wi-Fi。本專題使用此版本。

Go2 EDU 售價約 8,500 美元，除了配備 LiDAR 與 4G 模組外，還搭載 NVIDIA Jetson Orin NX（8GB）處理器，提供完整 ROS2 SDK 支援，連線方式支援 Wi-Fi 與 Ethernet。

備註：Air 與 Pro 版處理器規格官方未公開揭露，僅 EDU 版明確標示搭載 Jetson Orin NX。

### 2.2.3 感測器配置

Go2 內建多種感測器，本專題主要使用以下三種：

1. LiDAR 雷射雷達

型號為 Hesai XT16（16 線），掃描範圍為 360 度水平與正負 15 度垂直，掃描頻率在有線連線時為 10 Hz，在 Wi-Fi 連線時為 1 至 2 Hz，測距範圍為 0.3 公尺至 150 公尺，用途為環境建圖（SLAM）與障礙物偵測。

2. 前置 RGB 相機

解析度為 1920 乘 1080（Full HD），視野角超過 120 度（廣角鏡頭），幀率為 30 FPS，用途為環境影像擷取與物件偵測。

重要發現：Go2 採用廣角鏡頭（視野角超過 120 度），而多數深度估計模型（如 Depth Anything V2）以標準鏡頭（60 至 90 度）資料訓練，導致深度估計結果偏差。本專題透過實測校正解決此問題（詳見 2.5.1 節）。

3. IMU 慣性測量單元

包含三軸加速度計與三軸陀螺儀，用途為姿態估計與里程計輔助。

### 2.2.4 通訊架構

Go2 提供兩種主要連線方式：Wi-Fi 連線採用 WebRTC 協定，優點為無線便利，缺點為感測器頻率較低（約 1 至 2 Hz）；Ethernet 連線採用 CycloneDDS 協定，優點為低延遲與高頻率，缺點為僅 EDU 版支援。

本專題受限於硬體版本（非 EDU），僅能使用 Wi-Fi 加 WebRTC 連線。此限制影響 LiDAR 資料頻率，因此本專題採用「語意導航」策略（詳見 2.7 節）以適應低頻感測器資料。

### 2.2.5 座標系統

Go2 的 URDF（Unified Robot Description Format）定義了以下座標框架（Frame）：base_link 為機器狗本體中心，作為運動控制參考點；front_camera 為前置相機，注意此框架名稱並非一般常見的 camera_link；lidar_link 為 LiDAR 感測器，為點雲資料來源；imu_link 為 IMU 感測器，為姿態資料來源。

注意：Go2 預設 URDF 的相機座標框架名稱為 front_camera，而非一般常見的 camera_link。進行 TF 座標轉換時需特別注意。

## 2.3 SLAM 與自主導航概述

### 2.3.1 SLAM 技術簡介

SLAM（Simultaneous Localization and Mapping，同步定位與建圖）是移動機器人領域的核心技術，使機器人能在未知環境中同時建構地圖並定位自身位置。

SLAM 的基本原理分為六個步驟：首先從感測器（LiDAR 或相機）擷取資料，接著進行特徵提取，然後進行資料關聯（匹配已知特徵），再進行位姿估計（計算機器人位置），之後進行地圖更新（加入新觀測），最後進行迴圈閉合（修正累積誤差）。

### 2.3.2 slam_toolbox 套件

本專題使用 slam_toolbox 進行即時建圖，這是 ROS2 生態系中最成熟的 2D SLAM 解決方案。其主要特點包括：採用圖優化（Graph-based SLAM）架構、支援線上即時建圖與離線最佳化、可處理大規模環境（數百平方公尺）、產出 2D 佔據格點地圖（Occupancy Grid Map）。

本專題已成功建構室內環境地圖，地圖檔案格式包括 phase1.yaml（地圖元資料，含解析度與原點座標）與 phase1.pgm（地圖影像，灰階圖，黑色代表障礙物、白色代表通道、灰色代表未知區域）。

### 2.3.3 Nav2 導航框架

Nav2（Navigation2）是 ROS2 的標準自主導航解決方案，提供完整的路徑規劃與控制功能。

Nav2 核心模組包括：全域規劃器（Global Planner）負責計算起點到終點的最佳路徑；局部規劃器（Local Planner）負責即時避開動態障礙物；成本地圖（Costmap）負責將地圖資訊轉換為導航成本；行為樹（Behavior Tree）負責控制導航狀態與異常處理。

### 2.3.4 本專題的導航策略

本專題已完成 Nav2 整合驗證，但在 1/7 Demo 中選擇暫不啟用，原因如下：

首先是 LiDAR 頻率過低，Wi-Fi 連線僅 1 至 2 Hz，而 Nav2 建議 10 Hz 以上；其次是座標轉換複雜，需精確的相機與 LiDAR 外參校正；最後是開發時程緊迫，距離 Demo 僅剩 23 天。

替代方案為語意導航（詳見 2.7 節）。本專題採用「深度估計加 LLM 決策」的語意導航策略，避開傳統 Nav2 對高頻感測器的依賴，同時保留 Nav2 作為未來升級選項。

## 2.4 Model Context Protocol（MCP）整合架構

### 2.4.1 MCP 協定簡介

Model Context Protocol（MCP）是 Anthropic 公司於 2024 年底發布的開放標準協定，旨在讓大型語言模型（LLM）能夠安全、標準化地與外部工具和系統進行互動。

MCP 的核心理念是「讓 AI 不只是會說話，還能動手做事」。傳統上，LLM 僅能進行文字對話。透過 MCP 協定，LLM 可以呼叫外部 API、讀寫檔案系統、操控機器人、以及查詢資料庫。

### 2.4.2 MCP 協定架構

MCP 採用 Client-Server 架構，定義了以下核心概念：

1. MCP Host（主機）

執行 LLM 的應用程式，例如 Claude Desktop、Kilo Code（VS Code 插件）、或自行開發的 Python 應用。

2. MCP Server（伺服器）

提供工具功能的服務端程式，將外部系統的能力暴露給 LLM。本專題使用開源的 ros-mcp-server。

3. MCP Protocol（協定）

定義 Host 與 Server 之間的通訊格式，支援 stdio（標準輸入輸出，用於本機通訊）與 HTTP/SSE（用於網路通訊）。

MCP 架構流程為：MCP Host（Kilo Code）內執行 LLM（Claude 3.5 Sonnet），透過 MCP Protocol 與 ros-mcp-server 通訊，ros-mcp-server 再透過 WebSocket 連接至 rosbridge（Port 9090），最後透過 ROS2 Topics 與 Services 控制 Unitree Go2。

### 2.4.3 ros-mcp-server 工具集

本專題使用的 ros-mcp-server 提供 58 個 MCP 工具，涵蓋 ROS2 的所有核心功能：Topic 操作 12 個，包含 get_topics、subscribe_once、publish_once 等；Service 操作 8 個，包含 get_services、call_service 等；Action 操作 6 個，包含 send_action_goal（可控制 Nav2）；系統資訊 15 個，包含 get_nodes、get_ros_time 等；訊息定義 10 個，包含 get_message_definition 等；其他 7 個，包含 ping_rosbridge、analyze_image 等。

關鍵工具介紹：call_service 用於呼叫 ROS2 服務，本專題用於觸發拍照與移動控制；subscribe_once 用於訂閱單筆 Topic 資料，用於讀取感測器狀態；send_action_goal 用於發送 Nav2 導航目標點（未來擴充）。

### 2.4.4 rosbridge 通訊橋接

rosbridge 是 ROS2 的 WebSocket 橋接器，讓非 ROS 程式（如 JavaScript、Python）可透過 WebSocket 與 ROS2 通訊。技術規格為：通訊協定採用 WebSocket，預設 Port 為 9090，資料格式為 JSON，支援 Topic、Service、Action 全覆蓋。

### 2.4.5 本專題的 MCP 整合架構

本專題的完整資料流如下：使用者下達自然語言指令（如「往前走，有障礙物就繞開」），Kilo Code（MCP Host）內的 Claude 3.5 Sonnet 或 Nova 2 理解指令並規劃行動策略，透過 MCP Protocol 將指令傳送至 ros-mcp-server，ros-mcp-server 轉譯 LLM 指令為 ROS2 呼叫，透過 WebSocket 連接至 Ubuntu VM 上的 rosbridge_server（Port 9090），再透過 ROS2 服務（如 /capture_snapshot、/move_for_duration、/cmd_vel）控制 go2_robot_sdk Driver，最終透過 WebRTC（Wi-Fi）控制 Unitree Go2 機器狗實體。

### 2.4.6 創新亮點

本專題為國內首例將 MCP 協定應用於 ROS2 機器人控制，實現「自然語言到機器人動作」的直接轉換。

MCP 架構帶來的優勢包括：開發模式從撰寫程式碼定義行為轉變為 LLM 理解語意自動決策；指令格式從結構化 API 呼叫轉變為自然語言；錯誤處理從需預先定義所有情境轉變為 LLM 可處理模糊指令；擴充性從需修改程式碼轉變為更新 Prompt 即可；開發時間從數週至數月縮短為數天。

## 2.5 視覺感知系統

### 2.5.1 深度估計：Depth Anything V2 Metric

2.5.1.1 技術背景

深度估計（Depth Estimation）是從 2D 影像推斷場景 3D 深度資訊的技術。傳統方法需要雙目相機或 LiDAR 等深度感測器，而基於深度學習的單目深度估計僅需單張 RGB 影像即可產生深度圖。

Depth Anything V2 是 2024 年由香港大學與 TikTok 聯合發表的單目深度估計模型，在多項基準測試中達到 SOTA（State-of-the-Art）水準。該模型提供 Relative（相對深度）與 Metric（絕對距離）兩種版本。

2.5.1.2 模型選擇

本專題選擇 Depth Anything V2 Metric Large 版本（以下簡稱 DA2-Metric），主要考量如下：

DA2-Relative-Small 輸出相對深度（0 至 1），VRAM 需求約 0.5 GB，推論速度約 50ms，適用於藝術應用。DA2-Relative-Large 輸出相對深度（0 至 1），VRAM 需求約 1.5 GB，推論速度約 200ms，適用於研究用途。DA2-Metric-Large 輸出絕對深度（公尺），VRAM 需求約 1.5 GB，推論速度約 230ms，適用於機器人應用，為本專題採用之版本。

選擇 Metric 版本的原因：機器人避障需要「前方 0.8 公尺有障礙物」這樣的絕對距離資訊，而非「深度值 0.7」這樣的相對比例。Metric 版本使用 Hypersim 室內資料集訓練，適合居家環境。

2.5.1.3 Go2 廣角鏡頭校正問題

問題發現：在實測過程中，我們發現 DA2-Metric 持續高估距離約 50 至 60%。例如實際距離 0.6 公尺時，DA2-Metric 輸出 1.16 公尺（高估 93%）；實際距離 1.0 公尺時，DA2-Metric 輸出 1.51 公尺（高估 51%）。

原因分析：Go2 採用廣角鏡頭（視野角超過 120 度），而 DA2-Metric 訓練資料多來自標準鏡頭（視野角 60 至 90 度）。廣角鏡頭的透視變形導致模型誤判距離。

解決方案：實拍校正。我們透過 Go2 相機在多個已知距離拍攝校正圖，計算校正係數。實際距離 0.6 公尺時原始輸出 1.16 公尺，校正係數為 0.52；實際距離 1.0 公尺時原始輸出 1.51 公尺，校正係數為 0.66；平均校正係數為 0.60。

校正結果驗證：測試距離 15 公分時校正後輸出 14 公分，誤差負 7%，通過；測試距離 30 公分時校正後輸出 28 公分，誤差負 7%，通過；測試距離 45 公分時校正後輸出 41 公分，誤差負 9%，通過；測試距離 60 公分時校正後輸出 70 公分，誤差正 17%，通過；測試距離 1.0 公尺時校正後輸出 91 公分，誤差負 9%，通過。

結論：校正後在 15 公分至 1 公尺範圍內誤差小於 20%，完全滿足室內避障需求。

2.5.1.4 Perception API 服務

本專題在 GPU 伺服器上部署 Perception API，提供深度估計與避障建議服務。技術規格為：框架採用 FastAPI，Port 為 8050，GPU 為 NVIDIA Quadro RTX 8000（48GB），推論時間約 230 至 330ms（640 乘 480 輸入），VRAM 使用約 1.5 GB。

API 端點為 POST /perceive，輸入為 JPEG 圖片，輸出為 JSON（包含三區距離與避障建議）。回應範例包含 left_m（左區距離，如 1.31 公尺）、center_m（中區距離，如 0.73 公尺）、right_m（右區距離，如 1.21 公尺）、front_obstacle_m（正前方最近障礙物距離，如 0.54 公尺）、suggestion（避障建議，如「正前方 0.5 公尺有障礙，建議向左繞行，左側 1.3 公尺較空曠」）、inference_ms（推論時間，如 326.4 毫秒）。

2.5.1.5 區域分析與避障決策

Perception API 將影像分為左、中、右三個區域進行深度分析，並計算中央下半部最近障礙物距離（front_obstacle_m）。

避障決策邏輯如下：若 front 大於 1.0 公尺，則前方暢通，可安全前進；若 front 小於 1.0 公尺且 right 大於 left，則建議向右繞行；若 front 小於 1.0 公尺且 left 大於 right，則建議向左繞行；若 left 小於 0.5 公尺且 right 小於 0.5 公尺，則三面受阻，建議後退。

### 2.5.2 物件偵測：YOLO-World

2.5.2.1 YOLO 技術演進

YOLO（You Only Look Once）是目前最廣泛使用的即時物件偵測演算法，由 Joseph Redmon 於 2015 年首次提出。相較於傳統的 R-CNN 系列演算法，YOLO 採用單階段（One-stage）架構，將物件偵測視為端對端的迴歸問題，大幅提升偵測速度。

YOLO 版本演進如下：YOLOv1 於 2015 年首創單階段偵測；YOLOv3 於 2018 年引入多尺度偵測與殘差網路；YOLOv5 於 2020 年以 PyTorch 實現並易於部署；YOLOv8 於 2023 年達到更高精度與更快速度；YOLO-World 於 2024 年引入開放詞彙偵測。

2.5.2.2 YOLO-World 開放詞彙偵測

本專題採用 YOLO-World-L（Large 版本），這是 2024 年發表的開放詞彙物件偵測模型。

傳統 YOLO 與 YOLO-World 的差異：傳統 YOLO 偵測類別固定為 80 類（COCO 資料集），擴充方式需重新訓練，適用於通用物件場景；YOLO-World 可偵測任意文字描述的類別，擴充方式僅需修改文字提示，適用於自定義目標物場景。

開放詞彙的優勢：傳統 YOLO 只能偵測 COCO 資料集中的 80 類物件（如 person、car、chair）。若要偵測「藥盒」或「眼鏡盒」等非標準類別，需要收集大量標註資料並重新訓練模型。YOLO-World 透過視覺-語言預訓練，可接受任意文字描述作為偵測目標。

2.5.2.3 本專題偵測類別

根據「居家尋物」應用場景，本專題設定以下偵測類別：水瓶（water bottle）用於「幫我找水」情境；眼鏡（glasses）用於「眼鏡在哪裡」情境；藥盒（medicine box）用於「找我的藥」情境；遙控器（remote control）用於「電視遙控器不見了」情境；手機（phone）用於「我的手機呢」情境；鑰匙（keys）用於「鑰匙放哪了」情境。

障礙物類別（用於避障）：椅子（chair）、桌子（table）、紙箱（box）、人（person），皆用於避免碰撞與安全考量。

2.5.2.4 技術規格與性能

模型為 YOLO-World-L（Large 版本），輸入解析度為 640 乘 640，與 Perception API 一致。推論時間約 150ms（測試環境為 RTX 8000，batch 為 1）。信心度門檻設定為 0.7。mAP@0.5 為 52.3（官方數據，LVIS 資料集）。

輸出格式為 JSON，包含 detections 陣列，每個偵測結果包含 class（類別）、confidence（信心度）、bbox（邊界框座標）與 center（中心點座標）。

2.5.2.5 整合方式

YOLO-World 整合於 Perception API，與 DA2 深度估計共用同一 GPU 伺服器（Port 8050）。兩個模型分別進行物件偵測與深度估計，結果合併後輸出物件加距離的綜合資訊。合併輸出範例：「偵測到水瓶，位置在畫面右側，距離 1.2 公尺，建議向右轉約 30 度後前進。」

## 2.6 生成式 AI 與 LLM Agent

### 2.6.1 大型語言模型（LLM）簡介

大型語言模型（Large Language Model, LLM）是基於 Transformer 架構的深度學習模型，透過海量文字資料訓練，具備理解與生成自然語言的能力。

LLM 的核心能力包括：語意理解，能理解使用者意圖，例如將「幫我找眼鏡」理解為尋物任務；推理規劃，能分解任務步驟，例如規劃巡視路線與避障策略；上下文記憶，能記住對話歷史，例如追蹤尋物進度；程式碼生成，能生成可執行指令，例如產生 ROS2 服務呼叫。

### 2.6.2 視覺語言模型（VLM）

視覺語言模型（Vision Language Model, VLM）是結合視覺與語言能力的多模態 AI 模型，可同時理解圖像與文字。

VLM 的工作流程為：輸入圖像與文字提問，經由 Vision Encoder 提取圖像特徵，再經由 Language Model 結合文字與圖像特徵，最後輸出文字回應。

本專題的 VLM 應用場景：輸入 Go2 相機截圖加上「前方有什麼障礙物？」的問題，輸出「前方約 1 公尺處有一個紙箱，建議向右繞行」的回答。

### 2.6.3 本專題使用的 LLM

經過多款模型實測評估，本專題測試了超過 15 種視覺語言模型，包括 Gemini、Claude、Amazon Nova、Mistral、Qwen、GLM、Grok 等系列。評估標準包含回應時間、成本、障礙物識別準確度與工具呼叫能力。

測試結果顯示，多數模型存在以下問題：Gemini 3 Pro 回應時間過長（60 至 120 秒），不適合即時控制；Claude 4.5 Haiku 漏報障礙物，存在安全風險；GLM-4.6V 與 Gemini 2.5 系列無法正確讀取檔案；部分小型模型如 Ministral 14B 與 Nemotron Nano 12B 無法正確呼叫工具。

最終篩選出三個可用方案：

Amazon Nova 2 Lite 由 AWS 提供，回應時間約 3 秒，費用為免費，能正確識別障礙物並建議繞行方向，為目前開發測試的首選方案。小缺點為物體細節判斷偶有誤差。

Mistral Large 由 Mistral AI 提供，回應時間約 10 秒，費用約每次 0.23 美元，具備細膩的物體辨識能力且能正確執行避障決策，為性價比最高的方案。

Qwen3-VL-235B 由阿里雲提供，回應時間約 10 秒，費用約每次 0.32 美元，執行避障能力強，為備選方案。

最終選擇：Amazon Nova 2 Lite（開發測試用，免費）加上 Mistral Large（Demo 用，性價比最高）。

### 2.6.4 LLM Agent 架構

本專題將 LLM 設計為「機器狗大腦」，扮演決策核心角色。架構包含三個模組：感知模組（Perception）負責讀取影像、呼叫 API、理解場景；決策模組（Planning）負責分析環境、規劃路徑、避障決策；執行模組（Execution）負責呼叫服務、控制移動、回報結果。

LLM Agent 的角色設定（System Prompt）：「你是小狗，一隻貼心的 AI 機器狗。你的主人是一位年長的爺爺或奶奶。你說話要溫暖、簡短、易懂。你會幫主人找東西，並用簡單的話描述位置。可用工具包括 /capture_snapshot 拍照觀察環境、/move_for_duration 移動指定時間、/stop_movement 緊急停止。行為準則：收到尋物指令後先拍照觀察；分析環境後規劃移動策略；遇到障礙物自動繞行；找到目標物後溫暖地告訴主人位置。」

## 2.7 語意導航策略

本節為專題核心創新，詳細說明本專題與傳統機器人導航方案的差異。

### 2.7.1 傳統導航方案的挑戰

傳統機器人導航（如 Nav2）採用精確座標導航，需要以下技術支援：首先進行 LiDAR 掃描（頻率需 10 Hz 以上），然後進行 SLAM 建圖，接著進行全域路徑規劃，再進行局部避障，最後進行速度控制。

本專題面臨的困境：LiDAR 頻率過低，Wi-Fi 連線僅 1 至 2 Hz（Nav2 建議 10 Hz 以上），導致無法即時避障；座標轉換複雜，需相機內參與 LiDAR 外參校正，開發時間長；精度要求高，差幾公分可能導致碰撞，測試困難；動態障礙處理困難，傳統方案需快速反應，低頻感測器無法勝任。

### 2.7.2 語意導航：全新思維

本專題提出「語意導航」策略，核心理念是「給 LLM 吃熟食，而非生肉」。

什麼是「生肉」？傳統方案將原始感測器資料直接餵給決策模組：LiDAR 為 360 個浮點數（每個角度的距離值）；相機為 640 乘 480 乘 3 等於 921,600 個像素值；IMU 為 6 個軸的加速度與角速度。

什麼是「熟食」？本專題先將感測器資料處理為語意化摘要，再交給 LLM 決策：「正前方 0.8 公尺有障礙物」、「左側較空曠，建議左轉」、「偵測到水瓶，位於右前方 1.2 公尺」。

### 2.7.3 傳統方案與語意導航對比

指令格式：傳統座標導航使用 goto(x=1.5, y=0.3)，本專題語意導航使用「往前走，有障礙物就繞開」。

感測器需求：傳統座標導航需要 LiDAR 10 Hz 以上，本專題語意導航使用 Snapshot 加 DA2（容忍 1 至 2 Hz）。

座標轉換：傳統座標導航需精確的 TF 轉換，本專題語意導航不需要。

避障決策：傳統座標導航使用 Costmap 加 Local Planner，本專題語意導航使用 LLM 語意判斷。

開發複雜度：傳統座標導航高（需校正、調參），本專題語意導航低（更新 Prompt 即可）。

錯誤處理：傳統座標導航需預定義所有情境，本專題語意導航的 LLM 可處理未知情況。

適應性：傳統座標導航為固定邏輯，本專題語意導航可透過對話調整行為。

### 2.7.4 語意導航資料流

步驟 1：使用者下達指令「Go2，幫我找桌上的水。」

步驟 2：LLM 規劃任務「收到！我來幫您找水。先拍照觀察環境...」

步驟 3：拍照加深度估計，呼叫 /capture_snapshot 後上傳至 Perception API。

步驟 4：接收語意化摘要，包含 front_obstacle_m 為 0.8 公尺，suggestion 為「正前方 0.8 公尺有障礙，建議右轉」。

步驟 5：LLM 決策「前面有東西，我往右繞一下...」，呼叫 /move_for_duration 設定 angular_z 為負 0.5、duration 為 2.0。

步驟 6：確認安全後繼續，再次拍照確認前方暢通後前進。

步驟 7：找到目標「爺爺，水在前面桌上喔！」

### 2.7.5 為何不使用傳統座標轉換

本專題最初計畫採用傳統的座標轉換方案（將相機偵測的 2D 像素座標轉換為 3D 世界座標），但經評估後放棄。

傳統座標轉換流程：相機圖像（u, v）經相機內參矩陣 K 轉換為相機座標（Xc, Yc, Zc），再經相機-LiDAR 外參 T 轉換為 LiDAR 座標，接著經 LiDAR-base_link 轉換，再經 TF2 查詢，最終得到世界座標（X, Y, Z）。

實際困難：相機內參校正需棋盤格標定，Go2 廣角鏡頭變形嚴重；外參校正需精確對齊相機與 LiDAR，誤差會累積；時間同步要求相機與 LiDAR 時間戳需對齊；維護成本高，任何硬體調整都需重新校正。

語意導航的優勢：相機校正傳統方案需精確校正，語意導航不需要；座標轉換傳統方案需多級轉換，語意導航由 DA2 直接估計距離；誤差累積傳統方案嚴重，語意導航以校正係數補償；開發時間傳統方案需數週，語意導航僅需數天。

### 2.7.6 創新亮點總結

本專題的「語意導航」策略具備以下創新價值：

1. 降低硬體門檻：不需要 EDU 版本的高頻感測器。
2. 簡化開發流程：無需複雜的座標轉換與校正。
3. 提升適應性：LLM 可處理模糊指令與未知情境。
4. 人機互動友善：使用者以自然語言溝通，機器狗以人話回應。

核心創新：本專題證明，在低頻感測器環境下，透過「深度估計加 LLM 語意決策」可實現可靠的避障導航，為低成本機器人應用提供新的技術路線。

## 2.8 安全設計與系統限制

### 2.8.1 Safety Layer 設計

LLM 雖然具備強大的語意理解能力，但可能產生「幻覺」（Hallucination），輸出不合理的數值。為確保機器狗不會因 LLM 錯誤輸出而發生危險，本專題設計了多層安全機制。

安全層架構：LLM 原始輸出（可能的幻覺輸出，如速度 10.0、時間 5 秒）經過 Safety Layer 處理，包含速度限制（max_linear 為 0.5 m/s、max_angular 為 1.0 rad/s）、時間限制（max_duration 為 10 秒）、數值驗證（確保所有參數在合理範圍內），最終輸出安全的執行指令（限速後的安全輸出，如速度 0.5、時間 5 秒）。

速度限制設計：前進或後退最大速度為 0.5 m/s，設計考量為人能輕鬆追上並攔下；左右轉最大速度為 1.0 rad/s，設計考量為避免快速旋轉導致摔倒；單次移動時間最大為 10 秒，設計考量為限制最大移動距離。

緊急停止機制：透過 /stop_movement 服務，可在 0.27 秒內完全停止。

# 專案願景：老人與狗

> 「AI 不該只是螢幕裡的文字，而是能走進生活、回應眼神、記得名字的陪伴夥伴。」

---

## 1. 願景聲明

我們正在打造一個以 Unitree Go2 Pro 為載體的 **embodied AI 互動陪伴平台**。這不是一台遙控玩具，也不是一台功能堆砌的機器。這是一個能夠「看見」長者、「聽懂」需求、「記得」互動歷程，並在恰當時機給予回應的智能夥伴。

這個專案的核心理念是：

- **技術為人服務**：不是展示 AI 有多強，而是讓長者感受到被陪伴、被理解
- **互動優於功能**：一個會搖尾巴打招呼的狗，勝過一台功能齊全但冰冷的機器
- **展示即是價值**：每個功能都必須能在實際場景中「被看見」，而非停留在程式碼中

我們的目標是在 2026 年 4 月 13 日前，完成一套**可展示、可互動、可穩定運作**的系統，證明 embodied AI 不只是實驗室的玩具，而是能走進真實生活場景的陪伴夥伴。

## 2. 問題定義與使用情境

### 2.1 我們想解決的問題

現代社會面臨高齡化挑戰，許多長者獨居或與家人分隔兩地。現有的解決方案多集中在「監控」與「提醒」，缺乏真正的「陪伴感」。螢幕介面對長者來說往往是障礙，而非橋樑。

我們想問的是：

> 如果 AI 能「走過來」、能「看著你說話」、能「記得你的名字和喜好」，陪伴會是什麼樣子？

### 2.2 目標使用情境

本系統設計針對以下場景：

| 場景 | 使用者互動 | 系統回應 |
|------|-----------|---------|
| **迎接** | 長者走進客廳 | 機器狗抬頭、搖尾、主動靠近，用語音打招呼並詢問今日狀況 |
| **陪伴** | 長者坐在沙發上 | 機器狗安靜陪伴，偶爾抬頭觀察，在長者看過來時給予眼神回應 |
| **對話** | 長者呼喚機器狗名字 | 機器狗轉向聲源、靠近，啟動語音互動，理解簡單指令（播放音樂、詢問天氣） |
| **探索** | 長者對某處感興趣 | 機器狗跟隨視線或手勢方向，移動並回報該區域狀況 |
| **安全** | 長者跌倒或長時間靜止 | 系統偵測異常，嘗試語音確認，必要時通知監護人 |

### 2.3 目標使用者輪廓

- **主要使用者**：65 歲以上獨居長者，行動自如，渴望陪伴但不想成為負擔
- **次要使用者**：家屬與照護者，需要了解長者狀況與互動歷程
- **展示對象**：學術評審、潛在合作夥伴、對 embodied AI 有興趣的開發者

## 3. 核心價值

我們用五個「可」來定義這個專案的價值主張：

| 價值 | 定義 | 為何重要 |
|-----|------|---------|
| **可互動** | 不只是執行指令，而是能理解情境、主動回應、記得互動歷程 | 長者感受到的是「被理解」，而非「被控制」 |
| **可模組化** | 感知、決策、執行三層分離，每個模組可獨立開發、測試、替換 | 團隊可並行工作，新功能可快速整合 |
| **可擴充** | 架構預留擴充點，支援新增感知模組、決策策略、動作類型 | 從 P0 到 P2，系統能持續成長 |
| **可降級** | 當雲端服務、特定硬體失效時，系統能優雅降級，維持基本功能 | 陪伴不中斷，安全有保障 |
| **可展示** | 每個功能都有對應的測試網頁或 Demo 腳本，能獨立展示運作 | 評審與合作夥伴能「看見」價值，而非閱讀文件 |

## 4. 專案設計原則

以下原則指導所有技術決策與開發優先順序：

### 4.1 安全優先（Safety First）

- **硬體保護**：動作執行前有 Safety Guard 確認，避免機械傷害
- **隱私保護**：語音與影像資料本地處理優先，敏感資料不上雲
- **緊急機制**：任何異常狀況下，系統能進入安全模式並通知人員

### 4.2 控制權集中（Centralized Control）

- **單一仲裁點**：Interaction Executive v1 是所有指令的唯一出口
- **狀態透明**：系統狀態隨時可查，決策邏輯可追蹤
- **避免競態**：多個感知模組同時觸發時，有明確的優先順序與仲裁機制

### 4.3 雲邊協作（Edge-Cloud Collaboration）

- **邊緣即時**：Jetson Orin Nano 負責需要低延遲的任務（人臉追蹤、VAD、基礎動作控制）
- **雲端智能**：RTX 8000 負責重型推理（ASR、LLM、TTS、人臉識別 embedding）
- **無縫切換**：網路不穩時，邊緣能獨立運作基礎功能

### 4.4 漸進式交付（Incremental Delivery）

- **先求穩定，再求豐富**：P0 穩定運作前，不追求 P1/P2 功能
- **功能可獨立展示**：每個完成的功能都能單獨 Demo，不依賴其他未完成模組
- **文件同步更新**：功能交付時，對應的測試文件與展示腳本一併交付

## 5. 為何選擇這套硬體與架構

我們的技術選型不是追求「最新」或「最強」，而是追求**最適合這個願景的組合**。

### 5.1 硬體選型理由

| 元件 | 選擇 | 關鍵價值 |
|-----|------|---------|
| **載體** | Unitree Go2 Pro | 35+ 內建動作、開放 SDK、四足行走適應居家地形、社群資源豐富 |
| **感知** | Intel RealSense D435 | RGB-D 同步、ROS2 原生支援、Jetson 相容、價格合理 |
| **邊緣運算** | NVIDIA Jetson Orin Nano | 足夠處理即時視覺與 VAD、功耗低、ROS2 支援完善 |
| **雲端運算** | RTX 8000（遠端工作站）| 可運行 7B 級 LLM、同時處理多個重型推理任務 |

### 5.2 架構設計理由

```
┌─────────────────────────────────────────────────────────────┐
│                    雲端層（RTX 8000）                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────────────┐ │
│  │   ASR   │  │   LLM   │  │   TTS   │  │ Face Embedding  │ │
│  └─────────┘  └─────────┘  └─────────┘  └─────────────────┘ │
└────────────────────┬────────────────────────────────────────┘
                     │ WebSocket / ROS2
┌────────────────────┴────────────────────────────────────────┐
│                   邊緣層（Jetson Orin Nano）                  │
│  ┌─────────────────────────────────────────────────────────┐│
│  │           Interaction Executive v1（仲裁中心）            ││
│  └─────────────────────────────────────────────────────────┘│
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────────────┐ │
│  │  人臉追蹤  │  │   VAD   │  │  Safety │  │   Go2 Driver    │ │
│  │         │  │         │  │  Guard  │  │                 │ │
│  └─────────┘  └─────────┘  └─────────┘  └─────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

**為何這樣設計**：

1. **成本與效能平衡**：不是買最貴的 Jetson AGX，而是讓 RTX 8000 處理真正需要算力的任務
2. **延遲與品質兼顧**：人臉追蹤需要 30fps，必須在邊緣。LLM 推理可以容忍 500ms 延遲，適合上雲
3. **擴充彈性**：新增感知模組只需接入 Jetson，新增智能只需擴充雲端服務
4. **故障隔離**：雲端斷線時，Jetson 仍能維持基礎互動與安全保護

## 6. 產品感與展示感的定義

我們追求的「產品感」不是功能清單的堆砌，而是**一連串有意義的事件流**。

### 6.1 什麼是「產品感」

一台有產品感的機器狗，應該讓使用者感受到：

| 時刻 | 有產品感 | 沒有產品感 |
|-----|---------|-----------|
| **看見長者** | 抬頭、眼神接觸、耳朵轉動、慢慢靠近 | 機械式轉頭，直接衝過去 |
| **聽見呼喚** | 轉向聲源、停下當前動作、等待下一步 | 繼續執行當前任務，或同時回應多個指令 |
| **對話中** | 適時點頭、尾巴微擺、在長者停頓時給予眼神鼓勵 | 毫無反應，或過度活躍打斷對話 |
| **離開視線** | 安靜等待，或緩慢搜尋，不造成壓迫感 | 大聲叫喚、快速奔跑、讓長者感到被追趕 |

### 6.2 什麼是「展示感」

每個功能都必須能回答這個問題：

> 「我能在 30 秒內，讓評審『看見』這個功能的價值嗎？」

因此我們要求：

- **每個模組有獨立測試網頁**：輸入模組 IP，就能看見即時狀態與輸出
- **每個功能有 Demo 腳本**：一行指令啟動，自動執行標準展示流程
- **狀態可視化**：系統內部狀態（情緒、注意力、意圖）能即時呈現，讓觀眾理解「它在想什麼」

### 6.3 事件流範例

以下是我們追求的互動事件流：

```
[長者走進客廳]
    ↓
[視覺偵測到人形] → [情緒狀態：好奇] → [動作：抬頭、耳朵轉向]
    ↓
[確認為已知人物] → [情緒狀態：開心] → [動作：搖尾、慢慢靠近]
    ↓
[距離適當] → [語音：「早安，張先生，今天感覺怎麼樣？」]
    ↓
[長者回應] → [ASR 辨識] → [LLM 理解] → [TTS 回應 + 點頭動作]
    ↓
[對話結束] → [情緒狀態：平靜] → [動作：安靜趴下，維持視線接觸]
```

這不是「人臉偵測 + 語音辨識 + 語音合成」的功能列表，而是**一個完整的陪伴時刻**。

## 7. 成功樣貌與非目標

### 7.1 2025/04/13 的成功樣貌

在展示日當天，以下情境應該能穩定運作：

**必須完成（P0 - 核心互動）**：

- 長者走進展示區，機器狗能在 3 秒內偵測並主動靠近
- 長者呼喚機器狗，能啟動完整語音對話流程（ASR → LLM → TTS）
- 對話過程中，機器狗能維持適當的眼神接觸與身體語言
- 所有動作執行前有 Safety Guard 確認，無碰撞風險
- 當雲端服務中斷時，系統能優雅降級並通知使用者

**展示流程（20 分鐘）**：

| 時間 | 內容 | 展示價值 |
|-----|------|---------|
| 0-5 min | 系統介紹 + 硬體展示 | 讓評審理解架構設計思維 |
| 5-15 min | 現場互動 Demo | 長者（或示範者）與機器狗實際互動 |
| 15-20 min | Q&A + 未來展望 | 說明 P1/P2 規劃與商業化可能性 |

### 7.2 非目標（我們不追求的事）

為了在時間與資源限制下達成核心願景，以下項目**明確不在範圍內**：

| 項目 | 原因 | 替代方案 |
|-----|------|---------|
| **全自主導航** | 4/13 前無法穩定完成 SLAM + 導航 | 基於 D435 的基礎避障 + 簡單跟隨 |
| **多語言支援** | 資源限制，優先做好中文互動 | 架構預留擴充點，未來可換模型 |
| **遠端遙控** | 偏離「自主互動」核心價值 | 僅保留開發測試用的遙控功能 |
| **硬體改裝** | Go2 Pro 已足夠，改裝風險高 | 使用現有動作庫與 SDK |
| **多機協作** | 單機互動優先，多機為 P2 或之後 | 架構支援多機，但不展示 |
| **醫療級應用** | 需要嚴謹認證，超出專案範圍 | 明確定位為「陪伴」而非「醫療照護」 |

## 附錄：關鍵術語表

| 術語 | 定義 |
|-----|------|
| **Embodied AI** | 具有實體載體的人工智慧，能透過感測器感知環境，並透過執行器與環境互動 |
| **Interaction Executive** | 本系統的中央仲裁模組，負責協調各感知模組輸入，決策並分派動作指令 |
| **Safety Guard** | 硬體安全守護模組，所有動作執行前的最後一道檢查，確保不會造成人身或設備傷害 |
| **VAD** | Voice Activity Detection，語音活動偵測，用於判斷何時開始/結束錄音 |
| **ASR** | Automatic Speech Recognition，自動語音辨識，將語音轉為文字 |
| **TTS** | Text-to-Speech，文字轉語音，將文字回應轉為自然語音 |
| **P0 / P1 / P2** | 優先級分類，P0 為必須完成，P1 為應該完成，P2 為可以完成 |

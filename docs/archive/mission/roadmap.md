## 1. 路線圖閱讀指南

### 1.1 本文件目的

這份文件定義「何時交付、做到哪裡、如何驗收」。它將策略轉化為可執行的時間表，並明確各角色的交付責任。

### 1.2 閱讀方式

| 角色 | 建議閱讀順序 | 重點章節 |
|------|-------------|----------|
| 專案管理者 | 第2章 → 第5章 → 第8章 | 總體時程、里程碑、風險 |
| 開發者 | 第3章 → 第6章 → 第9章 | Phase目標、角色交付物、工作節奏 |
| 驗收方 | 第2章 → 第7章 | 硬截止、Demo驗收線 |

### 1.3 關鍵術語

- **硬底線 (Hard Deadline)**: 不可移動的最終期限
- **里程碑 (Milestone)**: 關鍵檢查點，必須達成才能進入下一階段
- **凍結 (Freeze)**: 該項目不再接受新功能，僅修復缺陷
- **攻守交換**: 開發重心從基礎建設轉向應用整合的切換點
- **P0/P1/P2**: 功能優先級，P0為必須完成，P1為應該完成，P2為有時間才做

## 3. Phase 切分與目標

### 3.1 Phase 1: 基礎建設期（現在 ~ 3/16）

**核心目標**: 建立穩定的技術基礎與團隊協作框架

#### 3.1.1 我（架構/後端）的任務

| 任務項目 | 目標 | 交付物 |
|----------|------|--------|
| 架構定稿 | 確定系統架構不會大改 | 更新後的架構圖、技術選型文件 |
| 模組契約 | 定義各模組間接口 | API Contract文件、消息格式定義 |
| Baseline Skeleton | 可運行的基礎框架 | 各模組的空殼實現、能編譯/運行 |
| AI大腦骨架 | LLM整合框架就緒 | 提示詞模板、上下文管理、工具調用框架 |
| 硬體環境 Runtime | Jetson + Go2 可連線 | 部署腳本、環境配置說明 |
| Backend接口 | 網站後端API定義 | RESTful API規格、WebSocket事件定義 |

#### 3.1.2 楊、鄔（感知/AI）的任務

| 任務項目 | 目標 | 交付物 |
|----------|------|--------|
| 手勢模型研究 | 了解可用方案與限制 | 技術調查報告、可行性分析 |
| 姿勢模型研究 | 評估即時性能 | 性能測試報告、模型選型建議 |
| 小Demo驗證 | 證明核心技術可行 | 可運行的最小驗證程式 |

#### 3.1.3 團隊（全員）的任務

| 任務項目 | 目標 | 交付物 |
|----------|------|--------|
| 展示網站初版 | 前端框架與基本布局 | 可運行的網站骨架 |
| Docs結構 | 文件系統化 | 標準化的docs目錄結構 |

### 3.2 Phase 2: 整合穩定期（3/16 ~ 4/13）

**核心目標**: 將各模組整合為完整系統，並達到展示級穩定度

| 角色 | Phase 1 角色 | Phase 2 角色 |

| 鄔 | 守：技術調查 | 攻：應用層開發 |

#### 3.2.2 Phase 2 重點任務

| 任務領域 | 具體內容 | 負責人 |
|----------|----------|--------|
| 系統整合 | 各模組串接、端到端測試 | 我 + 全員 |
| Demo穩定化 | 錯誤處理、重試機制、日誌完善 | 全員 |
| 降級保底 | 實作降級策略、確保Fail-Safe | 我 |

#### 1. 診斷並解決 Nav2 導航卡頓問題

**問題描述：**
- 使用 CLI 發送 `/goal_pose` 後機器狗會自己旋轉不停
- 導航過程卡頓，反應遲緩
- Foxglove 中找不到 2D Goal Pose 工具

**根本原因分析：**
```
網路路徑：Go2 (WiFi) ↔ Mac VM (橋接) ↔ Mac Host (有線) ↔ Windows Foxglove
總延遲：20-50ms ≈ Nav2 控制週期 (50ms) → 指令積壓導致卡頓
```

**解決方案設計：**
- **方案 A**（最推薦）：Windows 本地安裝 ROS2 Humble + RViz2（零延遲）
- **方案 B**（快速方案）：X11 轉發（Mac RViz2 → Windows 顯示，延遲 10-50ms）
- **方案 C**（不推薦）：純 Foxglove（僅適合監控，不適合控制）

**技術亮點：**
- 識別出 **多重網路跳轉** 為導航卡頓的主因
- 提出 **RViz2 + Foxglove 雙工具並用策略**
- RViz2 用於即時控制（零延遲）
- Foxglove 用於數據監控與團隊展示

## 💭 開發反思

### 今日學到的最重要的事

**網路延遲對即時控制的影響遠超想像**

原本以為 20-50ms 的延遲「應該沒什麼」，但實際測試發現：
- Nav2 的控制週期是 50ms（20Hz）
- 20-50ms 的延遲 → 每次指令都會錯過一個控制週期
- 累積下來導致嚴重的卡頓和旋轉不穩定

**解決方案的啟示：**
- 即時控制系統需要 **零延遲** 或 **極低延遲**（< 5ms）
- 遠端監控可以容忍延遲，但即時控制不行
- **選對工具** 比 **調參數** 更重要

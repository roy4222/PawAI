## 🔄 今日重大決策討論

### 提議內容
將專題架構從**「傳統 ROS2 導航（座標轉換 + COCO VLM + FSM）」**轉向**「MCP + LLM Agent 架構」**。

### 提議來源
參考 GitHub 專案：[robotmcp/ros-mcp-server](https://github.com/robotmcp/ros-mcp-server)

### 新架構概念
使用 **MCP (Model Context Protocol)** 讓 **生成式 AI（Claude/GPT-4o）** 直接控制 ROS2 機器人，取代原本計畫的座標轉換 + VLM + FSM 模組。

**理想願景：**
- 使用者語音/文字輸入「幫我找眼鏡」
- LLM 理解指令並自主規劃行動
- 透過 MCP 介面呼叫 ROS2 Topic/Service/Action
- 機器狗執行巡邏、避障、物體識別、導航等任務
- LLM Vision 識別物體並引導導航

## 附錄：技術名詞解釋

- **MCP (Model Context Protocol)**：Anthropic 提出的標準協定，讓 LLM 可與外部工具通訊
- **Action (ROS2)**：ROS2 的非同步通訊協定，用於長時間執行的任務（如導航）
- **Nav2**：ROS2 的導航框架
- **SLAM**：Simultaneous Localization and Mapping（同步定位與建圖）
- **FSM**：Finite State Machine（有限狀態機）
- **VLM**：Vision Language Model（視覺語言模型）
- **LangGraph**：LangChain 的多代理人協作框架
- **tf2**：ROS2 的座標轉換函式庫
- **Prompt Engineering**：調教 LLM 輸出行為的技術

### ⚠️ 關鍵技術挑戰與解決方案

#### 挑戰 1：ros-mcp-server 不支援 Action 協定

**原評估：** 這是致命缺陷，需要 2-3 週開發轉接層

**重新評估：**

**方案 A（混合控制）：** LLM 不直接控制 Nav2 Action，而是：
```python
# LLM 透過 MCP 發布簡單指令
LLM → MCP → ROS2 Topic `/cmd_vel` (直接速度控制)
                   `/goal_simple` (自訂簡化目標)

# 底層 FSM 監聽指令並轉換為 Nav2 Action
FSM 節點 → 訂閱 `/goal_simple` → 發送 NavigateToPose Action
```

**優勢：**
- ✅ 不需要修改 ros-mcp-server
- ✅ LLM 只需發布簡單指令
- ✅ 保留 Nav2 完整功能
- ✅ 開發時間：< 1 週（僅需寫 FSM 轉接節點）

**方案 B（完全繞過 Action）：** 使用速度控制代替導航
```python
# LLM 直接控制機器狗運動
LLM → MCP → `/cmd_vel` → Go2 運動

# 優點：
- 最簡單，無需 Nav2
- Demo 效果可能更直觀（AI 直接控制）

# 缺點：
- 放棄 Nav2 路徑規劃與避障
- 需要 LLM 自行判斷障礙物（依賴視覺）
```

**建議：採用方案 A（混合控制）**
- 保留 Nav2 優勢
- LLM 專注高層決策
- FSM 處理底層轉換
- **開發時間：5-7 天**

#### 挑戰 2：LLM 推論延遲（3-8s）

**原評估：** 延遲過高，無法即時控制

**重新評估：**

**Snapshot 策略完美解決：**
```
傳統架構（即時影片）：
Camera → 30 FPS Stream → LLM (每秒處理 30 張，延遲累積)

MCP 架構（定時截圖）：
Camera → 每 3 秒截圖 1 張 → LLM 分析 → 發送指令

時間軸範例：
T=0s:  截圖 → 上傳給 LLM
T=3s:  LLM 回應「前進 2 公尺」
T=3-8s: 機器狗執行前進（此時 LLM 可處理下一張圖）
T=8s:  再次截圖 → 上傳給 LLM
```

**優勢：**
- ✅ 延遲不再是問題（機器狗執行期間 LLM 同步處理）
- ✅ 頻寬需求極低（3 秒 1 張圖 vs 30 FPS 影片）
- ✅ LLM 有充足時間進行推理
- ✅ 符合會議建議（學長 John 提出）

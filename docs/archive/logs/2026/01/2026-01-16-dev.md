## ✅ 解法與處理

### A) 固定有線網段到 192.168.123.x
- 修改 `enp0s2` 為 `192.168.123.100/24`
- 透過 `netplan` 固定設定，避免每次開機手動改

**指令：**
```bash
sudo ip addr del 192.168.1.200/24 dev enp0s2
sudo ip addr add 192.168.123.100/24 dev enp0s2
ping -c 4 192.168.123.161
```

**永久設定（netplan）：**
```yaml
network:
  version: 2
  ethernets:
    enp0s1:
      dhcp4: true
    enp0s2:
      dhcp4: no
      addresses:
        - 192.168.123.100/24
```

```bash
sudo netplan apply
```

> `Cannot call Open vSwitch: ovsdb-server.service is not running.` 可忽略。

### B) LZ4 LiDAR 解碼器最小移植
- 新增 LZ4 解碼器檔案
- `data_decoder.py` 優先使用 LZ4，失敗則 fallback 回 WASM
- `requirements.txt` 加入 `lz4>=4.0.0`

**新增/修改：**
- `go2_robot_sdk/infrastructure/sensors/lidar_decoder_lz4.py`
- `go2_robot_sdk/infrastructure/webrtc/data_decoder.py`
- `requirements.txt`

**確認：**
- `disableTrafficSaving(True)` 已在 WebRTC 連線後被呼叫（`webrtc_adapter.py:55`）

**安裝依賴：**
```bash
uv venv .venv
source .venv/bin/activate
uv pip install lz4
```

## 🐛 pointcloud_to_laserscan /scan all inf 問題調查 (下午)

### 問題描述

在 SLAM mapping 測試中發現 `/scan` topic 發布的雷射資料全為 `inf`（無窮大），導致 slam_toolbox 無法正常建圖。

### 調查過程

#### 1. 檢查 /scan topic 狀態

```bash
ros2 topic echo /scan --no-arr
```

**結果：** 所有 range 值皆為 `inf`

#### 2. 檢查來源 /point_cloud2 topic

```bash
ros2 topic info /point_cloud2
ros2 topic hz /point_cloud2
ros2 topic echo /point_cloud2 --once
```

**結果：**
- Publisher 存在：`/lidar_to_pointcloud`
- 頻率正常：約 14-15 Hz
- **問題發現：** `width=0`, `row_step=0`（點雲資料結構異常）

#### 3. 檢查 QoS 設定

```bash
ros2 topic info /scan --verbose
ros2 topic info /point_cloud2 --verbose
```

**結果：** QoS 設定正常（Reliable, Keep Last 10）

#### 4. 調整 pointcloud_to_laserscan 參數

修改 `robot.launch.py` 中的參數：

```python
parameters=[{
    'target_frame': 'base_link',
    'min_height': -2.0,  # 新增
    'max_height': 0.5,    # 調整原值
    'angle_min': -3.14,  # 新增
    'angle_max': 3.14,   # 新增
}],
```

重啟後測試：

```bash
colcon build --packages-select go2_robot_sdk
source install/setup.bash
ros2 launch go2_robot_sdk robot.launch.py slam:=true nav2:=false rviz2:=false
ros2 topic echo /scan --no-arr
```

**結果：** 仍為全 `inf`，無改善

#### 5. 檢查 TF 座標轉換

```bash
ros2 run tf2_ros tf2_echo map base_link
ros2 run tf2_ros tf2_echo base_link front_camera
ros2 run tf2_tools view_frames
```

**結果：** TF 樹狀結構正常，所有 frame 可正常轉換

#### 6. 檢查 lidar_processor 日誌

```bash
RCUTILS_LOGGING_SEVERITY=DEBUG ros2 launch go2_robot_sdk robot.launch.py slam:=true nav2:=false 2>&1 | grep -i "lidar\|voxel\|publish"
```

**關鍵發現：**
- 日誌顯示：`publisher context invalid`
- `/utlidar/voxel_map` 未發布
- `publish_raw_voxel` 預設關閉，導致 `/utlidar/voxel_map_compressed` 無資料

### 根本原因推測

1. **主要問題：** `/point_cloud2` 的 `width=0`、`row_step=0` 表示點雲資料格式錯誤，可能是：
   - LZ4 解碼器輸出格式未正確轉換為 ROS2 PointCloud2
   - PointCloudAggregator 未正確處理來自 lidar_processor 的資料

2. **次要問題：** `lidar_processor` 內部的 voxel map publisher 狀態異常（`publisher context invalid`）

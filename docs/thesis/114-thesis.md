# 第一章 系統描述

## 一、發展背景與動機

### 研究背景

近年來，人工智慧技術發展迅速，特別是大型語言模型（Large Language Models, LLMs，如 Qwen、LLaMA）的突破，使得機器系統具備了深度的語意理解與邏輯推理能力。同時，邊緣運算與電腦視覺（如 MediaPipe 姿態辨識、YuNet 人臉偵測技術）以及語音處理技術（如 SenseVoice 語音辨識與 edge-tts 語音合成）的日益成熟，讓資訊系統能夠以多模態（Multi-modal）的方式，精準感知周遭環境與人類的真實意圖。
<!-- 修正：Sherpa STT → SenseVoice、MeloTTS → edge-tts、RTMPose → MediaPipe/YuNet、Claude → Qwen/LLaMA -->

在終端硬體載具方面，四足機器人（如仿生機器狗）的機電控制與運動靈活性已有顯著的提升。然而，綜觀目前市面上的實體機器人應用，多半仍依賴傳統的遙控設備、手機 App 或是僵化的預設腳本進行操作。現有系統缺乏將「雲端高階 AI 大腦」與「邊緣終端硬體（ROS2）」深度結合的解決方案，導致人機互動往往流於單向、死板的指令接收，機器無法根據人類自然的肢體語言（手勢）、語音指令或當下的環境脈絡，進行即時的動態感知與自主決策。

### 研究動機

基於上述背景，本專題「PawAI」旨在打破傳統實體機器人僵化且高門檻的互動模式。我們運用資訊管理領域的系統整合（System Integration）與軟體工程技術，開發一套無縫串接視覺感知、語音互動與大型語言模型決策的智能控制系統。

本研究的核心動機包含以下幾點：

1. **實現自然且直覺的人機互動**：透過建構完整的多模態感知管線（Multi-modal Perception Pipeline），讓系統能即時捕捉並解析使用者的手勢與臉部特徵；結合語音雙向處理，讓使用者能以最自然的「說話」與「動作」與機器狗互動，免除繁瑣的傳統控制介面。
<!-- 修正：視覺推論拓撲 → 多模態感知管線 -->

2. **賦予終端設備動態決策能力**：藉由串接大型語言模型 Qwen2.5-7B-Instruct（透過 vLLM 部署於遠端 RTX 8000 伺服器）作為系統的核心邏輯中樞，將接收到的多模態感知資料轉化為結構化的意圖（Intent），讓機器狗不再只是單純執行寫死的程式碼，而是能「理解」複雜情境並規劃後續行為。
<!-- 修正：Claude API → Qwen2.5-7B-Instruct (vLLM on RTX 8000) -->

3. **軟硬體整合與控制迴路優化**：透過建構穩定的 ROS2（Robot Operating System 2）節點通訊與統一中控模組（Interaction Executive），確保 AI 模型的決策結果能以極低的延遲、安全且精確地轉化為實體機器狗的具體動作，完成從「感知」、「思考」到「行動」的完整閉環。
<!-- 修正：ralph-loop → Interaction Executive -->

總結而言，本專題期望透過 PawAI 系統的建置，探索大型語言模型結合實體機器人的無限可能，不僅為智慧陪伴、巡檢或展演等應用場景提供具高度擴充性的架構範例，更進一步實現真正意義上的智慧人機共作（Human-Robot Collaboration）。

## 二、系統發展目的

### 問題與解決方案對應表

| 現有問題 | 本系統之解決方法 |
|----------|------------------|
| 實體機器人多依賴遙控器或手機 App 操作，互動方式單向且僵化。 | 整合 MediaPipe 手勢/姿勢辨識與語音模組，實現自然直覺的多模態（手勢、語音、人臉）互動。 |
| 機器人僅能執行預先寫死的腳本（Hard-coding），缺乏情境理解與自主決策能力。 | 串接大語言模型（Qwen2.5-7B-Instruct）作為核心大腦，依據當下脈絡動態生成行為回饋。 |
| 雲端 AI 模型與邊緣硬體設備之間，缺乏穩定、低延遲的整合與執行機制。 | 開發基於 ROS2 的統一中控模組（Interaction Executive），精準且安全地驅動終端硬體作動。 |
| 傳統機器人系統架構較為封閉，難以輕易擴充新功能或進行系統除錯。 | 建立模組化感知管線與 PawAI Studio 網頁控制台，確保系統具備高擴充性與可觀測性。 |
<!-- 修正：RTMPose → MediaPipe、ralph-loop → Interaction Executive、Audit 稽核日誌 → PawAI Studio 可觀測性 -->

### 功能性需求

**(一) 整合多模態感知技術，提供自然直覺的人機互動體驗**

打破傳統單向且受限的控制介面，本系統導入 MediaPipe 模型進行即時的姿態與手勢推論，並輔以 YuNet + SFace 人臉辨識技術鎖定互動對象。同時，結合 STT（語音轉文字）與 TTS（文字轉語音）模組，讓使用者能以最日常的肢體動作與口語對話，與系統進行雙向、流暢的互動。

**(二) 串接大語言模型，建構具備情境理解的自主決策中樞**

為賦予機器狗動態決策的能力，本專案將前端擷取的多模態感知資料轉化為結構化的意圖（Intent），並透過 HTTP API 串接部署於遠端伺服器的大語言模型（Qwen2.5-7B-Instruct）。藉此，系統能深入理解使用者的語意與當下的互動脈絡，取代傳統預先寫死的控制腳本，自主規劃並生成最合適的行為回饋。
<!-- 修正：Claude API → Qwen2.5-7B-Instruct via HTTP -->

**(三) 開發 ROS2 統一中控，實現低延遲且穩定的硬體作動閉環**

為確保雲端 AI 的決策能精確落實於邊緣實體設備，本系統建構了基於 ROS2 的節點通訊機制。透過統一中控模組 Interaction Executive（以 State Machine 實現事件路由與優先序仲裁），系統能將抽象的行為指令，以極低的延遲轉譯為機器狗能直接執行的動作控制訊號，達成從「感知」、「運算」到「作動」的無縫銜接。
<!-- 修正：ralph-loop → Interaction Executive (State Machine) -->

**(四) 建立 PawAI Studio 網頁控制台，確保系統的高可觀測性與可維護性**

導入軟體工程的模組化設計思維，建立 PawAI Studio 作為系統的統一觀測與操控入口。Studio 整合即時感知面板（人臉、手勢、姿勢、物體、語音狀態）、語音/文字互動介面、以及即時影像串流，方便團隊進行後續的效能監控與錯誤排查，更保留了未來擴增功能的彈性。
<!-- 修正：Audit 稽核日誌 + 視覺推論拓撲 → PawAI Studio 可觀測性 -->

### 非功能性需求

- **低延遲**：語音端到端（E2E）互動延遲目標 < 5 秒（含 ASR + LLM + TTS）
- **高可用性**：三級降級策略（雲端 → 本地 → 規則），確保無網路時仍可基本互動
- **模組化**：每個感知模組為獨立 ROS2 package，可單獨替換或升級
- **安全性**：統一中控模組仲裁所有動作指令，緊急事件（跌倒）具最高優先序

## 三、系統範圍

本章根據第二章所列出的功能性需求，對每項功能進行進一步的系統應用描述與技術實作說明，內容以使用者操作情境為主軸，輔以後端整合技術說明，呈現 PawAI 系統在各模組中的應用與行為。PawAI 以 Unitree Go2 Pro 四足機器人為載體，搭配 NVIDIA Jetson Orin Nano 邊緣運算平台與 Intel RealSense D435 深度攝影機，整合人臉辨識、語音互動、手勢控制、姿勢監測、物體辨識五大感知模組，並以 PawAI Brain 三層決策架構（Safety / Policy / Expression）與 Interaction Executive 狀態機作為統一中控。系統以「居家互動為主、守護能力為輔」為設計核心（互動佔比約 70%、守護佔比約 30%），對應第二章「居家互動機器狗」的產品定位。

### 3-1 人臉辨識與個人化歡迎互動

PawAI 透過機身搭載的 Intel RealSense D435 攝影機持續捕捉 RGB 影像，於使用者進入視野後約 2 秒內完成人臉偵測與身份辨識，並發出個人化的語音問候。具體情境：當家人從玄關或客廳進入機器狗的視野範圍時，PawAI 會在 PawAI Studio 上顯示追蹤到的人臉框、姓名與相似度百分比；若為已註冊的家庭成員，系統會透過語音合成發出客製化問候（例如「爺爺，您回來了！」），機器狗同時會做出打招呼的動作（`ACTION_HELLO`），提供類似寵物迎接主人的溫度感。

若場景中同時有多位家人出現，系統最多可同時追蹤 5 位不同身份的人臉，並為每位追蹤對象維持獨立的 `track_id` 與穩定化狀態。若偵測到的人臉與資料庫中已註冊的使用者皆不匹配，系統會將對象歸類為「未知身份」，此時若切換至守護模式，未知身份且停留時間超過閾值的人員會觸發陌生人警戒流程（詳見 3-8 節）。

在後端技術層面，系統使用 OpenCV Zoo 內建的 YuNet 人臉偵測模型（2023 年版）即時定位 RGB 畫面中的人臉區域，接著由 SFace 模型（2021 年版）提取 128 維的臉部特徵向量，並與儲存於 Jetson 本地端 `face_db/` 的已註冊向量進行餘弦相似度比對。為降低單幀抖動造成的身份誤判，系統導入 Hysteresis 雙閾值穩定化機制（`sim_threshold_upper=0.30`、`sim_threshold_lower=0.22`）與多幀投票，並搭配 IOU（Intersection over Union）演算法維持跨幀的追蹤連續性。辨識成功後透過 ROS2 事件發布至統一中控模組，由中控模組根據使用者身份與冷卻期規則觸發個人化語音問候。

**對應功能性需求**：(一) 整合多模態感知技術，提供自然直覺的人機互動體驗。

### 3-2 中文自然語音對話

PawAI 支援使用者以繁體中文與機器狗進行雙向自然對話，涵蓋閒聊、問答、動作指令等三種互動模式。具體情境：使用者透過 PawAI Studio 網頁介面的按住說話（push-to-talk）按鈕錄音，系統會即時顯示目前的管線階段（聆聽 → 辨識 → 思考 → 說話），錄音結束後約 2 至 4 秒內完成端到端處理，最後透過 USB 外接喇叭或 Go2 機身播放語音回覆，同時在 PawAI Studio 聊天面板顯示逐字稿與 AI 回覆的對話氣泡。若使用者說出的是預設的動作指令（例如「坐下」「停」「站起來」），系統會繞過大型語言模型直接觸發對應的機器狗動作，確保低延遲的即時反應；若為一般對話內容，則送入大型語言模型生成情境化的文字回覆。

在後端技術層面，語音互動管線由四個子模組組成：

- **語音辨識（ASR）**：採用三級自動降級機制，主線為部署於遠端 RTX 8000 伺服器的 SenseVoice Cloud（FunASR 框架，中文辨識率達 92%），當雲端不可用時自動切換至 Jetson 本地的 SenseVoice Local（sherpa-onnx int8 量化版本），最終備援為 Whisper Small（faster-whisper CUDA 推理）。

- **意圖分類（Intent）**：ASR 輸出的文字會先經過規則式意圖分類器，高信心的動作指令直接映射為 `ACTION_SIT`、`ACTION_STAND`、`ACTION_STOP` 等控制意圖，其餘進入大型語言模型。

- **大型語言模型對話（LLM）**：採用阿里巴巴開源的 Qwen2.5-7B-Instruct 模型，透過 vLLM 推理框架部署於遠端 RTX 8000 伺服器，支援 Prefix Caching 加速，生成簡短合理的中文回覆（硬截斷約 25 字）。若雲端不可用，自動降級為 Jetson 本地的 Qwen2.5-1.5B，最終備援為 RuleBrain 規則模板。

- **語音合成（TTS）**：主線採用微軟 edge-tts 的 Xiaoxiao 神經語音（延遲約 0.7 秒、音質 A 級），備援為本地 Piper huayan 中文語音（離線可用）。

- **音訊播放**：支援兩條路徑，主線為透過 USB 外接喇叭直接播放（高清晰度、穩定），備援為透過 Go2 Megaphone DataChannel（按照 `ENTER → UPLOAD → EXIT` 協議上傳 16 kHz 音訊至 Go2 機身喇叭）。

**對應功能性需求**：(一) 整合多模態感知技術；(二) 串接大型語言模型，建構具備情境理解的自主決策中樞。

### 3-3 手勢遠距控制

PawAI 支援使用者在距離 D435 攝影機約 2 公尺內以手勢對機器狗發出控制指令，無需接觸機身或透過任何裝置。具體情境：使用者比出「張開手掌」代表停止、「比讚」代表肯定與開心動作、「OK 手勢」代表確認等，系統辨識後會透過 Interaction Executive 狀態機分派對應的 Go2 動作指令，並根據手勢意義播放適當的語音回應（例如比讚後說「收到！」）。手勢互動對使用者來說是一種靜音且遠距的控制方式，特別適合不便大聲說話或機器狗距離較遠的場景。

為避免使用者維持同一手勢造成重複觸發，系統對一般手勢設有 5 秒冷卻期；但緊急停止（張開手掌、`stop` 手勢）為安全優先，無冷卻限制，確保使用者隨時可在任何狀態下即時中斷機器狗的動作。

在後端技術層面，系統採用 Google MediaPipe Hands 模組與 Gesture Recognizer Task API 進行即時手勢辨識。系統從 D435 RGB 影像中偵測手部的 21 個 3D 關鍵點（每幀每隻手），接著分析手指伸直與彎曲狀態，將結果分類為預定義的手勢類別（`stop`、`thumbs_up`、`ok`、`fist`、`point`、`open_palm` 等）。辨識結果透過 `/event/gesture_detected` 事件發布至中控模組，再由中控模組根據手勢類型與目前的狀態機狀態決定是否執行動作（例如 `stop` 在任何狀態都觸發、`thumbs_up` 僅在 `IDLE` 或 `CONVERSING` 狀態生效）。

**對應功能性需求**：(一) 整合多模態感知技術；(三) 開發 ROS2 統一中控，實現低延遲且穩定的硬體作動閉環。

### 3-4 姿勢監測與跌倒緊急警示

PawAI 在使用者出現於視野後持續監測人體姿勢狀態，並針對跌倒等緊急情境觸發安全警示流程。具體情境：系統將人體姿勢分類為站立、坐姿、蹲姿、躺臥、跌倒五種狀態，並在 PawAI Studio 姿勢面板上即時顯示目前狀態與信心度，便於家屬或操作者觀察使用者在視野中的活動狀況；當系統偵測到使用者呈現跌倒姿勢且持續達判定閾值時，觸發緊急警示流程（`EMERGENCY` 狀態），機器狗會立即停止任何移動、發出語音確認（例如「您還好嗎？」），並在 PawAI Studio 上以紅色邊框閃爍的方式醒目標示。

因應本專題已將定位調整為「居家互動為主、守護能力為輔」，跌倒偵測功能定位為「安全輔助功能」而非醫療級的跌倒監測；功能在 Demo 場景中會根據需求以 `enable_fallen` 參數啟用或關閉，避免因背景干擾造成誤報影響展示流程。

在後端技術層面，系統採用 Google MediaPipe Pose 模組辨識人體上的 33 個 3D 關鍵點。姿勢分類器透過計算 hip、knee、trunk 等關節的角度，以及 bbox 長寬比與 `vertical_ratio`（肩膀至髖部垂直距離除以軀幹長度）等相對尺度指標判定姿勢類別。跌倒判定設有多幀投票機制（20 幀多數決）與 `vertical_ratio < 0.4` 的防護條件，降低因背景複雜或姿勢誤判造成的誤報。姿勢狀態與事件透過 `/event/pose_detected` 發布，供 Interaction Executive 進行優先序仲裁（`POSE_FALLEN` 為最高優先級事件之一）。

**對應功能性需求**：(一) 整合多模態感知技術；(三) 開發 ROS2 統一中控。

### 3-5 環境物體情境感知

PawAI 持續分析視野中出現的物體，對特定高價值類別觸發情境式互動話術，使機器狗具備對環境有感知且能主動引起對話的能力。具體情境：當使用者於機器狗前方放下水杯，PawAI 會透過視覺偵測到物體並發出「你要喝水嗎？」的語音提示；看到書本會說「在看書啊」；看到水瓶會提醒喝水。物體辨識結果同時顯示於 PawAI Studio 的 Live View 面板，可視化呈現每個偵測到的物體類別與信心度。為避免同一物體持續存在於視野中造成重複觸發，系統對每一個類別都設有獨立的 5 秒冷卻期。

在後端技術層面，系統採用 YOLO26 Nano 物體偵測模型（ONNX 格式，檔案大小約 9.5 MB），並透過 NVIDIA TensorRT FP16 混合精度推理加速，在 Jetson Orin Nano 上可達到每秒 15 幀的穩定推理速度。模型採用 NMS-free 設計，直接輸出 `(1, 300, 6)` 形狀的邊界框與類別概率張量。系統支援 COCO 資料集定義的 80 類常見物體，並可透過 `class_whitelist` 參數動態調整要關注的類別範圍。高價值物體的情境話術由統一中控模組根據 `OBJECT_TTS_MAP` 映射表決定，且僅在 `IDLE` 狀態下觸發以避免打斷正在進行的對話。

**對應功能性需求**：(一) 整合多模態感知技術；(二) 串接大型語言模型。

### 3-6 PawAI Studio 網頁控制台

PawAI Studio 是系統的統一觀測與操控入口，讓操作者（開發人員、展示人員、指導老師）能透過瀏覽器遠端監看所有感知模組的即時狀態、與機器狗進行文字或語音對話、並觀察機器狗眼中的世界，大幅降低現場操作與除錯的門檻。具體情境如下：

- **首頁（Mission Control）**：提供 Chat 對話面板作為主要互動入口，使用者可透過文字輸入或按住麥克風錄音與機器狗對話，對話歷史以 bubble 方式呈現。錄音時介面會顯示 7 條即時音量動畫，提供即時的錄音回饋。

- **感知面板**：六個獨立分頁分別顯示人臉、語音、手勢、姿勢、物體的即時事件與歷史紀錄。人臉面板顯示追蹤中的人名、相似度、距離與邊界框；姿勢面板顯示目前的姿勢狀態與信心度，並於跌倒事件時以紅色警示；物體面板顯示目前視野中所有偵測到的物體類別與信心度。

- **Live View 即時影像面板**：三欄並列顯示人臉偵測、姿勢與手勢、物體偵測的 debug 影像（各自以獨立的 binary WebSocket 傳輸 JPEG，頻率約 5 FPS），操作者可直接看到機器狗「眼中的世界」並判斷各感知模組是否正常運作。

- **系統狀態列**：頂部顯示 Gateway 連線狀態、Jetson 溫度、記憶體餘量等運行資訊。

在後端技術層面，PawAI Studio 前端採用 Next.js 16 與 React 19 建構，透過 Tailwind CSS 實作 UI 樣式；後端 Gateway 以 Python FastAPI 結合 rclpy（ROS2 Python Client Library）橋接 ROS2 Topic 與瀏覽器 WebSocket。事件資料以 JSON 格式透過 `/ws/events` WebSocket 廣播，影像資料則透過 `/ws/video/face`、`/ws/video/vision`、`/ws/video/object` 三條獨立的 binary WebSocket 傳輸，以避免影像佔用主事件通道頻寬。

**對應功能性需求**：(四) 建立 PawAI Studio 網頁控制台，確保系統的高可觀測性與可維護性。

### 3-7 統一中控模組（Interaction Executive 與 PawAI Brain）

統一中控模組是整個 PawAI 系統的決策核心，負責聚合所有感知事件、仲裁優先序、維持狀態機、並分派最終的動作指令至語音合成與 Go2 動作通道。本專案的中控層分為兩個協作元件：**Interaction Executive**（以 ROS2 節點實作的狀態機與事件仲裁器，主要處理低階事件路由與安全保證）與 **PawAI Brain**（規劃中的三層決策引擎，負責高階的技能選擇與回覆生成）。

**具體情境**：使用者走到機器狗前方 → 人臉模組發出 `face_identity` 事件 → Executive 進入 `GREETING` 狀態並觸發個人化問候 → 使用者說話 → 語音模組發出 `speech_intent_recognized` 事件 → Executive 切換至 `CONVERSING` 狀態並將文字送入 LLM → LLM 生成回覆 → Executive 分派至 TTS 與 Go2 動作通道 → 使用者比出 stop 手勢 → gesture 事件觸發最高優先級的 `STOP` 動作 → Executive 立即中斷目前所有進行中的動作。整個流程中，Executive 負責確保同一時間只有一個動作被執行（避免衝突）、確保緊急事件（跌倒、停止手勢）能立即中斷其他活動（安全優先）、並避免重複事件造成的重複觸發（去重機制）。

在後端技術層面，Interaction Executive 採用顯式狀態機設計，管理 `IDLE → GREETING → CONVERSING → EXECUTING → EMERGENCY → OBSTACLE_STOP` 六種狀態的轉移。事件優先序由高至低為：跌倒（最高）> 障礙物 > 語音停止 > 語音意圖 > 手勢 > 人臉 > 物體（最低）。系統對同一來源與類型的事件採用 5 秒滑動視窗去重，防止感知模組連續發布相同事件造成重複觸發；每個非 `IDLE` 狀態設有 30 秒超時自動返回機制，避免狀態卡死。Executive 透過 `/tts` topic 發送要說的文字至語音合成模組、透過 `/webrtc_req` topic 發送動作指令（如 `api_id: 1009` 代表坐下、`api_id: 1003` 代表緊急停止）至 Go2 驅動節點。

**PawAI Brain** 為規劃中的高階決策引擎，採用三層架構：

- **Safety Layer（安全層）**：最高優先的 deterministic guard，負責 stop、emergency、障礙物偵測、banned_api 白名單攔截等強制規則。

- **Policy Layer（策略層）**：基於情境的技能選擇層，結合人物記憶（`person_profiles`）與規則覆寫機制，從 PawAI Skills 集合中選出合適的技能並組成 skill queue。

- **Expression Layer（表達層）**：負責語氣、用詞、互動風格的生成，結合 LLM 的語言能力但受 Policy Layer 約束。

Brain 透過 Skill Queue 機制支援多步驟行為序列（例如自主展示、陌生人警戒等複合行為）；Safety Layer 可隨時清空 queue 並中斷當前技能執行，確保緊急事件能立即被處理。Brain 的完整實作規劃於 4/11 系統設計規格中，目前 Interaction Executive 已實作 v0 版本的事件路由與狀態機，並預留與 Brain 的介面（`/state/pawai_brain` topic）供後續整合。

**對應功能性需求**：(二) 串接大型語言模型；(三) 開發 ROS2 統一中控。

### 3-8 陌生人警戒與主動守護

作為「守護能力」功能群的 Demo 代表，陌生人警戒整合了人臉辨識、中控狀態機與語音合成，提供一個具體可展示的守護情境。具體情境：使用者透過語音或 PawAI Studio 介面將機器狗切換至守護模式後，若系統偵測到未註冊的人員（face_identity 結果為 `unknown`）進入視野且停留時間超過預設閾值（Demo 設定為 2 秒），機器狗會發出語音警告（如「您好，我不認識您，請告訴我您是誰」）並在原地警戒，同時在 PawAI Studio 的事件面板記錄警戒事件。為避免誤報造成困擾，系統設有反誤報策略（anti-false-positive policy）：閾值包含停留時間門檻、人臉辨識信心度門檻與光線條件檢查；在非守護模式下，陌生人不會觸發警戒。

本功能目前對應 4/11 方向收斂中「互動 70% / 守護 30%」中的 30% 守護比例，在 Demo 流程中作為展示 PawAI 同時具備陪伴與安全兩種角色的核心案例。推播通知（例如通知家屬手機）屬於遠端存取範疇，本次未實作，詳見第五章系統限制。

**對應功能性需求**：(一) 整合多模態感知技術；(三) 開發 ROS2 統一中控。

### 3-9 自主展示與自我介紹（Demo Wow Moment）

PawAI 具備主動執行多步驟行為序列的能力，其中「自我介紹」是 Demo 開場的核心展示項目（Wow Moment）。具體情境：操作者透過語音指令（例如「PawAI，介紹一下你自己」）或 PawAI Studio 按鈕觸發後，機器狗會依序執行下列動作：先站起身面向觀眾 → 發出打招呼動作配合語音「大家好，我是 PawAI」→ 說明自己的主要功能（語音對話、手勢辨識、人臉識別等）→ 演示代表性動作（轉圈、坐下、搖頭）→ 最後以「很高興認識大家」作結。整個序列約 60 至 90 秒，展現機器狗能主動表達而非被動等待指令，強化使用者對「這是一隻有個性的機器狗」的第一印象。

在後端技術層面，此功能透過 PawAI Brain 的 Skill Queue 機制實作，將整段自我介紹拆解為一系列可獨立驗證的子技能（play_stand_animation、speak_greeting、play_hello_gesture、speak_intro、play_demo_action 等），依序交由 Safety Layer 驗證通過後由 Executive 分派執行。使用者隨時可以透過「停」語音指令或 stop 手勢中斷整個序列，Safety Layer 會立即清空 queue 並將 Executive 狀態回復至 `IDLE`。

**對應功能性需求**：(二) 串接大型語言模型；(三) 開發 ROS2 統一中控。

### 3-10 自主導航避障（條件式功能，依賴外接雷達）

PawAI 規劃具備自主導航與避障能力，讓機器狗能在家中自由移動、主動避開家具與人員障礙物、並在使用者呼叫時安全地走到指定位置。具體情境：使用者從沙發上呼叫「PawAI，過來！」後，機器狗能自動規劃從目前位置到使用者附近的路徑，途中遇到椅腳、門檻、散落的玩具等障礙物時自動繞行或停下；跟隨模式下機器狗可跟隨使用者在家中移動，維持安全距離；巡邏模式下可在預設路徑上定時巡視並回報異常。

**本功能為條件式啟用**：由於 Go2 Pro 內建 Hesai XT16 LiDAR 實測覆蓋率僅約 18%、D435 深度攝影機因安裝角度無法有效偵測低矮障礙物（詳見第五章第 1 點），本功能依賴外接 SLAMTEC RPLIDAR A2M12 雷達（360° 全向覆蓋、每秒 10 次掃描）作為主要的感知輸入，搭配 ROS2 Nav2 導航堆疊與 `slam_toolbox` 建圖框架實作。RPLIDAR 採購流程透過指導老師的國科會計畫變更進行，於本文件繳交時到貨時程尚未確定。若雷達如期到貨且 Go2 四足里程計漂移量驗證通過，本功能將於後續版本正式啟用；若未到貨或漂移過大導致 SLAM 不可行，則退而求其次採用 360° 反應式避障（不依賴建圖），若仍不可行則本功能維持為文件級的未來工作。

在後端技術層面，若外接雷達啟用，系統將採用以下整合方案：RPLIDAR 透過 USB 直連 Jetson，`rplidar_ros` 驅動節點發布 `/scan` topic → `slam_toolbox` 即時建圖並發布 `/map` → Nav2 導航堆疊訂閱 `/map` 與 `/scan` 進行路徑規劃 → 決策結果透過 `cmd_vel` topic 送至 Go2 驅動節點執行移動控制。安全層面由 `nav2_collision_monitor` 負責，確保機器狗在障礙物過近時強制停止。

**對應功能性需求**：(三) 開發 ROS2 統一中控，實現低延遲且穩定的硬體作動閉環（延伸至導航控制層）。

## 四、背景知識

### 4-1 系統相關背景知識

#### 源起

隨著高齡化社會的來臨與家庭結構的轉變，獨居長者、雙薪家庭與小家庭模式日益普遍。家人離家工作期間，長輩的日常安全、情感陪伴與社交互動成為許多家庭關心的議題。傳統的居家照護方案多以穿戴式裝置（如 Apple Watch 跌倒偵測）、固定式監視攝影機（如 Ring、Nest）或語音助理（如 Amazon Alexa、Google Home）為主，但這些方案各自存在限制：穿戴式裝置需使用者主動配戴、固定式攝影機缺乏主動關懷能力、語音助理則僅能被動回應語音指令且無實體存在感。

另一方面，人工智慧技術於近五年呈現爆發式發展：大型語言模型（Large Language Models, LLM）如 Qwen、LLaMA、Claude 等展現出類人水準的自然語言理解與生成能力；邊緣運算平台（如 NVIDIA Jetson 系列）讓複雜的電腦視覺與語音模型能在本地端即時運行，大幅降低雲端依賴與隱私風險；四足機器人平台（如 Unitree Go2）的成本從過往的數萬美元降至約 2,800 美元，讓機器人不再是研究機構的專屬裝備，而能進入家庭場景。

基於以上技術成熟條件，PawAI 專題的核心構想是：結合消費級四足機器人平台、邊緣 AI 運算、多模態感知融合與大型語言模型，打造一隻「居家互動為主、守護為輔」的家庭機器狗。相較於純粹的語音助理，PawAI 具備實體存在感與移動能力；相較於傳統監視攝影機，PawAI 能主動互動而非被動錄影；相較於封閉的情感陪伴機器人（如 Sony Aibo），PawAI 具備大型語言模型驅動的自然對話能力與開放的軟體架構。系統以互動佔比 70%、守護佔比 30% 為核心設計比例，定位為青壯年為長輩購買的陪伴平台。

#### 發展歷程

**早期階段：家用機器人與語音助理的分流**

2000 年代，家用機器人與語音助理循不同路線發展。家用機器人以 iRobot Roomba（2002 年推出）為代表，聚焦於單一功能（掃地、拖地），採輪式底盤，具移動能力但缺乏互動。情感陪伴機器人以 Sony Aibo（首代 1999 年推出）為代表，採四足機械犬造型，強調情感連結但缺乏語音理解能力。語音助理方面，Apple Siri（2011 年）、Amazon Alexa（2014 年）、Google Assistant（2016 年）等相繼推出，透過智慧手機或智慧音箱提供語音互動，但缺乏實體存在與移動能力。此階段呈現「能移動的不會對話、會對話的不能移動」的矛盾局面。

**商業化與互動機器人的興起**

2015 年後，家用互動機器人逐漸商業化。Sony 於 2018 年推出第二代 Aibo ERS-1000，整合深度學習與攝影機，具備個性化學習能力。美國新創公司 Anki 推出 Vector 與 Cozmo 桌面陪伴機器人，主打情感表達與小型互動。日本 GROOVE X 於 2019 年推出 LOVOT 情感互動機器人，強調心靈療癒功能。Amazon 於 2021 年推出家用輪式機器人 Astro，整合 Alexa 語音助理、家庭地圖建構與視訊通話功能。然而，這些產品多為封閉系統，缺乏開放的軟體架構，且未整合當時尚未成熟的大型語言模型，互動能力仍受限於預設腳本與規則。

**四足機器人進入消費級市場**

四足機器人原本是學術研究與商用檢測的專屬領域。Boston Dynamics 於 2019 年推出 Spot 四足機器人（售價約 75,000 美元），主打商用巡檢、保全與研究用途。Unitree Robotics（宇樹科技）於 2017 年推出首款四足機器人 Laikago，後續推出 A1、Go1 等研究平台，價格逐步降低。2023 年發布的 Unitree Go2 系列（Air / Pro / EDU 三版本）將消費級四足機器人的價格壓至 1,600 至 2,800 美元，並提供開放 SDK，讓學術與開源社群能以此為基礎開發自訂應用。本專題採用 Go2 Pro 版本，配備 Hesai XT16 LiDAR 與 4G 模組，透過 WebRTC DataChannel 進行控制通訊。

**大型語言模型時代的 Embodied AI**

2022 年底 OpenAI ChatGPT 發布引爆生成式 AI 熱潮，後續 GPT-4、Anthropic Claude、阿里巴巴 Qwen、Meta LLaMA 等模型相繼開源或商用。大型語言模型的 function calling 能力（呼叫外部工具）讓 AI 能從純文字生成延伸至控制實體系統。同一時間，邊緣 AI 運算平台（NVIDIA Jetson Orin、Qualcomm RB5、Apple M 系列）支援本地推理，降低雲端延遲與隱私風險。研究領域出現 Google RT-2、Physical Intelligence 的 VLA（Vision-Language-Action）模型等方向；商業界則有 Figure 01、Apptronik Apollo、Unitree G1 等人形機器人進入開發階段。PawAI 專題選擇的路徑是：不追求通用人形機器人的難度，而是聚焦於居家互動 + 輔助守護的具體場景，以既有的開源技術堆疊出可 Demo 的完整系統。

#### 本系統創新發展

PawAI 相較於現有的家用機器人、語音助理與監視攝影機產品，於以下幾個層面有所創新：

1. **居家場景特化的多模態互動**：整合人臉辨識、中文語音對話、手勢控制、姿勢監測、物體情境感知五大模組，對應居家場景中自然發生的互動形式。多數現有產品僅支援單一或兩種互動模態（語音或觸控），PawAI 則讓使用者能以最自然的方式與機器狗溝通。

2. **Harness-Oriented Agent 三層架構**：設計 PawAI Brain 三層決策架構（Safety / Policy / Expression），以 deterministic 的 Safety Layer 作為最高優先層級，結合大型語言模型的 Policy 與 Expression 層級，確保 AI 的創造力受到安全約束。這是相較於單層 LLM agent 的重要改進，避免 LLM 幻覺動作或觸發危險 API 的風險。

3. **全離線備援保證基本互動能力**：採用三級降級（Graceful Degradation）策略，語音辨識、大型語言模型、語音合成三個環節皆設計雲端主線 + 本地備援 + 規則引擎的三層 fallback，確保在網路斷線、雲端服務異常等情況下系統仍能維持基本互動能力，不會完全啞掉。這相較於強依賴雲端的 Alexa、Astro 等產品是重要的韌性提升。

4. **Edge AI 隱私設計**：所有視覺感知（人臉、手勢、姿勢、物體）皆在 Jetson 邊緣設備本地執行，不將影像上傳雲端；人臉資料庫儲存於本地而非雲端伺服器。此設計對應居家場景中特別敏感的隱私需求，呼應 Privacy-by-Design 的設計原則。

5. **PawAI Studio 統一介面降低使用門檻**：透過 Next.js 網頁介面提供統一的操作與觀測入口，家人、展示人員或研究者都可透過瀏覽器直接與機器狗互動，無需學習複雜的 ROS2 指令或命令列操作。同時提供 Live View 即時影像面板，讓使用者能看到機器狗「眼中的世界」。

#### 新概念

**Embodied AI（具身智慧）**

Embodied AI 是指具備實體、能感知物理世界、並能對物理環境做出動作的人工智慧系統。相較於純文字或純視覺的 AI 助理（如 ChatGPT 網頁版），Embodied AI 強調三個特性：實體存在感（使用者可看到、摸到、對話的對象）、主動感知能力（AI 能主動觀察環境而非被動等待輸入）、物理執行能力（AI 的決策能轉化為實體動作）。研究界認為 Embodied AI 是下一個 AI 演進階段的核心方向，因為它讓 AI 從「資訊處理者」轉變為「環境參與者」。PawAI 的定位即是 Embodied AI 在居家場景的實踐：讓機器狗具備物理存在、主動感知、語音互動、動作回饋等完整的具身智慧能力。

**Harness-Oriented Agent Design**

Harness-Oriented Agent 是指「AI 能力受環境約束」的設計哲學。傳統的 LLM agent 架構讓大型語言模型直接決定下一步動作，此方法在純文字場景可行，但應用於實體機器人時風險極高——LLM 可能生成幻覺動作、呼叫危險 API、或陷入無限循環。Harness-Oriented Design 的核心主張是：agent 的能力應受環境明確約束才是可靠的 agent。PawAI Brain 的三層架構（Safety / Policy / Expression）是此概念的具體實作：Safety Layer 是 deterministic 的硬約束（例如禁止危險 api_id）、Policy Layer 是從預定義的技能白名單中選擇、Expression Layer 才交給 LLM 發揮語言能力。三層架構確保「即使 LLM 出錯，機器狗也不會做出危險動作」，這是將 LLM 應用於實體機器人的重要安全保證。

**Privacy-by-Design 與 Edge AI**

Privacy-by-Design 是資料保護領域的設計原則，主張在系統設計階段就將隱私保護納入考量，而非在產品上市後才補強。Edge AI 是此原則在 AI 系統的具體實踐：將運算從雲端移至使用者本地設備，讓敏感資料（影像、語音、個人特徵）不離開設備邊界。對於居家機器人而言，此設計特別重要——家庭場景中的攝影機畫面涉及個人隱私、兒童影像、家庭作息等高度敏感資料，若以雲端處理會面臨外洩、駭入、被第三方存取的風險。PawAI 採用 Edge AI 設計，人臉辨識、手勢偵測、姿勢分類、物體辨識等視覺模組全數於 Jetson 本地執行，影像不離開機器狗本體；僅語音辨識與大型語言模型在必要時透過專屬的 SSH tunnel 送至內部 RTX 8000 伺服器。此架構在隱私保護與運算能力之間取得平衡。

**多模態感知融合（Multimodal Perception Fusion）**

多模態感知融合是指系統結合多種感測器與模型（視覺、聲音、語言、動作等）做出決策的方法。單模態系統有明顯限制：單純的人臉辨識無法理解使用者的意圖；單純的語音辨識缺乏視覺脈絡；單純的姿勢偵測無法知道對象身份。PawAI 採用事件驅動的融合架構：各感知模組獨立發布事件至 ROS2 Topic，中控層根據事件類型、優先序、使用者身份等進行仲裁與整合，最終產生適當的動作反應。例如「家人進入 → 人臉辨識觸發 → 中控查詢記憶判斷身份 → 搭配語音問候與動作」就是多模態融合的典型流程。

**三級降級（Graceful Degradation）**

三級降級是指系統在部分元件失效時仍能維持基本功能的設計模式。PawAI 的語音管線是此模式的典型實作：語音辨識採用 SenseVoice Cloud（主線，準確率最高）→ SenseVoice Local（本地備援，無網路亦可用）→ Whisper Local（最終備援，雖較慢但保證可用）；大型語言模型採用 Cloud Qwen-7B（主線）→ Local Qwen-1.5B（本地小模型備援）→ RuleBrain（規則引擎，保證可回覆）；語音合成採用 edge-tts（主線）→ Piper（本地備援）。此設計讓系統在網路斷線、雲端服務異常、硬體故障等情況下仍能維持最基本的互動能力，不會完全啞掉，對使用者的觀感影響為「變慢一些或回覆變簡單」，而非「整個系統無法使用」。

#### 新技術

**Unitree Go2 Pro 四足機器人平台**

Unitree Go2 是中國宇樹科技於 2023 年發布的消費級四足機器人，定位為「人人可負擔的機器狗」。Go2 提供三種版本：Go2 Air（約 1,600 美元，無 LiDAR 與 4G，僅 APP 控制）、Go2 Pro（約 2,800 美元，配備 Hesai XT16 LiDAR 與 4G 模組，Wi-Fi 連線）、Go2 EDU（約 8,500 美元，額外搭載 NVIDIA Jetson Orin NX 8GB 處理器與完整 ROS2 SDK，支援 CycloneDDS 直連）。本專題採用 Go2 Pro 版本，透過 WebRTC DataChannel 作為控制通道與 Jetson 進行雙向通訊。Go2 Pro 支援約 42 個 Sport Mode API（含站立、坐下、握手、握爪、舞蹈動作等），並提供 AudioHub API（4001-4005）讓外部裝置透過 Megaphone DataChannel 上傳音訊至機身喇叭播放，為語音互動提供實體音訊回饋能力。本專題利用 Sport Mode API 實現機器狗動作控制，透過 AudioHub 作為語音播放的備援通道（主線為 USB 外接喇叭）。

**YuNet 與 SFace 人臉辨識技術**

本系統採用 OpenCV Zoo 內建的 YuNet 人臉偵測器（2023 年版）與 SFace 人臉辨識器（2021 年版）。YuNet 是由 Chen 等學者於 2023 年提出的輕量級人臉偵測模型，採用 anchor-free 架構，輸出人臉邊界框與五個關鍵點（雙眼、鼻尖、雙嘴角），模型檔案小且在 CPU 上能達到即時推理速度（Jetson CPU 實測約 71 FPS）。SFace 則採用 ResNet 架構，提取人臉的 128 維特徵向量（embedding），透過餘弦相似度比對完成身份辨識。相較於傳統的 Dlib 或 face_recognition 函式庫，YuNet 與 SFace 在速度、精度與部署便利性上皆有顯著優勢，且以 OpenCV DNN 作為推理後端，無需額外安裝深度學習框架。本系統在 SFace 之上加入 Hysteresis 雙閾值穩定化機制與 IOU 追蹤，是針對多人動態場景的原創設計。

**Google MediaPipe 即時感知框架**

MediaPipe 是 Google 於 2019 年開源的即時多模態感知框架，專為 CPU 友善的邊緣運算場景設計，採用 Graph-based pipeline 架構。本專案使用 MediaPipe 的三個模組：Pose（辨識人體 33 個 3D 關鍵點，採用 BlazePose 論文提出的兩階段檢測-追蹤架構）、Hands（辨識單手或雙手的 21 個 3D 關鍵點）、Gesture Recognizer（基於 Hands 的手勢分類，內建 7 個預定義手勢類別）。MediaPipe 的優勢是在 Jetson CPU 上即可達到即時推理速度，無需佔用 GPU 資源；實測 Pose 模組約每秒 18.5 幀、Hands 約每秒 7.2 幀，可與其他 GPU 推理模組（如物體辨識的 YOLO26n）並行運行而不互相干擾。本專案原本評估 RTMPose 等 GPU 模型但因資源競爭問題改採 MediaPipe。

**YOLO26 物體偵測與 TensorRT 加速**

YOLO（You Only Look Once）是 Joseph Redmon 等學者於 2016 年提出的即時物體偵測演算法系列，歷經 v3、v5、v8、v11 等版本演進，目前最新版本為 YOLO26。本系統採用 YOLO26n（Nano 版本，ONNX 格式約 9.5 MB），採用 anchor-free 與 NMS-free 設計，模型輸出為 `(1, 300, 6)` 形狀的張量，直接包含邊界框座標、信心度與類別編號，省去後處理步驟。推理加速採用 NVIDIA TensorRT 作為 ONNX Runtime 的 Execution Provider，透過 FP16 混合精度推理將延遲壓至毫秒級；首次啟動時 TensorRT 需 3-10 分鐘編譯推理引擎並快取至本地，之後即可秒級啟動。YOLO26n 支援 COCO 資料集定義的 80 類常見物體，可透過 class_whitelist 動態篩選關注的類別。

**語音辨識技術（SenseVoice / Whisper）**

本系統採用三級語音辨識策略：

- **SenseVoice Cloud（主線）**：由阿里巴巴達摩院開發的中文優化語音辨識模型，透過 FunASR 框架部署於遠端 RTX 8000 伺服器，除了基本的 ASR 功能外亦支援語言辨識與情緒標籤。本專題實測中文辨識率達 92%，延遲約 600 毫秒。

- **SenseVoice Local（二級備援）**：透過 sherpa-onnx 框架將 SenseVoice 模型以 INT8 量化部署於 Jetson CPU，離線可用且準確率與雲端版本相當。

- **Whisper Local（最終備援）**：OpenAI 於 2022 年推出的多語言語音辨識模型，本專案採用 Small 版本並透過 faster-whisper（基於 CTranslate2 的 CUDA 加速框架）在 Jetson 上推理，推理過程採用 float16 混合精度（Jetson CUDA 不支援 int8 量化）。Whisper 作為最終備援僅在前兩級皆失效時啟用。

**語音合成技術（edge-tts / Piper）**

本系統採用雙重語音合成策略：

- **edge-tts（主線）**：基於微軟 Edge 瀏覽器的 Neural Voice 引擎的開源客戶端，透過網路 API 合成自然語音。實測 Xiaoxiao 中文女聲延遲約 0.7 秒、音質評級 A 級、無需 GPU 資源。主要限制是需要網路連線。

- **Piper（本地備援）**：由 Rhasspy 開源社群維護的離線 TTS 引擎，採用 VITS 架構（Variational Inference TTS），以 ONNX 格式部署於 Jetson CPU，本專案使用 zh_CN-huayan-medium 中文模型。優點是完全離線可用，缺點是音質略低於 edge-tts 且合成延遲較長（約 2 秒）。原生採樣率為 22050 Hz，但透過 Go2 Megaphone 播放時會被硬限制降至 16 kHz，外接 USB 喇叭則可保留原生音質。

**大型語言模型（Qwen2.5-7B-Instruct）**

Qwen 是阿里巴巴於 2023 年開源的中文優化大型語言模型系列，本專題採用 Qwen2.5-7B-Instruct（70 億參數版本），相較於更小的 1.5B、0.5B 版本，7B 在中文對話、指令遵循、JSON 格式輸出等任務上有顯著優勢。模型透過 vLLM 推理框架部署於遠端 RTX 8000 伺服器，vLLM 的 PagedAttention 與 Prefix Caching 機制能大幅降低重複 prompt 的處理時間，實測本專題的互動場景下 LLM 延遲約 1.5 秒。當雲端不可用時，系統自動降級為 Jetson 本地的 Qwen2.5-1.5B（Ollama 部署），若本地模型亦不可用則最終回退至 RuleBrain 規則模板。

**NVIDIA Jetson Orin Nano SUPER Developer Kit 8GB**

NVIDIA Jetson Orin Nano 是專為 AI 邊緣推理設計的嵌入式運算模組，採用 Ampere 架構 GPU（6 核 CUDA + Tensor Core）與 6 核 Arm Cortex-A78AE CPU。本專案使用 SUPER 版本（8GB 統一記憶體），搭配 JetPack 6 作業系統（CUDA 12.6、TensorRT 10.x、cuDNN 9.x）。統一記憶體架構讓 CPU 與 GPU 共享同一塊 8GB 實體記憶體，避免傳統 PCIe 架構的資料複製開銷，特別適合多模態推理場景。Jetson 的功耗模式可透過 nvpmodel 配置，MAXN 模式下 TDP 約 25 瓦，適合本專案的即時推理需求。本系統利用 Jetson 運行所有本地感知模組（人臉、手勢、姿勢、物體、本地 ASR/LLM 備援、TTS）與 ROS2 runtime，並透過 WebRTC DataChannel 與 Go2 Pro 進行通訊。

**Intel RealSense D435 深度攝影機**

Intel RealSense D435 是一款 RGB-D 深度攝影機，整合 RGB 彩色感測器與紅外線結構光深度模組。規格包含 RGB 解析度最高 1920×1080（30 FPS）、深度解析度最高 1280×720（90 FPS）、水平視角約 87 度、最小偵測距離 0.3 公尺。本系統以 D435 作為主要視覺輸入，同一支攝影機同時供人臉偵測、姿勢辨識、手勢辨識、物體偵測使用，RGB 影像透過 ROS2 image topic 廣播，深度資訊則用於估算人臉與攝影機的距離。D435 的結構光技術相較於 ToF（Time-of-Flight）方案精度更高、近距離效果更好，但在直射陽光下會受干擾。

**WebRTC DataChannel — Go2 通訊機制**

WebRTC（Web Real-Time Communication）是一個支援瀏覽器與設備間即時通訊的開放標準，支援音訊、視訊、資料通道三種傳輸。本系統透過 WebRTC DataChannel 與 Go2 Pro 進行雙向通訊：動作控制指令（站立、坐下、走動等）透過 DataChannel 以 JSON 格式傳送，訊息類型為 `req` 或 `msg`，包含 api_id 參數指定要執行的動作；音訊播放則透過 Megaphone API（api_id 4001 至 4003），依照 `ENTER → UPLOAD × N → EXIT` 三階段協議上傳 base64 編碼的音訊至 Go2 機身喇叭。本專案發現 Megaphone API 的文件與實際行為有諸多不一致之處（chunk size 必須為 4096 base64 字元、訊息 type 必須為 `"req"`、payload 須包含 `current_block_size` 欄位），透過逆向工程與反覆實驗後成功整合。

**PawAI Studio 技術棧（Next.js + FastAPI）**

PawAI Studio 前端採用 Next.js 16（React 19 基礎）、TypeScript 與 Tailwind CSS 建構，後端 Gateway 以 Python FastAPI 結合 rclpy（ROS2 Python Client Library）實作，透過 WebSocket 將 ROS2 Topic 上的感知事件與影像轉發至瀏覽器，同時接收使用者的文字或語音輸入並反向送入 ROS2 語音管線。此架構讓前端開發者能以熟悉的 Web 技術構建介面，而不需理解 ROS2 的內部細節；Gateway 作為清晰的介面層分離了前後端技術棧，方便各自獨立演進。Next.js 16 是 2024 年末發布的新版本，支援 React Server Components 與 Partial Prerendering，為本專案的即時資料面板提供高效能基礎。

### 4-2 競品背景知識分析

目前市場上已存在多款家用機器人、四足機器人平台、語音互動助理與居家監控產品。這些產品在各自的定位中已表現優秀，但在「居家互動 + 四足機動 + 大型語言模型對話 + 輔助守護」的整合性需求上皆有不足。以下分析四款代表性產品（Sony Aibo、Boston Dynamics Spot、Amazon Astro、Unitree Go2 原廠方案）的特色與局限，作為 PawAI 設計參考與差異化基礎。

#### 1. Sony Aibo ERS-1000

Sony Aibo 是 Sony 於 1999 年首次推出的家用機器犬，當代版本 ERS-1000 於 2018 年推出，定位為情感陪伴型家用機器人。Aibo ERS-1000 採四足機械犬造型，搭載 22 個可動關節、多個攝影機、麥克風陣列與觸控感測器，支援雲端個性化學習，售價約 2,900 美元。Aibo 透過逐步學習家人的互動習慣，展現出類似真實寵物的個性化行為，情感連結強。

**優點**：外型設計精緻具玩具與陪伴雙重價值；情感表達成熟（多種眼神、叫聲、動作組合）；雲端個性化學習讓每台 Aibo 展現獨特個性；Sony 品牌信任度高；硬體穩定可靠。

**局限**：無自然語言對話能力，僅支援預設叫聲與短語；無實用守護功能，不具備陌生人警戒、跌倒偵測等安全能力；封閉系統無第三方 SDK，開發者無法擴充功能或訓練自訂行為；依賴 Sony 雲端服務運行，若 Sony 停止服務則部分功能受限（先前 1999 年款 Aibo 即曾發生此問題）；價格偏高，目前未於台灣正式販售。

**對 PawAI 的啟發**：情感表達需要動作與聲音的組合搭配，機器狗單純播放語音而沒有配合的動作會顯得生硬；個性化互動是居家陪伴機器人的重要差異化特色，但應避免封閉系統造成的更新依賴問題。

#### 2. Boston Dynamics Spot

Boston Dynamics Spot 是美國 Boston Dynamics 於 2019 年推出的商用四足機器人平台，定位為工業檢測、保全巡邏、研究開發用途，售價約 75,000 美元。Spot 重約 32.5 公斤，搭載 5 個立體視覺攝影機、360 度 LiDAR 選配、機械手臂選配，具備高度穩定的四足運動控制與多地形適應能力，支援 ROS 與 Python SDK 開發。

**優點**：機動性與穩定性業界領先（可跨越樓梯、陡坡、不平地面）；開放 SDK 與 ROS 支援，讓第三方開發者能擴充功能；可掛載機械臂、熱像儀、氣體偵測等模組；已在工業巡檢、災害救援、施工現場監測等場景實際部署；品牌知名度高。

**局限**：價格極高（約為 Go2 Pro 的 25 倍），非消費級產品；體型過大（32.5 公斤）且行走時噪音大，不適合居家環境；互動能力弱（無語音對話、無情感表達）；操作與維護需要專業人員；電池續航約 90 分鐘；主要使用場景為工業與研究，非家庭。

**對 PawAI 的啟發**：開放 SDK 與 ROS 生態整合讓第三方開發者能快速上手，這是 Go2 Pro（相較於 Spot）最大的價格優勢；同時也證明四足機器人 + 開放平台 + 第三方擴充的商業模式可行。PawAI 採用類似的開放架構，但聚焦於居家場景而非商用。

#### 3. Amazon Astro

Amazon Astro 是 Amazon 於 2021 年推出的家用輪式機器人，定位為居家巡邏、通訊、娛樂與 Alexa 語音助理延伸，售價約 1,600 美元（僅限美國且需邀請購買）。Astro 採輪式底盤（非四足），搭載 RGB-D 攝影機、多麥克風陣列、觸控螢幕與輪式移動機構，整合 Alexa 語音助理並支援家庭地圖建構與自主巡邏。

**優點**：完整整合 Alexa 語音助理，支援自然語言對話與 Alexa 技能生態；家庭地圖建構與自主巡邏能力成熟；家人辨識功能可提供個人化互動（例如為不同家人播放不同音樂）；遠端視訊與通訊功能可讓外出的家人查看家中情況；Amazon 品牌背書，客戶服務與更新支援穩定。

**局限**：輪式底盤無法跨越門檻、樓梯、地毯邊緣等居家障礙；Alexa 雖然語音辨識強但缺乏對使用者情緒的感知；美國以外地區無販售（台灣無法購買），且需要邀請才能購買；封閉生態系統，難以深度客製化；情感表達僅限螢幕表情，缺乏實體動作回饋；外型定位為「智慧家電」而非寵物，情感連結較弱。

**對 PawAI 的啟發**：家庭地圖建構與多模態感知整合的架構值得參考；但輪式底盤的限制確認了四足機動在居家環境的必要性（跨越門檻、地毯、樓梯）；情感連結不僅靠語音，實體動作與外型造型同樣重要。

#### 4. Unitree Go2 原廠 App 與 dimOS 社群方案

Unitree 為 Go2 Pro 提供原廠 iOS / Android App，支援基本的遙控操作、拍照、動作觸發（握手、舞蹈、空翻等 42 個預設動作）。原廠 App 的定位是「讓一般消費者開箱即用」，無需任何程式開發即可操控機器狗。

除原廠 App 外，開源社群亦陸續發展出針對 Go2 的第三方框架。其中 dimOS（v0.0.11 beta）是較為知名的案例，提供基於 Go2 Pro 搭配外接 Intel RealSense D435 深度攝影機實現自主巡邏的完整方案。dimOS 採用 VoxelGrid costmap 繞過 Go2 內建 LiDAR 覆蓋率不足的限制，並提供 Skills 與 Spatial Memory 架構，讓使用者能以高階技能語法定義機器狗行為。

**原廠 App 優點**：官方支援穩定；開箱即用無需程式開發；支援 42 個預設動作；整合 Unitree 官方維護的韌體更新機制。

**原廠 App 局限**：無 AI 對話能力；無自動化腳本或事件觸發機制；無開放 API 讓第三方擴充；功能侷限於遙控與預設動作。

**dimOS 優點**：證明 Go2 Pro + D435 深度攝影機可以實現自主巡邏（推翻了「Go2 Pro 無法用於 SLAM」的普遍認知）；提供 Skills + Spatial Memory 的高階抽象層；社群開源可自由擴充。

**dimOS 局限**：仍為 beta 版本（v0.0.11），穩定性與文件尚未成熟；架構與 ROS2 生態有耦合衝突，難以直接整合至既有的 ROS2 專案；社群規模小，技術支援有限；無 LLM 整合、無多模態互動；聚焦於巡邏而非互動。

**對 PawAI 的啟發**：Unitree 原廠方案提供了 Sport Mode API 與 AudioHub API 的核心基礎，但缺乏 AI 互動能力，需自行整合上層框架；dimOS 證明了 Go2 Pro + D435 VoxelGrid 可以達成自主巡邏（這對 PawAI 的導航避障規劃有重要參考價值），但其架構不適合本專題直接整合，因此本專題採用自行實作的 ROS2 Nav2 方案。

#### 差異化結論

上述四款產品在各自的定位中已展現優勢，但在「居家互動 + 四足機動 + 大型語言模型對話 + 輔助守護」的整合性需求上皆有不足：Sony Aibo 有情感陪伴但無對話能力與守護功能、Boston Dynamics Spot 有四足機動但無互動能力且價格過高、Amazon Astro 有 Alexa 整合但為輪式且封閉系統、Unitree 原廠與 dimOS 提供四足平台但缺乏 AI 大腦與多模態互動。

PawAI 透過整合以下元素填補上述空缺：

- **Unitree Go2 Pro 四足平台**：以消費級價位（2,800 美元）提供穩定的四足機動能力與情感表達基礎，克服輪式機器人的居家障礙限制。

- **Qwen2.5-7B 大型語言模型 + 三級降級**：提供自然中文對話能力，並以三級 fallback 保證網路不穩時的基本功能。

- **多模態感知融合**：結合人臉、語音、手勢、姿勢、物體五種感知模態，支援多樣化的自然互動方式。

- **PawAI Brain 三層架構**：透過 Safety / Policy / Expression 分層確保 AI 行為受約束，避免大型語言模型的幻覺動作造成安全風險。

- **Edge AI 隱私設計**：感知全本地化，視覺資料不離開機器狗本體，保護居家隱私。

- **PawAI Studio 統一介面**：以網頁介面降低操作門檻，讓非技術背景的家人亦能直接使用。

最終，PawAI 的差異化定位為：**在消費級價位下，同時具備四足機動、自然對話、多模態互動、隱私優先的居家陪伴與輔助守護平台**。

## 五、系統限制（可行性分析）

本節說明使用者可能期望、但因硬體能力、運算資源、模型表現、時程安排或倫理考量而未能完整納入本次系統範圍的需求，以及各項限制的成因與影響。部分限制已於開發過程中實際嘗試克服但最終未能達成可用水準，部分則為評估後主動排除於本次開發範圍之外，另有部分屬於必須搭配外部硬體到貨後才能啟用的條件式功能。

### 1. 自主導航避障

**使用者期望**：機器狗能在家中自由移動、主動避開家具、人員與小型障礙物，並在使用者呼叫時安全地走到指定位置；跨越房間、繞過沙發與餐桌均不應撞到物體或絆倒使用者。

**對使用者的重要性**：高。在實際居家或展示場景中，機器狗需要在有家具、人員、寵物的環境中移動。自主避障是所有衍生功能（如跟隨、巡邏、主動接近互動對象）的基礎前提，也是評審與一般使用者最常質疑的項目之一。

**未包含原因**：本專案在開發過程中實際建構了完整的避障程式，包含基於 D435 深度影像的近距離避障節點，以及基於 LiDAR 360° 安全防護的 `nav2_collision_monitor` 框架。軟體單元測試全數通過，但真機驗證階段遭遇多重硬體限制：

- **D435 安裝角度限制**：D435 攝影機安裝於機身前方、鏡頭角度偏上方，低於鏡頭高度的障礙物（如椅腳、門檻、散落於地面的小型物品）需近距離才會進入視野範圍。加上從偵測到機器狗完全停止的延遲鏈較長（包含軟體去抖動、WebRTC 通訊延遲、Go2 減速時間約 1 至 1.5 秒），三輪真機防撞測試均未通過，機器狗在偵測到障礙後仍因慣性撞上目標物。

- **Go2 內建 LiDAR 覆蓋率不足**：Go2 Pro 內建的 Hesai XT16 固態 LiDAR 實測有效點數僅約 18%（約 22 點 / 120 點），且發布頻率低於每秒 2 次，遠低於 Nav2 導航所需的每秒 10 次以上。有效點覆蓋亦受機身結構遮蔽。綜合導致 SLAM 地圖建構與路徑規劃無法穩定運行，此為 Go2 Pro 韌體層級的硬體設計限制，軟體無法克服。

- **外接 RPLIDAR A2M12 採購進行中**：基於上述限制，團隊評估以 SLAMTEC RPLIDAR A2M12（360° 全向覆蓋、每秒 10 次掃描頻率、16000 points/sec、USB 直連 Jetson）作為替代方案。可行性研究結論指出記憶體預算安全（額外 0.75 至 1.15 GB 增量）、CPU 用量需謹慎管理（`slam_toolbox` 預估 70%，導航進行時需關閉手勢辨識模組）、供電風險需併入評估。採購流程透過指導老師的國科會計畫變更進行，於本文件繳交時雷達到貨時程尚未確定。

- **四足里程計漂移風險**：即使外接 RPLIDAR 到貨並建立 360° 覆蓋，Go2 四足步態的里程計漂移（odom drift）仍可能導致 SLAM 失敗，此為四足機構固有的誤差積累問題。團隊已規劃雷達到貨首日的 bag 錄製驗證流程，若 10 秒內漂移超過 0.3 公尺則 SLAM 方案不可行。

**替代方案**：本次 Demo 場景採用預先圈定活動範圍、人工監控、搭配語音「停」指令與手勢 stop 作為主要安全機制。若外接 RPLIDAR 到貨且里程計漂移驗證通過，自主導航避障功能將於後續版本正式啟用；若無法到貨或漂移過大導致 SLAM 失敗，則自主導航維持為文件級的未來工作，改以 360° 反應式避障（不依賴 SLAM 建圖）作為最低備援。

### 2. 機器狗本體語音對話

**使用者期望**：直接對機器狗說話即可互動，不需要額外的裝置、手機或網頁介面，就像和寵物或家人說話一樣直覺。

**對使用者的重要性**：高。這是自然互動的核心體驗之一，若互動必須透過中介裝置（筆電、手機），會顯著降低陪伴感與臨場感。

**未包含原因**：Go2 機器人內建散熱風扇持續運轉並產生寬頻噪音，安裝於機身上的 USB 麥克風訊噪比（Signal-to-Noise Ratio, SNR）嚴重不足。實測中，機身麥克風在安靜環境下的語音辨識正確率約為 80%，但在機器狗啟動、風扇持續運轉的條件下，正確率驟降至 20% 左右，低於可用門檻。團隊嘗試調高麥克風增益（mic_gain 從 4.0 放大至 8.0、12.0），但背景噪音同步放大、SNR 並未改善；亦嘗試了能量式語音活動偵測（Energy VAD）與 Whisper 幻覺黑名單過濾等軟體手段，皆無法克服物理層級的 SNR 劣化。此為風扇與麥克風配置的硬體特性，非軟體調校可解決。

**替代方案**：將語音互動入口從機身麥克風移至 PawAI Studio 網頁介面的按住說話（push-to-talk）功能，實測端到端延遲約 2 秒，辨識率與音質皆大幅改善。此方案犧牲了「直接對機器狗說話」的直覺性，但換來可靠的辨識率。長期解決方案需改用指向性麥克風陣列、主動降噪演算法（如 RNNoise、DeepFilterNet）或將麥克風移至遠離風扇的位置。

### 3. 多人同時互動

**使用者期望**：多人同時與機器狗互動，每個人都能被獨立辨識並得到個別回應；一家人可以同時跟機器狗對話、做不同手勢動作，機器狗能分辨誰對它發出指令並針對該對象採取差異化反應。

**對使用者的重要性**：中。展示場景經常有多人圍觀，若機器狗只能與一人互動會降低展示效果；居家場景中一家多口同時存在是常態。

**未包含原因**：人臉辨識模組採用 IOU 追蹤演算法，可支援最多 5 人同時追蹤並為各自維持獨立身份與追蹤編號（track_id）。然而，手勢辨識與姿勢辨識模組（均基於 Google MediaPipe）目前僅支援單人推理：MediaPipe Pose 預設只處理畫面中最顯著的一位人員，MediaPipe Gesture Recognizer 雖支援雙手但無跨人關聯機制。更關鍵的是，感知模組之間缺乏統一的 person_id 關聯機制，當多人同時出現時，系統無法判斷某個手勢或姿勢歸屬於哪位已辨識的使用者，也無法根據身份採取差異化的互動反應（例如叫小孩名字而非大人）。

**替代方案**：現階段多人場景下，系統採用「最靠近」原則自動選擇主要互動對象，手勢與姿勢僅對該對象生效。完整的多人互動支援需重新設計跨模組的 person_id 關聯架構，將所有視覺模組的偵測結果綁定至統一的身份，並在中控層做跨模態融合，預估工作量超過一個完整的 Sprint，無法納入本次開發時程，列為未來工作。

### 4. 多語言對話支援

**使用者期望**：除繁體中文外，也能以英文、台語、客語、粵語或其他方言與機器狗對話；部分年長使用者更習慣以母語方言交流。

**對使用者的重要性**：中。在國際展示、多元文化家庭或跨世代溝通的場景下有此需求。

**未包含原因**：本系統的語音辨識（SenseVoice Cloud / SenseVoice Local / Whisper）、語音合成（edge-tts / Piper）與大型語言模型（Qwen2.5-7B-Instruct）雖然多數模型原生支援多語言（Whisper 支援 99 種語言），但各模型的表現仍需個別驗證：SenseVoice 的優勢在中文與少數亞洲語言；edge-tts 的中文聲音風格與英文聲音不一致；LLM 的系統提示（system prompt）、意圖規則與規則備援（RuleBrain）皆以繁體中文撰寫；台語、客語等方言則缺乏高品質開源模型。擴充多語言支援需逐一驗證各環節的辨識率、合成品質與回覆風格一致性，在本次有限的開發時程下優先確保主線繁體中文互動的穩定性。

**替代方案**：無。本次開發範圍明確限定繁體中文。多語言支援列為未來工作，優先順序低於多輪對話記憶與遠端存取等其他需求。

### 5. 多輪對話記憶與個人化

**使用者期望**：機器狗能記住過往對話內容、認識每位家人的偏好與習慣、隨時間累積互動經驗；使用者期望聽到「上次你說過⋯」「我記得你喜歡⋯」「你今天看起來很累，要不要休息一下」這類個人化、連貫性的回應。

**對使用者的重要性**：中。這是情感陪伴型機器人與純功能性語音助理的核心差異化特色，也是現有產品（Aibo、Alexa、Siri）共同缺乏的能力。

**未包含原因**：

- **無狀態 LLM 呼叫**：當前大型語言模型呼叫採用無狀態模式，每次對話獨立送入雲端伺服器，不攜帶歷史上下文。

- **回覆長度硬截斷**：為避免 Go2 Megaphone 或 USB 喇叭播放過長音訊導致互動節奏拖沓，LLM 回覆被硬截斷至約 25 個中文字，無法展開詳細對話或多輪推理。

- **本地備援模型能力有限**：Jetson 本地備援採用 Qwen2.5-1.5B（相較雲端 7B 參數量少 4 倍），在語意理解、JSON 格式輸出、記憶檢索等任務上的能力遠低於雲端主線。

- **記憶框架整合時程不足**：團隊評估過 Mem0（per-person 長期記憶框架）、LangChain Memory、自建向量資料庫等方案，但整合、測試與真機驗證的時程無法在本次開發週期內完成。

**替代方案**：目前在 PawAI Studio 聊天面板顯示當前 session 的對話歷史，供使用者自行回顧，但離線後即消失。長期記憶列為第一優先的未來工作，優先順序高於多語言支援。

### 6. 自動充電與供電穩定性

**使用者期望**：機器狗在電量不足時能自動返回充電座、全天候無人值守運行；使用者不需要手動監控電池狀態或搬動機器狗至充電位置。

**對使用者的重要性**：中。居家陪伴場景需要機器狗能長時間自主運作，充電動作本身也是展示「智慧行為」的重要情境。

**未包含原因**：

- **無自動對接充電樁**：Go2 Pro 硬體不具備自動對接充電座的機械結構與感測器，Unitree 原廠亦未提供相關的充電樁方案。自動充電需要結合 SLAM 精確定位、視覺標記識別、機械對接機構三項能力，均不在本次開發範圍。

- **Jetson 外接供電不穩**：Jetson Orin Nano 的供電依賴外部 XL4015 降壓模組從 Go2 電池取電（28.8 伏特降壓至 19 伏特）。穩態負載下運作正常（穩態約 10 瓦），但在高負載時段（多模組同時推理可達 15 瓦尖峰）或 Go2 行走過程中的震動環境下偶發電壓不穩，Sprint 期間累計發生多次 Jetson 因電壓下降而強制重啟的情形。團隊嘗試將輸出電壓從 18.8 伏特調整至 19.2 伏特稍有改善但未根治，此為嵌入式系統的基礎設計問題。

- **電池續航限制**：Go2 Pro 滿電續航約 2 至 4 小時，不足以支持全日運行。

**替代方案**：展示期間採人工監控供電狀態，並於 Jetson 附近配置備援電源供緊急使用。長期解決方案需升級降壓模組（更高電流容量的 LM2596HV 或 DC-DC buck 轉換器）或為 Jetson 配置獨立的 18650 電池組。自動充電功能需等待 Unitree 官方或第三方提供充電樁方案後才可能實作。

### 7. 物體互動與搬運

**使用者期望**：使用者可告訴機器狗「幫我拿水杯」「把遙控器拿過來」，機器狗能辨識物品位置、靠近物品並取回。

**對使用者的重要性**：低（屬於進階功能期望，多數使用者並不預期機器狗能執行物理操作）。

**未包含原因**：Go2 Pro 為純四足機器人，機身無機械手臂結構，物理上無法抓取或搬運物品。即使外掛手臂也會顯著影響機器狗的重心平衡與續航時間，並增加整體控制複雜度；Unitree 原廠亦未提供官方手臂模組。本系統的物體辨識模組定位為情境互動提示（看到水杯時詢問「你要喝水嗎？」），而非執行實際的物理操作。

**替代方案**：物體辨識轉向情境提醒功能，運用視覺感知提升對話的情境豐富度。例如看到水瓶提醒喝水、看到書本表示關心閱讀情況、看到零食給予健康建議等。實際的物品取送交由人類使用者完成，機器狗作為「情境觀察者」而非「搬運工」。

### 8. 視覺辨識的邊界條件

**使用者期望**：機器狗能在各種光線、角度、距離條件下穩定辨識人臉、姿勢與物體，不論白天黑夜、近看遠看、側面正面皆應正常運作。

**對使用者的重要性**：中。居家場景中光線變化大（白天、夜晚、開關燈）、使用者姿勢多變、物體大小距離不一，辨識穩定性直接影響互動體驗。

**未包含原因**：本系統採用的深度學習模型（YuNet、SFace、MediaPipe Pose、MediaPipe Gesture Recognizer、YOLO26n）在理想條件下表現良好，但在邊界場景會出現顯著的精度下降：

- **光線敏感性**：所有視覺模型在低光源環境下精度下降（光線不足時手部關鍵點偵測率下降超過 30%）；D435 的紅外線結構光在直射陽光下會受干擾，深度資訊失準。

- **手勢距離限制**：MediaPipe 手部關鍵點偵測在 D435 前方約 2 公尺內才能穩定辨識，超出範圍時手指關節點的像素量不足以支援分類；近於 0.5 公尺時因超出最小對焦距離亦會失效。

- **姿勢角度依賴**：側面坐姿、蹲姿因 hip 角度與 trunk 角度計算偏差，分類結果不穩定；Demo 場景需要使用者面向相機以確保穩定度。

- **物體偵測小物件率**：YOLO26n（Nano 版本）對小物件（遠距離水瓶、平放書本、小型手機）偵測率偏低，已規劃升級至 YOLO26s（mAP +6.3%，參數量約 2 倍）作為 Sprint 後期的改進項目。

- **多人場景交叉辨識**：多人同時出現時手勢與姿勢模組可能追錯目標，需搭配人臉身份資訊才能穩定，此問題已於第 3 項多人互動中詳述。

- **跌倒誤判**：姿勢偵測的跌倒判定雖已加入 vertical_ratio 防護條件與 20 幀投票機制降低誤報，但在背景複雜（衣架、倒下的掃把、橫放的抱枕）的場景下仍可能鎖定非人物體誤判為跌倒。

**替代方案**：Demo 場景統一採用充足照明、預設使用者距離範圍（1 至 3 公尺）與面向相機的姿勢；將高風險偵測（特別是跌倒）的信心度閾值調高以降低誤報；Demo 前執行完整的 preflight check，實測各模組在該場地的表現並調整參數。

### 9. 遠端存取與行動端應用

**使用者期望**：家屬外出時能透過手機 App 查看家中狀況、接收機器狗偵測到的異常通知（如長輩跌倒、陌生人闖入）、遠端對機器狗下達指令或查閱事件歷史紀錄。

**對使用者的重要性**：中。這是現代智慧家電的標準功能，也是守護場景中不可或缺的一環：若家人發生意外，外出的家屬應立即收到通知，否則守護功能等同於失效。

**未包含原因**：

- **Studio 僅支援區域網路**：PawAI Studio 目前為網頁應用程式（Next.js + FastAPI Gateway），預設僅支援區域網路存取，不具備雲端 relay 或 NAT 穿透能力。

- **無帳號與身份驗證**：遠端存取需要完整的帳號系統、身份驗證機制（如 OAuth、JWT）與隱私保護設計，本次未實作。

- **無原生行動端應用**：行動端原生 App 需針對 iOS 與 Android 分別開發，並整合推播通知服務（Apple Push Notification Service、Firebase Cloud Messaging），工作量遠超本次開發範圍。

- **無推播通知整合**：跌倒、陌生人警戒等守護事件目前僅顯示於 Studio 事件面板，未推送至外部裝置。

**替代方案**：技術上可透過 VPN 服務（如 Tailscale、ZeroTier）從外部網路存取 PawAI Studio，但需使用者自行設定。本次 Demo 不展示遠端存取功能，守護類事件僅在同網段內的 Studio 面板呈現。行動端應用與推播整合列為產品化階段的核心工作項目。

### 10. 資料隱私與角色權限

**使用者期望**：家庭成員的人臉、語音、互動紀錄等敏感資料受到妥善保護；不同家人（父母、長輩、小孩、訪客）擁有不同的權限與個人化設定，例如小孩無法關閉守護功能、訪客僅能基本互動。

**對使用者的重要性**：中。居家場景中隱私特別敏感，尤其涉及長者或兒童時需要更嚴格的資料保護設計，部分國家的個資法規亦對此類資料有明確規範。

**未包含原因**：

- **人臉資料無靜態加密**：人臉資料庫（儲存於 `/home/jetson/face_db/`，包含每位使用者的 SFace 128 維特徵向量與姓名）目前以明文檔案儲存於 Jetson 本地，未實作靜態加密或存取權限控制。

- **語音雲端傳輸**：語音辨識與大型語言模型主線採用雲端服務，使用者語音會透過內部 SSH tunnel 傳送至遠端 RTX 8000 伺服器，雖為私有網段但未實作端對端加密。

- **無帳號與角色系統**：face_db 僅記錄使用者姓名，無「管理者 / 一般成員 / 訪客」角色分級，所有已註冊使用者對系統的操作權限相同。

- **無長期稽核日誌**：系統事件僅短期保留於 ROS2 topic 緩衝區，無持久化稽核日誌供事後追溯異常存取或誤操作。

**替代方案**：採用「感知全本地化」的設計原則——所有視覺感知（人臉、手勢、姿勢、物體）全在 Jetson 邊緣設備本地運行，不將影像上傳；人臉資料庫儲存於 Jetson 本地而非雲端；Demo 期間所有資料僅在區域網路內流動；語音雲端傳輸限於專屬 tunnel，不經公開網路。正式產品化階段需補足靜態加密、帳號系統、角色權限模型與完整稽核日誌等隱私保護機制。

### 11. 戶外使用與特殊環境

**使用者期望**：把機器狗帶到公園散步、陽台曬太陽、社區活動場合使用；機器狗應能適應各種戶外光線、地形與天氣條件。

**對使用者的重要性**：低。本專題定位為居家互動機器狗，戶外使用屬於衍生需求，但部分使用者仍有此期待。

**未包含原因**：

- **防水能力不足**：Go2 Pro 原廠防護等級為 IP54（防塵、防潑水），但不足以應對持續雨水；外掛的 Jetson Orin Nano、D435 攝影機與 XL4015 降壓模組均無防水設計，戶外遇雨即損壞風險極高。

- **續航不足**：Go2 Pro 滿電續航約 2 至 4 小時，不支援長時間戶外活動。

- **定位依賴**：戶外場景需要全球定位（GPS），本系統未整合 GNSS 模組。

- **網路依賴**：雲端語音辨識與大型語言模型需要穩定網路連線，戶外 4G 聯網需要額外的 SIM 卡與服務方案。

- **感知模組需重新調校**：戶外光線（直射陽光）、背景噪音、移動範圍等條件與居家環境顯著不同，各感知模組的參數均需重新驗證與調校。

**替代方案**：無。戶外場景明確排除於本次開發範圍之外，僅在受控的室內環境進行 Demo。

### 12. 情緒辨識與主動關懷

**使用者期望**：機器狗能察覺使用者情緒變化（開心、難過、焦慮、疲倦、憤怒）並主動提供關懷或調整互動方式，例如看出使用者情緒低落時主動靠近、播放輕快音樂或提出關心話題。

**對使用者的重要性**：中。這是情感陪伴型機器人的核心差異化特色，也是區分「功能性助理」與「情感陪伴夥伴」的關鍵能力。

**未包含原因**：

- **無情緒辨識模型**：面部情緒辨識（Facial Expression Recognition, FER）模型尚未整合至視覺管線；語音語調情緒分析需要額外的專屬模型訓練。

- **辨識準確度疑慮**：情緒辨識本身受文化差異、年齡、個體特質與資料多樣性影響，開源模型在東亞面孔與複雜表情上的準確度有限，誤判風險高。

- **倫理與隱私界線**：過度解讀使用者情緒可能引發隱私疑慮，尤其在居家場景中針對長輩或兒童使用時；主動「關懷」行為若基於錯誤的情緒判讀可能造成反效果甚至心理壓力。

- **整合複雜度**：情緒訊號需融合面部、語音、姿勢等多模態資訊，並與 LLM 的對話內容協同，系統整合複雜度高。

**替代方案**：目前系統透過對話內容關鍵字間接推測使用者狀態（例如使用者提到「累」「不舒服」「想睡」時觸發關懷式回應），屬於淺層規則判斷而非真正的情緒辨識。完整的情緒感知與倫理評估列為未來工作，優先度低於多輪對話記憶與遠端存取。

### 13. 開發時程與工程債務

**使用者期望**：系統運作穩定可靠、符合規格描述、所有功能在展示時皆能順利運行而不當機或失敗；文件與程式碼實作完全一致。

**對使用者的重要性**：高。從評審與使用者角度而言，「能不能跑」比「功能多寡」更重要。一個穩定展示 80% 功能的系統優於不穩定展示 100% 功能的系統。

**未包含原因**：

- **文件與實作同步差距**：本專題的核心設計規格文件（如 2026-04-11 PawAI Brain 三層架構 spec）已更新至最新版本，但實作仍在持續追趕中，個別模組行為可能與最新規格有微小差異，例如 PawAI Brain 的完整 Skill Queue 機制規劃中但尚未完全實作。

- **整合測試覆蓋率有限**：目前以手動 Demo 驗證為主，自動化整合測試（含 CI pipeline 契約測試、端到端場景測試、壓力測試）覆蓋率約為核心邏輯的 60%，跨模組修改後仍需人工回歸驗證，無法完全保證修改的安全性。

- **備案劇本（Plan B）依賴**：為確保 Demo 場面穩定，團隊針對 GPU 伺服器斷線、雲端 API 限流、網路中斷等極端場景設計了固定台詞備案（Plan B 固定回覆約 15 組）。此備案可在極端情況下維持 Demo 流暢度，但無法提供完整的 AI 對話體驗，僅作為最壞情況的安全網。

- **硬體供電風險**：如第 6 項所述，Jetson 供電穩定性於 Demo 期間仍為已知的最大硬體風險項目，無法於文件繳交前根治。

**替代方案**：開發團隊於 Sprint 期間透過每日站會同步進度、每週跨模組對齊會議、重要路徑的 pair programming 降低整合風險；Demo 前執行完整的 dry run 與 preflight check 流程驗證；語音對話閉環、人臉問候、跌倒偵測等關鍵路徑均已於真機上多次驗收通過。Demo 當天配置專人監控系統狀態並準備 Plan B 腳本。

# 第三章 軟體設計規格

## 一、資料庫設計

### 使用之資料庫說明

本系統**不使用傳統關聯式資料庫（RDBMS）**，因為系統的核心功能是即時感知與互動，資料以串流形式在 ROS2 節點之間傳遞，不需要持久化儲存。系統中的資料儲存方式如下：

| 資料類型 | 儲存方式 | 說明 |
|----------|----------|------|
| 人臉資料庫 | 檔案系統 + pickle | 已註冊使用者的人臉照片與特徵向量 |
| 感知事件 | ROS2 Topics（JSON） | 即時串流，不持久化 |
| 前端狀態 | Zustand in-memory | 瀏覽器端狀態管理，關閉即消失 |

### 人臉資料庫結構（NoSQL — 檔案系統）

**儲存位置**：`/home/jetson/face_db/`

**目錄結構**：
```
/home/jetson/face_db/
├── roy/
│   ├── image1.png
│   ├── image2.png
│   └── ...
├── grama/
│   ├── image1.png
│   └── ...
└── model_sface.pkl    ← 自動生成的特徵模型
```

**model_sface.pkl 資料結構（JSON 範例說明）**：
```json
{
  "embeddings": {
    "roy": [[0.12, -0.34, ...], [0.15, -0.31, ...]],
    "grama": [[0.08, -0.42, ...]]
  },
  "centroids": {
    "roy": [0.135, -0.325, ...],
    "grama": [0.08, -0.42, ...]
  },
  "counts": {
    "roy": 3,
    "grama": 2
  }
}
```

- `embeddings`：每位使用者的所有人臉特徵向量（128 維 float32 陣列）
- `centroids`：每位使用者特徵向量的平均值，用於快速比對
- `counts`：每位使用者的註冊照片數，用於偵測資料庫變更並自動重新訓練

**註冊流程**：將使用者的人臉 PNG 照片放入以其名字命名的資料夾，系統啟動時自動偵測、提取特徵並生成 pickle 模型。若照片數量有變動，系統自動重新訓練。

### ROS2 感知事件資料結構（JSON over Topics）

**PawAI Event 通用信封格式**（Studio Gateway 發送至前端）：
```json
{
  "id": "evt_1712345678_face",
  "timestamp": "2026-04-07T10:30:00Z",
  "source": "face",
  "event_type": "identity_stable",
  "data": {
    "track_id": 1,
    "stable_name": "roy",
    "sim": 0.42,
    "distance_m": 1.8
  }
}
```

# 第四章 系統專題實作檢討

## 一、發展中遭遇到問題、困難與解決方法

### 1. WebRTC 連線相容性問題

**問題**：aiortc 套件自動升級至 1.14.0 後，SCTP 握手超時（30 秒以上），DataChannel 無法開啟，機器狗完全失聯。
**解決方法**：降級 aiortc 至 1.9.0 並移除 STUN 配置，SCTP 握手時間降至 0.5 秒。後續將版本鎖定寫入 requirements.txt。
**教訓**：依賴套件版本必須鎖定，自動升級在嵌入式環境極易造成不可預期的相容性問題。

### 2. Go2 Megaphone 音訊播放 API 格式陷阱

**問題**：團隊花費約兩週判定 Megaphone API「已失效」，無法從機器狗喇叭播放語音。根因為 payload 格式錯誤 — chunk_size 必須為 4096 base64 字元、msg type 必須為 "req" 而非 "msg"，官方文件未記載。
**解決方法**：透過逆向分析 Go2 韌體通訊協議找出正確格式，修正後連續 20/20 測試通過。
**教訓**：硬體 API 文件不完整時需逆向驗證，優先懷疑格式而非硬體。

### 3. Go2 Pro 有線模式架構限制

**問題**：嘗試建立 Ethernet 控制通道以降低延遲，連線成功後卻完全沒有 ROS2 topic 資料流。Go2 Pro 內部 Wi-Fi 與有線網段隔離，有線只能到達 MCU。
**解決方法**：放棄有線模式，維持 Wi-Fi WebRTC。
**教訓**：硬體架構設計決定軟體的可能性邊界，應在投入開發前先確認硬體能力。

### 4. SLAM 建圖與 Nav2 導航的多重挫折

**問題**：經歷 Nav2 參數不生效（未 colcon build 同步到 install/）、/scan 全為 inf（intensity 過濾過嚴）、Jetson 點雲頻率掉至 2 Hz（多實例 driver + 低效序列化，經七輪優化恢復至 5.1 Hz）、D435 鏡頭角度導致低障礙物看不到（三輪真機防撞測試全部撞上）等問題。
**解決方法**：導航避障功能完全停用。程式碼保留（含 20 個 unit tests），Demo 改為人工監控 + 手勢 stop。
**教訓**：導航涉及硬體（鏡頭角度、LiDAR 覆蓋率）、網路（WebRTC 延遲）、演算法（SLAM 頻率）三重限制，任一環節不達標整個功能即不可用。

### 5. 多重網路跳轉導致控制延遲

**問題**：開發初期透過 Go2 (Wi-Fi) → Mac VM (橋接) → Windows (有線) 多層跳轉進行 Nav2 控制，延遲 20-50ms 接近控制週期（50ms），指令積壓導致機器狗原地旋轉。
**解決方法**：改為 Windows 本地安裝 ROS2 + RViz2（零延遲控制），Foxglove 僅用於監控。
**教訓**：即時控制應減少網路跳轉層數，選對工具比調參數更重要。

### 6. 模型選型三次推翻

**問題**：手勢/姿勢辨識模型選型經歷三次推翻：RTMPose wholebody（GPU 滿載）→ DWPose（TensorRT 精度暴跌 + MMPose JetPack 6 零支援）→ MediaPipe（原判定不可行，實測推翻，CPU 即時推理）。
**解決方法**：最終採用 MediaPipe CPU-only 方案，手勢可達即時推理速度，GPU 佔用 0%。
<!-- source: docs/手勢辨識/research/選型過程.md, docs/姿勢辨識/README.md -->
**教訓**：文獻調查不能替代實測，Jetson 生態碎片化嚴重。

### 7. ASR 三級 Fallback 的逐步演進

**問題**：Whisper Small 中文短句+噪音辨識率已達瓶頸，無法滿足 Demo 需求。
**解決方法**：引入 SenseVoice 雲端（辨識率大幅提升）+ SenseVoice 本地（離線備援）形成三級降級。另實作 10+ 項 workaround（幻覺過濾 22 pattern、短文字過濾、Echo Gate 等）。
<!-- source: docs/mission/sprint-b-prime.md Day 2 -->
**教訓**：語音辨識在真實噪音環境下表現遠低於實驗室數據，需多層防禦。

### 8. 跌倒偵測正面站姿誤判

**問題**：使用者正面站在鏡頭前時被錯誤判定為「跌倒」（肩膀展開導致 bbox 寬高比觸發條件）。
**解決方法**：新增 vertical_ratio 防護條件（肩膀到髖部垂直距離 / 軀幹長度），相對尺度不受距離影響。
<!-- source: docs/姿勢辨識/README.md, sprint-b-prime.md Day 3-6 -->
**教訓**：基於 bbox 的簡單規則在邊界情況易失效，需多維度防護。

### 9. Jetson 環境污染事件

**問題**：在 Jetson 上執行 `pip install ultralytics`，自動拉入不相容的 PyTorch + NumPy 版本，整個 Python 環境崩潰。
**解決方法**：環境修復耗時半天。建立規範：Jetson 嚴禁直接 pip install 大型框架，正確做法是開發機匯出 ONNX 再 scp 到 Jetson。
<!-- source: docs/mission/sprint-b-prime.md Day 9 -->
**教訓**：邊緣裝置 Python 環境極脆弱，任何可能覆蓋系統 wheel 的安裝都是高風險。

### 10. LLM 視覺避障的根本局限

**問題**：早期嘗試透過 LLM 分析攝影機影像進行避障。LLM 能識別障礙物且建議轉向正確，但無法判斷「機器狗自身大小，能否通過」。
**解決方法**：評估多個 LLM 模型後認定純視覺方案只適合粗略避障，精確導航仍需 SLAM + 深度感測。
<!-- source: docs/archive/logs/ -->
**教訓**：LLM 視覺理解有根本局限，不能完全取代傳統感測器。

### 11. ROS2 QoS 策略不匹配

**問題**：多個模組間因 QoS 策略不一致導致訊息無法接收（RELIABLE subscriber 無法接收 BEST_EFFORT publisher）。
**解決方法**：統一感測器類 topic 使用 BEST_EFFORT QoS。
**教訓**：多模組開發時 QoS 策略需在介面契約中明確規範。

### 12. Jetson 供電危機

**問題**：Sprint 11 天期間 Jetson 反覆被強制關機（Go2 電池 → XL4015 降壓 → Jetson，高負載 spike 超出降壓模組能力）。
**解決方法**：調整電壓 18.8V → 19.2V 稍有改善但未根治。至 Sprint 結束時仍為最大硬體風險。
<!-- source: docs/mission/sprint-b-prime.md Day 4, 9, 10, 11 -->
**教訓**：嵌入式系統供電設計需考慮負載峰值而非僅平均功耗。

### 13. Go2 風扇噪音導致機身 ASR 不可用

**問題**：在 Go2 真機上 ASR 正確率因散熱風扇寬頻噪音驟降至不可用水準。調高 mic_gain 無效（噪音同步放大）。
**解決方法**：策略轉向 — Demo 改為「視覺互動為主 + 網頁語音輔助」，語音入口從機身移到 PawAI Studio 瀏覽器端。
<!-- source: docs/mission/sprint-b-prime.md Day 6, Day 11 -->
**教訓**：物理環境限制無法透過軟體增益解決，需從系統架構層面尋找替代方案。

### 14. SSH Tunnel 與遠端 GPU Server 連線

**問題**：雲端 ASR/LLM 依賴 SSH tunnel 連接 RTX 8000，tunnel 不穩定導致全 timeout。多個子問題疊加（Port 不匹配、Server 未啟動、缺少依賴、async blocking）。
**解決方法**：推 async fix 到 Server 並重啟；Jetson 建立 systemd user service 永久化 SSH tunnel（開機自動起）。
**教訓**：遠端連線涉及多層級，需逐層驗證每個環節。

## 二、系統優缺點（SWOT）評估

| | 正面 | 負面 |
|---|------|------|
| **內部因素** | **優勢（Strengths）**：多模態感知管線完成度高（六大模組全部開發完成）；三級降級策略確保無網路時仍可基本互動（ASR/LLM/TTS 全有 fallback）；模組化 ROS2 架構使每個感知模組可獨立替換或升級；PawAI Studio 提供遠端觀測與操控能力，提升展示與開發效率。 | **劣勢（Weaknesses）**：Jetson 供電不穩為最大硬體風險；Go2 風扇噪音導致機身 ASR 不可用，需改由網頁端提供語音入口；人臉追蹤存在抖動問題尚未根治；手勢/姿勢辨識僅支援單人追蹤；OpenCV 版本鎖定無法享受新版本改進。 |
| **外部因素** | **機會（Opportunities）**：長者陪伴與智慧居家市場需求持續增長；邊緣 AI 硬體（如 NVIDIA Jetson 系列）持續降價，降低部署門檻；ROS2 生態日益成熟，可復用大量社群套件加速開發。 | **威脅（Threats）**：Jetson 生態碎片化嚴重（不同 JetPack 版本間套件不相容）；Go2 Pro 韌體開放程度有限（無 CycloneDDS 直連），限制了即時控制的可能性；競品如小米 CyberDog 2、Boston Dynamics Spot 擁有更多資源與更完整的生態系。 |

## 三、未來展望

基於系統限制章節的分析以及開發過程中的經驗，以下為後續可改進的方向：

1. **導航避障功能**：重新設計 D435 攝影機的安裝角度（建議俯視約 -45°），使低障礙物能在更遠距離被偵測到；或考慮升級至 Go2 EDU 版本以獲得 CycloneDDS 直連能力，降低控制延遲。

2. **機身語音互動**：採用指向性麥克風取代全向性 USB 麥克風，搭配軟體降噪演算法（如 RNNoise），改善機器狗本體的語音辨識環境。

3. **多人互動支援**：建立跨模組的使用者 ID 關聯機制，將人臉辨識的 track_id 與手勢/姿勢偵測結果綁定，實現多人場景下的個別互動。

4. **物體互動擴充**：若未來搭配機械手臂（如 WidowX），可實現「看到→理解→操作」的完整物體互動鏈路。在此之前，可先擴充語音引導互動（如「杯子在你的左前方」）。

5. **供電穩定化**：改用獨立的 18650 鋰電池組供電，或升級為更高品質的 DC-DC 降壓模組，確保 Jetson 在 Go2 行走時的供電穩定性。

6. **AI 大腦進化**：將目前的 rule-based Interaction Executive 升級為 LLM-driven 決策中樞，讓機器狗能基於多模態上下文進行更複雜的情境推理與行為規劃。

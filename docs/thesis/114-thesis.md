# 第一章 系統描述

## 一、發展背景與動機

### 研究背景

近年來，人工智慧技術發展迅速，特別是大型語言模型（Large Language Models, LLMs，如 Qwen、LLaMA）的突破，使得機器系統具備了深度的語意理解與邏輯推理能力。同時，邊緣運算與電腦視覺（如 MediaPipe 姿態辨識、YuNet 人臉偵測技術）以及語音處理技術（如 SenseVoice 語音辨識與 edge-tts 語音合成）的日益成熟，讓資訊系統能夠以多模態（Multi-modal）的方式，精準感知周遭環境與人類的真實意圖。
<!-- 修正：Sherpa STT → SenseVoice、MeloTTS → edge-tts、RTMPose → MediaPipe/YuNet、Claude → Qwen/LLaMA -->

在終端硬體載具方面，四足機器人（如仿生機器狗）的機電控制與運動靈活性已有顯著的提升。然而，綜觀目前市面上的實體機器人應用，多半仍依賴傳統的遙控設備、手機 App 或是僵化的預設腳本進行操作。現有系統缺乏將「雲端高階 AI 大腦」與「邊緣終端硬體（ROS2）」深度結合的解決方案，導致人機互動往往流於單向、死板的指令接收，機器無法根據人類自然的肢體語言（手勢）、語音指令或當下的環境脈絡，進行即時的動態感知與自主決策。

### 研究動機

基於上述背景，本專題「PawAI」旨在打破傳統實體機器人僵化且高門檻的互動模式。我們運用資訊管理領域的系統整合（System Integration）與軟體工程技術，開發一套無縫串接視覺感知、語音互動與大型語言模型決策的智能控制系統。

本研究的核心動機包含以下幾點：

1. **實現自然且直覺的人機互動**：透過建構完整的多模態感知管線（Multi-modal Perception Pipeline），讓系統能即時捕捉並解析使用者的手勢與臉部特徵；結合語音雙向處理，讓使用者能以最自然的「說話」與「動作」與機器狗互動，免除繁瑣的傳統控制介面。
<!-- 修正：視覺推論拓撲 → 多模態感知管線 -->

2. **賦予終端設備動態決策能力**：藉由串接大型語言模型 Qwen2.5-7B-Instruct（透過 vLLM 部署於遠端 RTX 8000 伺服器）作為系統的核心邏輯中樞，將接收到的多模態感知資料轉化為結構化的意圖（Intent），讓機器狗不再只是單純執行寫死的程式碼，而是能「理解」複雜情境並規劃後續行為。
<!-- 修正：Claude API → Qwen2.5-7B-Instruct (vLLM on RTX 8000) -->

3. **軟硬體整合與控制迴路優化**：透過建構穩定的 ROS2（Robot Operating System 2）節點通訊與統一中控模組（Interaction Executive），確保 AI 模型的決策結果能以極低的延遲、安全且精確地轉化為實體機器狗的具體動作，完成從「感知」、「思考」到「行動」的完整閉環。
<!-- 修正：ralph-loop → Interaction Executive -->

總結而言，本專題期望透過 PawAI 系統的建置，探索大型語言模型結合實體機器人的無限可能，不僅為智慧陪伴、巡檢或展演等應用場景提供具高度擴充性的架構範例，更進一步實現真正意義上的智慧人機共作（Human-Robot Collaboration）。

## 二、系統發展目的

### 問題與解決方案對應表

| 現有問題 | 本系統之解決方法 |
|----------|------------------|
| 實體機器人多依賴遙控器或手機 App 操作，互動方式單向且僵化。 | 整合 MediaPipe 手勢/姿勢辨識與語音模組，實現自然直覺的多模態（手勢、語音、人臉）互動。 |
| 機器人僅能執行預先寫死的腳本（Hard-coding），缺乏情境理解與自主決策能力。 | 串接大語言模型（Qwen2.5-7B-Instruct）作為核心大腦，依據當下脈絡動態生成行為回饋。 |
| 雲端 AI 模型與邊緣硬體設備之間，缺乏穩定、低延遲的整合與執行機制。 | 開發基於 ROS2 的統一中控模組（Interaction Executive），精準且安全地驅動終端硬體作動。 |
| 傳統機器人系統架構較為封閉，難以輕易擴充新功能或進行系統除錯。 | 建立模組化感知管線與 PawAI Studio 網頁控制台，確保系統具備高擴充性與可觀測性。 |
<!-- 修正：RTMPose → MediaPipe、ralph-loop → Interaction Executive、Audit 稽核日誌 → PawAI Studio 可觀測性 -->

### 功能性需求

**(一) 整合多模態感知技術，提供自然直覺的人機互動體驗**

打破傳統單向且受限的控制介面，本系統導入 MediaPipe 模型進行即時的姿態與手勢推論，並輔以 YuNet + SFace 人臉辨識技術鎖定互動對象。同時，結合 STT（語音轉文字）與 TTS（文字轉語音）模組，讓使用者能以最日常的肢體動作與口語對話，與系統進行雙向、流暢的互動。

**(二) 串接大語言模型，建構具備情境理解的自主決策中樞**

為賦予機器狗動態決策的能力，本專案將前端擷取的多模態感知資料轉化為結構化的意圖（Intent），並透過 HTTP API 串接部署於遠端伺服器的大語言模型（Qwen2.5-7B-Instruct）。藉此，系統能深入理解使用者的語意與當下的互動脈絡，取代傳統預先寫死的控制腳本，自主規劃並生成最合適的行為回饋。
<!-- 修正：Claude API → Qwen2.5-7B-Instruct via HTTP -->

**(三) 開發 ROS2 統一中控，實現低延遲且穩定的硬體作動閉環**

為確保雲端 AI 的決策能精確落實於邊緣實體設備，本系統建構了基於 ROS2 的節點通訊機制。透過統一中控模組 Interaction Executive（以 State Machine 實現事件路由與優先序仲裁），系統能將抽象的行為指令，以極低的延遲轉譯為機器狗能直接執行的動作控制訊號，達成從「感知」、「運算」到「作動」的無縫銜接。
<!-- 修正：ralph-loop → Interaction Executive (State Machine) -->

**(四) 建立 PawAI Studio 網頁控制台，確保系統的高可觀測性與可維護性**

導入軟體工程的模組化設計思維，建立 PawAI Studio 作為系統的統一觀測與操控入口。Studio 整合即時感知面板（人臉、手勢、姿勢、物體、語音狀態）、語音/文字互動介面、以及即時影像串流，方便團隊進行後續的效能監控與錯誤排查，更保留了未來擴增功能的彈性。
<!-- 修正：Audit 稽核日誌 + 視覺推論拓撲 → PawAI Studio 可觀測性 -->

### 非功能性需求

- **低延遲**：語音端到端（E2E）互動延遲目標 < 5 秒（含 ASR + LLM + TTS）
- **高可用性**：三級降級策略（雲端 → 本地 → 規則），確保無網路時仍可基本互動
- **模組化**：每個感知模組為獨立 ROS2 package，可單獨替換或升級
- **安全性**：統一中控模組仲裁所有動作指令，緊急事件（跌倒）具最高優先序

## 三、系統範圍

本節根據使用者的互動情境，盤點系統的功能性需求。以下各功能模組構成 PawAI 系統的完整範圍，讓讀者在閱讀第二章需求規格前，對系統功能有概略性的了解。

### (一) 語音互動模組

系統支援使用者透過語音與機器狗進行自然對話。語音入口包含機器狗本體的 USB 麥克風（持續監聽模式）以及 PawAI Studio 網頁端的按住說話（push-to-talk）模式。

主要功能：
- **語音辨識（ASR）**：將使用者的中文語音即時轉換為文字，採用三級自動降級機制（SenseVoice 雲端 → SenseVoice 本地 → Whisper 本地）
<!-- source: docs/語音功能/README.md -->
- **意圖分類（Intent）**：對辨識出的文字進行意圖分析，高信心指令（如「坐下」「停」）直接映射為動作，一般對話送入 LLM
- **AI 對話（LLM）**：透過大語言模型（Qwen2.5-7B-Instruct）根據對話語境生成簡短合理的回覆
- **語音合成（TTS）**：將回覆文字合成為自然語音，採用 edge-tts 雲端合成為主、Piper 本地合成為備援
- **音訊播放**：透過 USB 外接喇叭或 Go2 Megaphone DataChannel 播放語音回覆

### (二) 人臉辨識模組

系統能即時偵測畫面中的人臉，辨識已註冊使用者的身份，並觸發個人化的問候互動。

主要功能：
- **人臉偵測**：使用 YuNet 模型即時偵測 RGB 影像中的人臉位置
<!-- source: docs/人臉辨識/README.md -->
- **身份辨識**：使用 SFace 模型提取人臉特徵向量，與資料庫中已註冊的使用者進行比對
- **多人追蹤**：透過 IOU（Intersection Over Union）演算法追蹤多張人臉（最多 5 人），各自獨立進行身份辨識
- **身份穩定化**：透過 Hysteresis 三層穩定機制，避免單幀抖動導致的身份誤判
- **問候互動**：辨識出已註冊使用者後，觸發 Interaction Executive 發送個人化 TTS 問候

### (三) 手勢辨識模組

系統能辨識使用者的手勢，並將特定手勢映射為機器狗的控制指令。

主要功能：
- **手勢偵測與分類**：使用 MediaPipe Gesture Recognizer 即時辨識手勢類別（如 thumbs_up、open_palm、stop 等）
<!-- source: docs/手勢辨識/README.md -->
- **動作映射**：將辨識出的手勢轉譯為對應的機器狗動作指令（如：張開手掌 → 停止移動、比讚 → 開心動作）
- **去重機制**：同一手勢在 5 秒冷卻期內不重複觸發

### (四) 姿勢辨識模組

系統能即時辨識使用者的身體姿勢，用於跌倒偵測等安全監護場景。

主要功能：
- **姿勢分類**：使用 MediaPipe Pose 辨識人體 33 個關鍵點，分類為 standing、sitting、crouching、lying、fallen 等狀態
<!-- source: docs/姿勢辨識/README.md -->
- **跌倒偵測**：當偵測到 fallen 狀態時，觸發緊急警報（EMERGENCY），機器狗發出語音提醒
- **持續狀態廣播**：以事件（Event）形式發布姿勢變化，供 Interaction Executive 和 PawAI Studio 訂閱

### (五) 物體辨識模組

系統能即時辨識畫面中的物體，並對特定物體觸發情境互動。

主要功能：
- **物體偵測**：使用 YOLO26n 模型搭配 TensorRT FP16 加速，即時偵測 COCO 80 類物體
<!-- source: docs/辨識物體/README.md -->
- **情境互動**：對高價值物體（如水杯、水瓶、書本）觸發 TTS 互動話術（例如看到杯子說「你要喝水嗎？」）
- **類別篩選**：可透過 `class_whitelist` 參數動態調整偵測範圍

### (六) PawAI Studio 網頁控制台

PawAI Studio 是系統的統一觀測與操控入口，讓操作者（開發人員、展示人員、教授）能透過瀏覽器遠端監看與互動。

主要功能：
- **即時感知面板**：同時顯示人臉、手勢、姿勢、物體、語音五大模組的即時狀態
- **文字與語音互動**：透過聊天面板輸入文字指令，或按住麥克風錄音進行語音互動
- **即時影像串流**：三欄即時影像（人臉偵測畫面 / 姿勢手勢畫面 / 物體偵測畫面）
- **Gateway 橋接**：FastAPI + WebSocket 橋接 ROS2 感知事件至瀏覽器

### (七) 統一中控模組（Interaction Executive）

Interaction Executive 是系統的事件路由中樞，負責聚合所有感知事件並仲裁動作分派。

主要功能：
- **狀態機**：管理 IDLE → GREETING → CONVERSING → EXECUTING → EMERGENCY 狀態轉移
<!-- source: interaction_executive/interaction_executive/state_machine.py -->
- **優先序仲裁**：EMERGENCY（跌倒）> obstacle > stop > speech > gesture > face
- **事件去重**：同一事件在冷卻期內不重複處理（greet 30s、gesture 5s、object per-class 5s）
- **動作分派**：將決策結果透過 `/tts` 和 `/webrtc_req` topic 分別送至 TTS 和 Go2 動作執行

## 四、背景知識

### (一) 機器人作業系統（ROS2）

#### 什麼是 ROS2

機器人作業系統（Robot Operating System, 簡稱 ROS）並非傳統意義上的作業系統，而是一個為機器人軟體開發所設計的彈性框架（Framework）。ROS2 作為新一代的版本，提供了硬體抽象化、底層設備控制、常用功能實作以及行程間的訊息傳遞（Message Passing）機制。在本專案中，ROS2 扮演了連接高階 AI 決策模型與終端實體機器狗（Unitree Go2）底層動作控制的關鍵橋樑。

#### ROS2 節點通訊機制與架構

ROS2 採用分散式架構，系統中的各個功能模組被封裝為獨立的「節點（Nodes）」。節點之間透過發布者/訂閱者（Publisher/Subscriber）模型進行非同步的資料交換。本系統利用此特性開發統一中控模組 Interaction Executive，確保決策指令能以低延遲的方式安全傳遞至硬體端。

### (二) Unitree Go2 四足機器人平台

#### Go2 簡介

Unitree Go2 是由中國宇樹科技（Unitree Robotics）於 2023 年發布的消費級四足機器人，定位為「人人可負擔的機器狗」。相較於 Boston Dynamics Spot（售價約 75,000 美元），Go2 以約 1,600 美元的價格提供類似的基礎功能，大幅降低四足機器人的研究門檻。

#### 硬體規格

Go2 提供三種版本：Go2 Air、Go2 Pro 與 Go2 EDU。

- **Go2 Air**：售價約 1,600 美元，不配備 LiDAR 與 4G 模組，僅提供基礎 APP 控制，連線方式為 Wi-Fi。
- **Go2 Pro**：售價約 2,800 美元，配備 Hesai XT16 LiDAR 與 4G 模組，提供基礎 APP 控制，連線方式為 Wi-Fi。**本專題使用此版本**。
- **Go2 EDU**：售價約 8,500 美元，除了配備 LiDAR 與 4G 模組外，還搭載 NVIDIA Jetson Orin NX（8GB）處理器，提供完整 ROS2 SDK 支援與 CycloneDDS 直連，連線方式支援 Wi-Fi 與 Ethernet。

本專案採用 Go2 Pro 版本，透過 WebRTC DataChannel 作為控制通道。相較 Go2 EDU 版本可透過 CycloneDDS 直連，Pro 版本的 WebRTC 路徑存在較高延遲，但成本較低且足以支援本專案的互動場景。

### (三) 電腦視覺與姿態辨識技術

#### MediaPipe 框架簡介

MediaPipe 是 Google 開發的一個專為即時串流媒體與多模態感知設計的機器學習框架。其最大的優勢在於高度最佳化的管線（Pipeline）架構，能在不依賴龐大 GPU 算力的情況下，於一般 CPU 或邊緣運算設備上實現極低延遲（Low-latency）的即時推論。此特性完美契合本專題將 AI 部署於機器狗本地端運算節點的需求。

本系統使用 MediaPipe 的兩個模組：
- **Gesture Recognizer**：辨識手勢（7 種內建手勢），CPU 即時推理
- **Pose Estimation**：推論人體 33 個關鍵點，用於姿勢分類與跌倒偵測

#### 手部追蹤與手勢辨識

本系統使用 MediaPipe Gesture Recognizer 進行手勢擷取與分類。系統透過攝影機擷取影像幀，模型能即時辨識手勢類別（如 thumbs_up、open_palm、pointing_up 等）。辨識結果透過手勢映射機制（Gesture Mapping），將靜態手勢轉譯為機器狗的控制指令（例如：張開手掌對應「停止」，比讚對應「開心動作」），實現直覺的人機互動。

#### 人臉偵測與辨識技術

本系統採用 OpenCV 內建的 YuNet 人臉偵測器與 SFace 人臉辨識器：
- **YuNet**（2023 年版）：輕量級人臉偵測模型，在 Jetson Orin Nano CPU 上可達即時推理速度，輸出人臉邊界框與信心度
<!-- source: docs/人臉辨識/README.md, benchmarks/results/ -->
- **SFace**（2021 年版）：人臉辨識模型，產出 128 維特徵向量，透過餘弦相似度比對已註冊的人臉資料庫

### (四) NVIDIA Jetson Orin Nano — 邊緣運算平台

NVIDIA Jetson Orin Nano 是一款專為 AI 邊緣推理設計的嵌入式運算模組。本專案使用 SUPER 版本（8GB 統一記憶體），搭配 JetPack 6 作業系統。Jetson 負責運行所有本地感知模組（人臉偵測、手勢辨識、姿勢辨識、物體偵測）、ASR/TTS 本地模型、ROS2 runtime，以及與 Go2 的 WebRTC 連線。

### (五) Intel RealSense D435 — RGB-D 深度攝影機

Intel RealSense D435 是一款具備 RGB 彩色攝影機與紅外線結構光深度感測器的攝影機模組。本系統利用其 RGB 影像進行人臉偵測、手勢辨識、姿勢辨識與物體偵測，同時利用深度資訊估算人臉與攝影機的距離。

### (六) 語音辨識技術（SenseVoice / Whisper）

本系統採用三級語音辨識（ASR）策略：

1. **SenseVoice 雲端**：阿里巴巴達摩院開發的中文語音辨識模型，部署於遠端 RTX 8000 伺服器，透過 FunASR 框架提供 HTTP API，延遲約 600ms，中文辨識準確率可達 92%
<!-- source: docs/語音功能/README.md, sprint-b-prime.md Day 2 -->
2. **SenseVoice 本地**：透過 sherpa-onnx 框架在 Jetson CPU 上運行的 INT8 量化版本，離線可用，準確率與雲端相當
3. **Whisper 本地**：OpenAI 開發的多語言語音辨識模型，透過 faster-whisper（CTranslate2 加速）在 Jetson CUDA 上運行，作為最終備援

### (七) 語音合成技術（edge-tts / Piper）

本系統採用雙重語音合成（TTS）策略：

1. **edge-tts**（雲端主線）：微軟 Edge 瀏覽器的 TTS 引擎，透過網路 API 合成自然語音，延遲低、音質佳
2. **Piper**（本地備援）：開源離線 TTS 引擎，使用 zh_CN-huayan-medium ONNX 模型，不需網路連線即可運行

### (八) 大型語言模型（Qwen2.5-7B-Instruct）

Qwen2.5 是阿里雲開發的開源大型語言模型系列。本系統採用 7B（70 億參數）Instruct 版本，透過 vLLM 推理框架部署於遠端 RTX 8000 伺服器。模型負責理解使用者的對話意圖，生成簡短合理的中文回覆（限制 25 字以內）。當雲端不可用時，系統自動降級為 RuleBrain 規則引擎（模板回覆）。
<!-- source: docs/語音功能/README.md -->

### (九) 即時物體辨識（YOLO）

YOLO（You Only Look Once）是一種即時物體偵測演算法。本系統採用 YOLO26n（Nano 版本，9.5MB），透過 ONNX Runtime 搭配 TensorRT FP16 加速在 Jetson 上運行，可偵測 COCO 資料集定義的 80 類常見物體。
<!-- source: docs/辨識物體/README.md -->

### (十) WebRTC DataChannel — Go2 通訊機制

WebRTC（Web Real-Time Communication）是一種支援瀏覽器與設備間即時通訊的開放標準。本系統透過 WebRTC DataChannel 與 Go2 Pro 進行雙向通訊，包含動作控制指令（站立、坐下、趴下等）與音訊播放（Megaphone API：ENTER → UPLOAD × N → EXIT）。

### (十一) PawAI Studio 技術棧（Next.js + FastAPI）

PawAI Studio 的前端採用 Next.js 16 + React 19 + TypeScript + Tailwind CSS 建構，後端 Gateway 採用 Python FastAPI + rclpy（ROS2 Python Client Library）。Gateway 透過 WebSocket 將 ROS2 感知事件即時轉發至瀏覽器前端，同時接收使用者的文字/語音輸入並轉發至 ROS2 語音管線。

## 五、系統限制（可行性分析）

本節描述使用者可能期望但未包含在系統範圍內的需求，說明其重要性以及未包含的原因。

### 1. 自主導航避障

**使用者期望**：機器狗能自動避開障礙物，安全地走到指定位置。

**對使用者的重要性**：高。在實際居家或展示場景中，機器狗需要在有家具、人員的環境中移動，自主避障是安全運行的基本需求。

**未包含原因**：本專案在開發過程中實際建構了完整的避障程式（D435 深度避障 + LiDAR 360° 安全防護），並通過了所有單元測試。然而，在真機驗證階段，由於 D435 攝影機安裝角度偏上方，低於鏡頭高度的障礙物（如椅腳、門檻）直到距離極近時才進入視野範圍；加上從偵測到機器狗完全停止的延遲鏈較長（包含軟體去抖動、WebRTC 通訊延遲、Go2 減速時間），導致三輪真機防撞測試均未通過。此外，Go2 Pro 的 LiDAR 感測器覆蓋率有限。基於安全考量，此功能在展示中暫時停用，程式碼保留供後續改進。
<!-- source: docs/mission/sprint-b-prime.md Day 7-8, docs/導航避障/README.md -->

### 2. 機器狗本體語音對話

**使用者期望**：直接對機器狗說話即可互動，不需要額外的裝置或網頁。

**對使用者的重要性**：高。這是自然互動的核心體驗，使用者期望像與寵物說話一樣直覺。

**未包含原因**：Go2 機器人內建散熱風扇持續運轉產生寬頻噪音，安裝在機身上的 USB 麥克風訊噪比（SNR）嚴重不足，實測語音辨識正確率顯著低於可用門檻。調高麥克風增益（mic_gain）並未改善，因為噪音也同步放大。此為物理硬體限制，無法透過軟體克服。目前改以 PawAI Studio 網頁端的按住說話（push-to-talk）功能作為語音互動入口。
<!-- source: docs/mission/sprint-b-prime.md Day 6 Gate C, Day 11 -->

### 3. 多人同時互動

**使用者期望**：多人同時與機器狗互動，每個人都能被辨識並得到個別回應。

**對使用者的重要性**：中。展示場景經常有多人圍觀，若機器狗只能與一人互動會降低展示效果。

**未包含原因**：人臉辨識模組支援最多 5 人同時追蹤，但手勢辨識與姿勢辨識模組（基於 MediaPipe）目前僅支援單人追蹤，且缺乏跨模組的使用者 ID 關聯機制。當多人同時出現時，系統無法判斷某個手勢屬於哪位使用者。完整的多人互動支援需重新設計追蹤管線架構，超出目前開發時程。

### 4. 多語言支援

**使用者期望**：除中文外，也能用英文、台語等語言與機器狗對話。

**對使用者的重要性**：中。國際展示或多元文化使用者場景下有此需求。

**未包含原因**：目前 ASR（SenseVoice / Whisper）、TTS（edge-tts / Piper）與 LLM（Qwen2.5）均針對中文進行調校與驗證。擴充其他語言需逐一驗證各環節的辨識率與合成品質，在有限的開發時程下優先確保中文互動的穩定性。

### 5. 自動充電與長時間自主運行

**使用者期望**：機器狗能自動回到充電座、在居家環境中長時間無人值守運行。

**對使用者的重要性**：中。居家陪伴場景需要機器狗能長時間自主運作。

**未包含原因**：Go2 Pro 硬體不具備自動對接充電座的功能。此外，Jetson 邊緣運算平台的供電依賴外部降壓模組，在 Go2 行走震動或高負載時偶發供電不穩的情形。目前展示環境需人工監控供電狀態。
<!-- source: docs/mission/sprint-b-prime.md Day 4, 9, 10, 11 -->

### 6. 物體互動指令

**使用者期望**：告訴機器狗「幫我拿水杯」，機器狗能找到並拿取物品。

**對使用者的重要性**：低（屬於進階功能期望）。

**未包含原因**：本系統的物體辨識模組（YOLO26n）能即時偵測畫面中的物體並觸發情境語音互動（例如看到杯子說「你要喝水嗎？」），但 Go2 Pro 機器狗不具備機械手臂，物理上無法抓取或搬運物品。物體辨識的定位是「看到什麼說什麼」的情境互動，而非實際操作。

# 第三章 軟體設計規格

## 一、資料庫設計

### 使用之資料庫說明

本系統**不使用傳統關聯式資料庫（RDBMS）**，因為系統的核心功能是即時感知與互動，資料以串流形式在 ROS2 節點之間傳遞，不需要持久化儲存。系統中的資料儲存方式如下：

| 資料類型 | 儲存方式 | 說明 |
|----------|----------|------|
| 人臉資料庫 | 檔案系統 + pickle | 已註冊使用者的人臉照片與特徵向量 |
| 感知事件 | ROS2 Topics（JSON） | 即時串流，不持久化 |
| 前端狀態 | Zustand in-memory | 瀏覽器端狀態管理，關閉即消失 |

### 人臉資料庫結構（NoSQL — 檔案系統）

**儲存位置**：`/home/jetson/face_db/`

**目錄結構**：
```
/home/jetson/face_db/
├── roy/
│   ├── image1.png
│   ├── image2.png
│   └── ...
├── grama/
│   ├── image1.png
│   └── ...
└── model_sface.pkl    ← 自動生成的特徵模型
```

**model_sface.pkl 資料結構（JSON 範例說明）**：
```json
{
  "embeddings": {
    "roy": [[0.12, -0.34, ...], [0.15, -0.31, ...]],
    "grama": [[0.08, -0.42, ...]]
  },
  "centroids": {
    "roy": [0.135, -0.325, ...],
    "grama": [0.08, -0.42, ...]
  },
  "counts": {
    "roy": 3,
    "grama": 2
  }
}
```

- `embeddings`：每位使用者的所有人臉特徵向量（128 維 float32 陣列）
- `centroids`：每位使用者特徵向量的平均值，用於快速比對
- `counts`：每位使用者的註冊照片數，用於偵測資料庫變更並自動重新訓練

**註冊流程**：將使用者的人臉 PNG 照片放入以其名字命名的資料夾，系統啟動時自動偵測、提取特徵並生成 pickle 模型。若照片數量有變動，系統自動重新訓練。

### ROS2 感知事件資料結構（JSON over Topics）

**PawAI Event 通用信封格式**（Studio Gateway 發送至前端）：
```json
{
  "id": "evt_1712345678_face",
  "timestamp": "2026-04-07T10:30:00Z",
  "source": "face",
  "event_type": "identity_stable",
  "data": {
    "track_id": 1,
    "stable_name": "roy",
    "sim": 0.42,
    "distance_m": 1.8
  }
}
```

# 第四章 系統專題實作檢討

## 一、發展中遭遇到問題、困難與解決方法

### 1. WebRTC 連線相容性問題

**問題**：aiortc 套件自動升級至 1.14.0 後，SCTP 握手超時（30 秒以上），DataChannel 無法開啟，機器狗完全失聯。
**解決方法**：降級 aiortc 至 1.9.0 並移除 STUN 配置，SCTP 握手時間降至 0.5 秒。後續將版本鎖定寫入 requirements.txt。
**教訓**：依賴套件版本必須鎖定，自動升級在嵌入式環境極易造成不可預期的相容性問題。

### 2. Go2 Megaphone 音訊播放 API 格式陷阱

**問題**：團隊花費約兩週判定 Megaphone API「已失效」，無法從機器狗喇叭播放語音。根因為 payload 格式錯誤 — chunk_size 必須為 4096 base64 字元、msg type 必須為 "req" 而非 "msg"，官方文件未記載。
**解決方法**：透過逆向分析 Go2 韌體通訊協議找出正確格式，修正後連續 20/20 測試通過。
**教訓**：硬體 API 文件不完整時需逆向驗證，優先懷疑格式而非硬體。

### 3. Go2 Pro 有線模式架構限制

**問題**：嘗試建立 Ethernet 控制通道以降低延遲，連線成功後卻完全沒有 ROS2 topic 資料流。Go2 Pro 內部 Wi-Fi 與有線網段隔離，有線只能到達 MCU。
**解決方法**：放棄有線模式，維持 Wi-Fi WebRTC。
**教訓**：硬體架構設計決定軟體的可能性邊界，應在投入開發前先確認硬體能力。

### 4. SLAM 建圖與 Nav2 導航的多重挫折

**問題**：經歷 Nav2 參數不生效（未 colcon build 同步到 install/）、/scan 全為 inf（intensity 過濾過嚴）、Jetson 點雲頻率掉至 2 Hz（多實例 driver + 低效序列化，經七輪優化恢復至 5.1 Hz）、D435 鏡頭角度導致低障礙物看不到（三輪真機防撞測試全部撞上）等問題。
**解決方法**：導航避障功能完全停用。程式碼保留（含 20 個 unit tests），Demo 改為人工監控 + 手勢 stop。
**教訓**：導航涉及硬體（鏡頭角度、LiDAR 覆蓋率）、網路（WebRTC 延遲）、演算法（SLAM 頻率）三重限制，任一環節不達標整個功能即不可用。

### 5. 多重網路跳轉導致控制延遲

**問題**：開發初期透過 Go2 (Wi-Fi) → Mac VM (橋接) → Windows (有線) 多層跳轉進行 Nav2 控制，延遲 20-50ms 接近控制週期（50ms），指令積壓導致機器狗原地旋轉。
**解決方法**：改為 Windows 本地安裝 ROS2 + RViz2（零延遲控制），Foxglove 僅用於監控。
**教訓**：即時控制應減少網路跳轉層數，選對工具比調參數更重要。

### 6. 模型選型三次推翻

**問題**：手勢/姿勢辨識模型選型經歷三次推翻：RTMPose wholebody（GPU 滿載）→ DWPose（TensorRT 精度暴跌 + MMPose JetPack 6 零支援）→ MediaPipe（原判定不可行，實測推翻，CPU 即時推理）。
**解決方法**：最終採用 MediaPipe CPU-only 方案，手勢可達即時推理速度，GPU 佔用 0%。
<!-- source: docs/手勢辨識/research/選型過程.md, docs/姿勢辨識/README.md -->
**教訓**：文獻調查不能替代實測，Jetson 生態碎片化嚴重。

### 7. ASR 三級 Fallback 的逐步演進

**問題**：Whisper Small 中文短句+噪音辨識率已達瓶頸，無法滿足 Demo 需求。
**解決方法**：引入 SenseVoice 雲端（辨識率大幅提升）+ SenseVoice 本地（離線備援）形成三級降級。另實作 10+ 項 workaround（幻覺過濾 22 pattern、短文字過濾、Echo Gate 等）。
<!-- source: docs/mission/sprint-b-prime.md Day 2 -->
**教訓**：語音辨識在真實噪音環境下表現遠低於實驗室數據，需多層防禦。

### 8. 跌倒偵測正面站姿誤判

**問題**：使用者正面站在鏡頭前時被錯誤判定為「跌倒」（肩膀展開導致 bbox 寬高比觸發條件）。
**解決方法**：新增 vertical_ratio 防護條件（肩膀到髖部垂直距離 / 軀幹長度），相對尺度不受距離影響。
<!-- source: docs/姿勢辨識/README.md, sprint-b-prime.md Day 3-6 -->
**教訓**：基於 bbox 的簡單規則在邊界情況易失效，需多維度防護。

### 9. Jetson 環境污染事件

**問題**：在 Jetson 上執行 `pip install ultralytics`，自動拉入不相容的 PyTorch + NumPy 版本，整個 Python 環境崩潰。
**解決方法**：環境修復耗時半天。建立規範：Jetson 嚴禁直接 pip install 大型框架，正確做法是開發機匯出 ONNX 再 scp 到 Jetson。
<!-- source: docs/mission/sprint-b-prime.md Day 9 -->
**教訓**：邊緣裝置 Python 環境極脆弱，任何可能覆蓋系統 wheel 的安裝都是高風險。

### 10. LLM 視覺避障的根本局限

**問題**：早期嘗試透過 LLM 分析攝影機影像進行避障。LLM 能識別障礙物且建議轉向正確，但無法判斷「機器狗自身大小，能否通過」。
**解決方法**：評估多個 LLM 模型後認定純視覺方案只適合粗略避障，精確導航仍需 SLAM + 深度感測。
<!-- source: docs/archive/logs/ -->
**教訓**：LLM 視覺理解有根本局限，不能完全取代傳統感測器。

### 11. ROS2 QoS 策略不匹配

**問題**：多個模組間因 QoS 策略不一致導致訊息無法接收（RELIABLE subscriber 無法接收 BEST_EFFORT publisher）。
**解決方法**：統一感測器類 topic 使用 BEST_EFFORT QoS。
**教訓**：多模組開發時 QoS 策略需在介面契約中明確規範。

### 12. Jetson 供電危機

**問題**：Sprint 11 天期間 Jetson 反覆被強制關機（Go2 電池 → XL4015 降壓 → Jetson，高負載 spike 超出降壓模組能力）。
**解決方法**：調整電壓 18.8V → 19.2V 稍有改善但未根治。至 Sprint 結束時仍為最大硬體風險。
<!-- source: docs/mission/sprint-b-prime.md Day 4, 9, 10, 11 -->
**教訓**：嵌入式系統供電設計需考慮負載峰值而非僅平均功耗。

### 13. Go2 風扇噪音導致機身 ASR 不可用

**問題**：在 Go2 真機上 ASR 正確率因散熱風扇寬頻噪音驟降至不可用水準。調高 mic_gain 無效（噪音同步放大）。
**解決方法**：策略轉向 — Demo 改為「視覺互動為主 + 網頁語音輔助」，語音入口從機身移到 PawAI Studio 瀏覽器端。
<!-- source: docs/mission/sprint-b-prime.md Day 6, Day 11 -->
**教訓**：物理環境限制無法透過軟體增益解決，需從系統架構層面尋找替代方案。

### 14. SSH Tunnel 與遠端 GPU Server 連線

**問題**：雲端 ASR/LLM 依賴 SSH tunnel 連接 RTX 8000，tunnel 不穩定導致全 timeout。多個子問題疊加（Port 不匹配、Server 未啟動、缺少依賴、async blocking）。
**解決方法**：推 async fix 到 Server 並重啟；Jetson 建立 systemd user service 永久化 SSH tunnel（開機自動起）。
**教訓**：遠端連線涉及多層級，需逐層驗證每個環節。

## 二、系統優缺點（SWOT）評估

| | 正面 | 負面 |
|---|------|------|
| **內部因素** | **優勢（Strengths）**：多模態感知管線完成度高（六大模組全部開發完成）；三級降級策略確保無網路時仍可基本互動（ASR/LLM/TTS 全有 fallback）；模組化 ROS2 架構使每個感知模組可獨立替換或升級；PawAI Studio 提供遠端觀測與操控能力，提升展示與開發效率。 | **劣勢（Weaknesses）**：Jetson 供電不穩為最大硬體風險；Go2 風扇噪音導致機身 ASR 不可用，需改由網頁端提供語音入口；人臉追蹤存在抖動問題尚未根治；手勢/姿勢辨識僅支援單人追蹤；OpenCV 版本鎖定無法享受新版本改進。 |
| **外部因素** | **機會（Opportunities）**：長者陪伴與智慧居家市場需求持續增長；邊緣 AI 硬體（如 NVIDIA Jetson 系列）持續降價，降低部署門檻；ROS2 生態日益成熟，可復用大量社群套件加速開發。 | **威脅（Threats）**：Jetson 生態碎片化嚴重（不同 JetPack 版本間套件不相容）；Go2 Pro 韌體開放程度有限（無 CycloneDDS 直連），限制了即時控制的可能性；競品如小米 CyberDog 2、Boston Dynamics Spot 擁有更多資源與更完整的生態系。 |

## 三、未來展望

基於系統限制章節的分析以及開發過程中的經驗，以下為後續可改進的方向：

1. **導航避障功能**：重新設計 D435 攝影機的安裝角度（建議俯視約 -45°），使低障礙物能在更遠距離被偵測到；或考慮升級至 Go2 EDU 版本以獲得 CycloneDDS 直連能力，降低控制延遲。

2. **機身語音互動**：採用指向性麥克風取代全向性 USB 麥克風，搭配軟體降噪演算法（如 RNNoise），改善機器狗本體的語音辨識環境。

3. **多人互動支援**：建立跨模組的使用者 ID 關聯機制，將人臉辨識的 track_id 與手勢/姿勢偵測結果綁定，實現多人場景下的個別互動。

4. **物體互動擴充**：若未來搭配機械手臂（如 WidowX），可實現「看到→理解→操作」的完整物體互動鏈路。在此之前，可先擴充語音引導互動（如「杯子在你的左前方」）。

5. **供電穩定化**：改用獨立的 18650 鋰電池組供電，或升級為更高品質的 DC-DC 降壓模組，確保 Jetson 在 Go2 行走時的供電穩定性。

6. **AI 大腦進化**：將目前的 rule-based Interaction Executive 升級為 LLM-driven 決策中樞，讓機器狗能基於多模態上下文進行更複雜的情境推理與行為規劃。

## 1. ❌ 禁用詞 / 禁用句式 checklist

### 1.1 「做不到」類（呂奇傑 5/22 + 雅文 5/20 共同訊號）

| 禁用 | 為什麼禁 | 替代 |
|---|---|---|
| 「**本來想做** X」 | 暴露 scope 縮減過程，讓觀眾覺得專題在縮水 | 講「我們做到了 Y」，X 留在 future work 段 |
| 「**現在還沒做到**」 | 主動把焦點導向缺口 | 講完成的部分，後面被問再說 |
| 「**比較差 / 不完整 / 不穩定**」 | 自己降低觀眾預期，反而讓人覺得不可信 | 「我們選擇 X 而不是 Y，因為 [理由]」 |
| 「**期待不要太高**」 | 呂奇傑明確說「這是報告技巧問題不是要降低期待」 | 直接展示，讓功能自己說話 |
| 「**現在比較陽春**」 | 同上 | 同上 |

### 1.2 「定位錯」類（雅文 5/20 + 葉承達 5/18 共同訊號）

| 禁用 | 為什麼禁 | 替代 |
|---|---|---|
| 「**玩具**」 | 雅文已用「玩具」貶過 PAI，自稱玩具等同認輸 | 「機構巡檢助理 / Physical AI POC」 |
| 「**單純居家陪伴**」 | 葉/雅文都質疑（安全 + 競品） | 「居家 / 機構四足互動機器人平台」（ADR-0002 雙層敘事）|
| 「**寵物狗**」 | 雅文質疑「20kg 機體當寵物有安全疑慮」 | 「機器狗形態的機構巡檢助理」 |
| 「**像 Siri / Alexa / 小米的語音助理**」 | 雅文明確說「差異化不足」 | 「能移動、有實體、多模態的具身 AI」（三鐵律）|

### 1.3 「跟 GPU 比」類（雅文 5/20 訊號）

| 禁用 | 為什麼禁 | 替代 |
|---|---|---|
| 「**比 GPU 慢**」 | 雅文：「不能拿 GPU 標準批評自己，Edge AI 是不同世界」 | 「Edge AI 8GB 共享記憶體跑五個感知模型的優化」 |
| 「**跑得不如人家好**」 | 同上 | 「在 Jetson Orin Nano 8GB 上同時跑 X / Y / Z 是工程挑戰」 |
| 「**運算力有限**」 | 暴露限制反而失去敘事掌控 | 「為了 edge deployment 我們做了 ONNX TRT INT8 量化」 |

### 1.4 「物理接觸」類（葉承達 5/18 + ADR-0001）

| 禁用 | 為什麼禁 | 替代 |
|---|---|---|
| 「**PawAI 會牽長者的手**」 | ADR-0001 明確紅線 | 「PawAI 在 1m 距離外做感知與回報」 |
| 「**PawAI 會攙扶**」 | 同上 | 「PawAI 偵測到跌倒會走近觀察並回報照護人員」 |
| 「**PawAI 摸頭安撫**」 | 同上 | 「PawAI 用語音陪伴」 |
| 「**PawAI 跟阿公阿嬤玩**」 | 同上 + 葉警示「拉手摔倒」 | 「PawAI 主動詢問狀況」 |

## 2. ✅ 必說句 checklist

### 2.1 開場（每場 demo / 簡報必出現）

> 「**沒有 PawAI 之前，Go2 只能用遙控器；有了 PawAI 之後，照護人員可以用自然語言派它巡檢、它能認得人、聽得懂指令、知道哪些動作不能做。**」

呂奇傑 5/22 親自示範句式：「**沒有我們之前 X 只能 Y，有了我們之後可以 Z**」。

### 2.2 為什麼是狗（每個鏡頭口頭強調）

> 「**這隻狗的不可替代性是：能移動、有實體、多模態感知三件事同時做到。手機做不到、固定攝影機做不到、雲端 AI 也做不到。**」

三鐵律（v2 spec §1）：移動 ✅ + 實體 ✅ + 多模態 ✅。**每個 demo 鏡頭必須口頭點出至少 2 個 ✅**。

### 2.3 整合挑戰（段 0 硬體 Before/After 必出現）

> 「**為了讓家用版 Go2 變成可程式控制的具身平台，我們做了 3D 列印背包、降壓板防燒 Jetson、雙系統有線整合、第三方 SDK 破解、NumPy 版本治理。**」

呂奇傑 5/22 最強訊號「**真正困難的硬體過程完全沒提**」。具體 5 件硬體挑戰 + 1 件軟體挑戰可任選。

### 2.4 POC framing（任何被問「為什麼不做 X」時用）

> 「**這是 Physical AI 在機構照護場域的原型驗證（POC），不取代照護人員，也不宣稱商用部署。**」

雅文 5/20「POC 階段不需做到完美」+ 葉承達 5/18 「v1.0 → v2.0」框架。

### 2.5 安全機制（段 5 安全拒絕後必出現）

> 「**我們設計了讓 LLM 不能直接控制機器人的安全層。這個動作 Safety Layer 攔截了。即使語意理解能產生『翻跟斗』這個意圖，執行層仍會阻擋。**」

呂奇傑 5/22 指定鏡頭。安全機制是 PawAI Brain 三層架構唯一可視化證據。

### 2.6 結尾（影片末 / 簡報結束必出現）

> 「**機構巡檢只是 6/18 POC 場景；PawAI 的平台能力可延伸到校園、導盲、救災、居家延伸。**」

ADR-0002 雙層敘事視覺化。**未來情境順序必為「校園 → 導盲 → 救災」**（grill Q3 鎖定）。

## 3. 安全聲明 checklist（責任邊界）

每份對外材料（文件 / 影片 / 簡報）**必須**至少出現一次：

> 「**PawAI 設計為非接觸式機構巡檢助理。所有物理協助仍由照護人員執行，PawAI 只負責感知、提醒、回報。**」

詳見 ADR-0001。被問到「為什麼 PawAI 不主動扶長者起來」時用：

> 「我們明確把 PawAI 定位為**非接觸式**。20 公斤機體 + 走路晃動 + sport mode 速度門檻，任何接觸式互動都是安全紅線。未來如要重啟物理接觸功能，必須另開安全設計與驗證。」

## 4. Before/After 句式範例

呂奇傑 5/22 最強訊號：**「所有東西有沒有用都來自於 Before / After」**。所有對外材料的功能介紹都用此句式。

### 4.1 段 0 硬體 Before/After 範例

| Before | After |
|---|---|
| Go2 Pro 是家用版，原廠只給遙控器 | PawAI 讓 Go2 具備視覺、語音、身份判斷、安全策略 |
| Go2 12V 電壓會燒 Jetson | 我們設計三層降壓板 + 散熱外殼 |
| Go2 SDK 是家用版，動作 API 受限 | 我們用第三方 SDK 破解出 30+ 動作可程式控制 |
| Jetson + Go2 是兩個獨立系統 | 我們做了有線整合 + 雙系統時鐘對齊 |
| ROS2 模組 NumPy 版本互打 | 我們做了 pip 版本治理 + Jetson AI Lab wheel 整合 |

### 4.2 段 2-5 功能 Before/After 範例

| Before | After |
|---|---|
| Go2 看不到誰在門口 | PawAI 認得 Roy，主動打招呼 |
| Go2 不會分辨陌生人 | PawAI 偵測到陌生人會主動追問身份 |
| Go2 動作要靠遙控器按鍵 | PawAI 聽「坐下 / 舉手」就執行 |
| Go2 SDK 開放所有動作就會出事 | PawAI 有安全層，「翻跟斗」會被拒絕 |

### 4.3 6/18 live demo 場景劇範例

| Before | After |
|---|---|
| 固定攝影機只能看 | PawAI 可以走到現場 |
| 照護人員不可能每秒巡視 | PawAI 持續巡檢 + 異常事件主動回報 |
| App 提醒長者沒人在身邊 | PawAI 走過去說話陪伴（非物理接觸） |
| 跌倒偵測攝影機沒人立刻處理 | PawAI 走到旁邊觀察 + 立刻語音回報「請工作人員立刻確認」 |

## 5. 整合挑戰寫法（葉承達 5/18 訊號）

葉明確說：「**整合派該講選型、整合、踩坑，不該講準確率**」（你們用現成模型，不是自己訓練）。

### 5.1 ❌ 不該寫

- 「我們的 face_identity 準確率 95%」（你們沒訓練 YuNet / SFace，不該宣稱準確率）
- 「我們的物件偵測 mAP 50」（你們用 YOLO26n，不該宣稱 mAP）
- 一大堆模型架構圖 + 演算法說明（葉：「**5 秒就講完了**」）

### 5.2 ✅ 該寫

| 領域 | 該講的踩坑 |
|---|---|
| 模型選型 | YOLO26n / YuNet / SFace / MediaPipe / Whisper / Piper / Gemini 為何這樣搭配，每個權衡點 |
| 整合 | NumPy / setuptools / onnxruntime-gpu 版本衝突解法（Jetson AI Lab wheel） |
| ROS2 接線 | event vs state topic 設計、QoS 選擇、跨 node 異步處理 |
| Brain 三層架構 | Safety / Policy / Expression 分離，LLM 不直接控機器人 |
| Studio 串接 | WebSocket → Brain → Skill → Go2 端到端延遲優化 |
| 硬體整合 | 3D 列印 + 降壓板 + USB bandwidth + 散熱 |

格式建議：「**我們用了 X，做 Y 用，踩了 Z 坑，靠 W 解**」三段式。

## 6. 文件結構（葉承達 5/18 訊號）

### 6.1 6/13 文件繳交格式硬規則

葉明確說：「**文件 40 頁太多，刪規格表、刪準確率章節、架構圖優先**」。

| 規則 | 落地 |
|---|---|
| **架構圖優先** | 每個模組首段放 1 張架構圖（v2 spec §3 任務閉環 / §4 三層架構等可直接用）|
| **三段式：用了什麼 + 做什麼用 + 踩了什麼坑** | 每個模組（face / gesture / pose / object / speech / brain / studio / nav）按此格式寫 |
| **規格表壓附錄** | 模型 input/output size / FPS / 模型大小等放附錄，正文不放 |
| **刪準確率章節** | 整合派不該宣稱準確率，全刪 |
| **整合挑戰獨立章節** | 寫硬體 + 軟體 5-6 大類踩坑（§ 5.2 該寫的清單）|
| **User Story 別硬套** | 葉明說「對 PAI 不適合」，改用「情境 → function → Before/After」結構 |

### 6.2 段落長度建議

- 每段 ≤ 5 行（葉「看的蠻吃力的」反饋）
- 每個 bullet ≤ 2 行
- 每個 table 列 ≤ 8 條
- 程式碼 snippet 只放精要 5-10 行，完整在附錄

## 7. 報告 / 影片 / 簡報順序（呂奇傑 5/22 訊號）

呂奇傑明確指出正確 demo 流程：

1. **先講你做什麼**（讓老師 / 觀眾知道要看什麼）
2. **現場展示**（看到他了）
3. **講困難點**（這不簡單）
4. **講技術實作**（放最後）

❌ 不要做：
- 一上來就講限制與失敗
- 把所有功能 demo 完才講情境（應該情境包裝著 demo）
- 技術細節（CNN 架構 / 模型選型 / API spec）放最前面

✅ 5/27 影片 / 6/18 live demo 順序自我檢核：
- [ ] 開場 3 句內出現「沒有 PawAI 之前 → 有了之後」（§ 2.1）
- [ ] 第 1 個展示鏡頭出現後才講該功能的技術細節
- [ ] 困難點（硬體 / 整合）放在 demo 之後當 highlight，不放開頭
- [ ] 不出現任何「現在還沒做到」「比較差」字樣

## 8. 自我 audit checklist（寫完任何對外材料對著跑）

寫完 5/27 影片旁白 / 6/13 文件 / 6/18 live 現場話術 / 簡報後，過下列 checklist：

```
[ ] § 1.1 「做不到」類禁用詞全刪
[ ] § 1.2 「定位錯」類禁用詞全刪（玩具 / 寵物 / 純語音助理）
[ ] § 1.3 「跟 GPU 比」類禁用詞全刪
[ ] § 1.4 「物理接觸」類禁用詞全刪
[ ] § 2.1 開場 Before/After 句式存在
[ ] § 2.2 為什麼是狗 三鐵律至少口頭點出 2 個
[ ] § 2.3 整合挑戰 5 件硬體挑戰至少出現 3 件（適用段 0 硬體）
[ ] § 2.5 安全機制聲明存在（適用段 5 / 任何 demo）
[ ] § 2.6 結尾雙層敘事 + 未來情境順序對
[ ] § 3 安全聲明至少出現一次
[ ] § 4 Before/After 句式覆蓋 ≥ 3 個功能
[ ] § 5 整合挑戰寫法（不寫準確率）
[ ] § 6 文件結構規則（適用 6/13 文件）
[ ] § 7 順序對（先做什麼 → 展示 → 難點 → 技術）
```

過 13 條 → 可對外。

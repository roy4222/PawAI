# Go2 導航避障落地計畫 v2

更新日期：2026-03-02

## 1. 目標與原則

本計畫用於把「深度研究報告」轉成目前專案可直接執行的路線。

優先順序固定如下：

1. 安全（先零碰撞）
2. 穩定（再降 ABORT / 卡住）
3. 體感流暢（最後追求邊牧級）

本文件採納原則：

- 採納可在現有 ROS2 Humble + Nav2 主線上直接落地者
- 延後高複雜度或生態不成熟路線
- 所有升級都必須通過量化 gate 後才可進下一階段

## 2. 現況基線（依 repo 事實）

目前主線是 Nav2 + AMCL/localization + DWB，而非 Cartographer/TEB/MPPI 主線。

關鍵證據：

- `go2_robot_sdk/config/nav2_params.yaml`
  - `FollowPath.plugin: "dwb_core::DWBLocalPlanner"`
- `go2_robot_sdk/launch/robot.launch.py`
  - `localization_launch.py`（nav2=true 且 slam=false 時啟用）
  - `online_async_launch.py`（slam_toolbox）
- `scripts/start_nav2_localization.sh`
  - 預設 `LIDAR_POINT_STRIDE=8`
- `go2_robot_sdk/launch/robot.launch.py`
  - `pointcloud_to_laserscan` 已使用 `min_height=-1.0`

## 3. 可採納 / 條件採納 / 暫緩

### 3.1 立即採納（Now）

1. 保持現有主線：Nav2 + AMCL + DWB
2. 先做安全 gate：零碰撞優先於流暢
3. 先修感測品質：優先處理 scan 稀疏與更新穩定性
4. 建立固定可觀測流程：`scan/costmap/cmd_vel/odom/action status`
5. 以小步 A/B 調參，不同變數分離測試
6. 以 D435 補強近場偵測，作為 local costmap 第二觀測源

### 3.2 條件採納（Later, after gates）

1. MPPI（僅在 DWB 穩定後）
2. Cartographer 3D（僅在證明定位是主瓶頸後）
3. D435 近場補強（在 LiDAR stride 修復後，作為 Phase A4 加入）

### 3.3 暫緩（Hold）

1. TEB 作為 ROS2 Humble 主線控制器
2. NoMaD/ViNT/DRL 直接上實機主線

## 4. 三階段落地路線

## Phase A：安全與穩定止血（1-2 週）

### A1. 感測密度與近場可見度

目標：避免「看不到小障礙、看太晚」。

執行：

- 針對 `LIDAR_POINT_STRIDE` 做 A/B 測試（8 -> 2 -> 1）
- 每次固定量測：`/scan` 實際有效回波比例、更新頻率、CPU
- 保留 `min_height=-1.0` 設定

驗收：

- `/scan` 穩定 >= 5 Hz（建議 >= 8 Hz 更佳）
- 不再長時間大量全 inf 區段

### A2. 安全包絡與恢復策略

目標：即使感測偶發掉點，也要先不撞。

執行：

- 強化 local costmap 對近場障礙物敏感度
- 調整 inflation/footprint padding 使避障提早發生
- 縮短「卡住判定時間」以加快 recovery（避免 20 秒僵持）

驗收：

- 連續 50 次短距導航（0.5m/0.8m）0 碰撞
- ABORT 率明顯下降（目標 <= 10%）

### A3. 可觀測性與復盤

目標：每次異常都可追溯根因。

執行：

- 固定 tmux pane：`cmd_vel`、`odom`、`action status`
- 固定視覺化：local/global costmap + scan
- 每輪測試保留一致紀錄欄位（成功/失敗、原因分類、參數版本）

驗收：

- 每次失敗可在 10 分鐘內歸類到以下之一：
  - 感測稀疏/延遲
  - costmap 幾何/膨脹
  - controller 參數
  - TF/時間戳

### A4. D435 近場補強（Stride 修復後）

目標：利用 Intel RealSense D435 填補 LiDAR 近場盲區，提升低矮障礙物偵測能力。

前置條件：A1 stride 已確定，`/scan` >= 8Hz 穩定。

執行：

1. 安裝 D435 驅動：`sudo apt install ros-humble-realsense2-camera`
2. 切換 URDF 至 `go2_with_realsense.urdf`
3. 在 `robot.launch.py` 或獨立 launch 檔啟動 realsense2_camera：
   - 解析度：424×240，幀率：15fps，停用 RGB，啟用 decimation+temporal+spatial 濾波
4. 在 `nav2_params.yaml` local costmap 新增 `d435` 觀測源（PointCloud2，voxel_layer）
5. 驗證 TF：`camera_link → base_link` 在 RViz 中正確
6. 配置 `nav2_collision_monitor` 以 D435 作為短程急停源（0.2-1.5m）

驗收：

- D435 PointCloud2 穩定 >= 10Hz
- local costmap 可見 D435 偵測的障礙物（RViz 確認）
- D435 加入後 CPU 增量 <= 15%
- D435 離線時自動回退 LiDAR-only，導航不中斷
- 連續 30 次短距導航 0 碰撞（含低矮障礙物場景）
## Phase B：控制體感優化（1-2 週）

前置條件：Phase A gate 全過。

### B1. DWB 進階優化

目標：減少「怕怕的狗」體感（抖動、猶豫、急停）。

執行：

- 在安全前提下微調 critics 比例與速度平滑
- 逐步降低不必要停滯，提升轉向連續性
- 保持保守上限速度，不做激進提速

驗收：

- 連續 100 次短距目標：成功率 >= 95%，0 碰撞
- 主觀體感：轉向更連續，急停頻率下降

### B2. MPPI 小規模評估（可選）

前置條件：DWB 已達到上面指標。

執行：

- 小樣本切換 MPPI（低 batch、低算力負擔配置）
- 比較同一路線的成功率、抖動、CPU、延遲

驗收：

- 若 MPPI 在 Jetson 上穩定且優於 DWB 才考慮升級
- 若出現不穩或算力壓力，回退 DWB，不硬推

## Phase C：架構升級評估（2-4 週）

前置條件：Phase A/B 已穩定量產。

### C1. 是否導入 Cartographer

僅在以下情況成立才進入：

- 已證明主要問題是定位/地圖一致性，而非感測稀疏與 controller 調參
- Jetson 算力與記憶體有明確 headroom

### C2. 多感測進階應用與 AI 模型

順序：

1. D435 進階應用：depth-to-costmap 精細化、D435 輔助視覺子目標建議
2. 再評估 NoMaD/ViNT/DRL（研究化）

原則：

- 研究模型不直接替代現有安全主線
- 必須先通過沙盒測試與明確降級策略

> **注意**：Isaac ROS Visual SLAM 對 D435（無 IMU）支援不足，暫不採用。
## 5. Gate 定義（進階前硬條件）

### Gate G1：安全

- 0 碰撞 / 0 實體接觸（至少 200 次短距任務或 10 小時監看測試）

### Gate G2：可靠

- 短距目標成功率 >= 95%
- ABORT 率 <= 5%

### Gate G3：感測與時序

- `/scan` 更新穩定，無明顯掉速波動
- 無持續 TF extrapolation/time-jump 問題

### Gate G4：資源

- 測試期間無長時間 CPU 打滿導致控制頻率崩潰
- 記憶體使用穩定，無連續上升失控

## 6. 每日執行節奏（建議）

1. 固定 preflight 清場
2. 跑 1 個變數 A/B（只動一個參數）
3. 執行短距任務集（0.5m/0.8m + 單障礙）
4. 記錄結果與失敗分類
5. 當日只接受可重現改善，不追求一次到位

## 7. 不做事項（避免誤入）

1. 不在同一輪同時改多組關鍵參數
2. 不在安全 gate 未過前追求高速/激進流暢
3. 不把研究型模型直接接管實機主控制

## 8. 結論

這份 v2 計畫把深度報告轉成「先活下來、再跑漂亮」的工程路線：

- 報告的長期方向可保留
- 但執行順序必須改為：
  - 先安全穩定（Phase A）
  - 再體感優化（Phase B）
  - 最後才做架構升級（Phase C）

只要按 gate 推進，專案就能在降低碰撞風險的同時，穩定往「流暢導航避障」前進。

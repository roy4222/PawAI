# 2026-01-13 開發日記

**日期：** 2026/1/13 (週二)  
**階段：** W12 - 有線 WebRTC 影像 + LiDAR 低頻排查  
**撰寫時間：** 10:30

---

## 📌 今日問題

### 影像頻率極低
- `/camera/image_raw` 僅 **~2 Hz**，即便有線 WebRTC 連線成功。
- 帶寬壓縮（image_transport）後，頻率仍無改善。

### LiDAR 頻率極低（WiFi / 有線皆然）
- `/point_cloud2` 約 **0.03–0.17 Hz**
- `/scan` 約 **0.07–0.14 Hz**
- `/map` 約 **0.2 Hz**（SLAM 更新非常慢）

---

## 🔍 根因判斷

### 影像
- **瓶頸不在網路頻寬**，而是 **VM 內部 Raw 圖搬運與 CPU 解碼負載**。
- Raw 影像單張約 **2.76 MB**，在 ROS2 內部傳遞成本高。

### LiDAR
- **WebRTC 本身低頻限制**，非有線特例。
- WiFi 與有線都低頻，代表是 WebRTC 模式的通道瓶頸。

---

## ✅ 解法與處理

### A) 影像：驅動層直接發布 CompressedImage
- **關閉 Raw Image 發布（預設）**
- **驅動層直接輸出 JPEG** (`/camera/image_raw/compressed`)
- **避免 image_transport 轉發**

**新增參數（go2_driver_node）：**
- `publish_raw_image`：預設 `False`
- `publish_compressed_image`：預設 `True`
- `jpeg_quality`：預設 `80`

**成果：**
- `/camera/image_raw/compressed` 約 **13–15 Hz**
- 單張約 **0.12 MB**
- 帶寬約 **1.6–1.9 MB/s**

### B) LiDAR：自動啟用開關
- 發現 WebRTC 下必須先送 `rt/utlidar/switch=on` 才有 LiDAR 資料。
- 已加入 `enable_lidar`（預設 `True`），在 robot validated 後自動送。

---

## 🧪 驗證流程

### 影像
```bash
ros2 topic list | grep -E "camera|compressed"
ros2 topic hz /camera/image_raw/compressed
ros2 topic bw /camera/image_raw/compressed
```

### LiDAR
```bash
ros2 topic hz /point_cloud2
ros2 topic hz /scan
ros2 topic hz /map
```

> 注意：若 `/camera/image_raw/compressed` 出現 2 個 publisher，代表殘留壓縮節點。

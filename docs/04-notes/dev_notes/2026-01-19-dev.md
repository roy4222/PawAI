# 4. /scan 資料有效性

# 5. /odom 漂移測試
# 機器人原地不動 30 秒，記錄起始和結束 pose

## 今日實測結果：LiDAR 低頻問題診斷

**測試時間：** 2026-01-19  
**測試環境：** WebRTC 連線，Go2 Pro L1 LiDAR，Ubuntu 22.04 + ROS2 Humble

### 觀察到的問題現象

| Topic | 實測頻率 | 預期頻率 | 狀態 |
|-------|---------|---------|------|
| `/point_cloud2` | ~0.3-0.8 Hz | ≥10 Hz | ❌ 嚴重偏低 |
| `/scan` | ~0.1-2 Hz | ≥5 Hz | ❌ 不穩定 |
| `/camera/image_raw/compressed` | ~15 Hz | - | ✅ 正常 |
| `/odom` | ~18-19 Hz | ≥20 Hz | ⚠️ 略低 |
| `/imu` | ~19 Hz | ≥50 Hz | ⚠️ 略低 |

### 診斷指令與結果

```bash
# 1. 檢查感測器頻率
ros2 topic hz /point_cloud2
# 實測：0.3-0.8 Hz（遠低於預期 10 Hz）

ros2 topic hz /scan
# 實測：0.1-2 Hz（波動大，目標 ≥5 Hz）

ros2 topic hz /camera/image_raw/compressed
# 實測：15 Hz（正常）

ros2 topic hz /odom
# 實測：18-19 Hz（略低於目標 20 Hz）

ros2 topic hz /imu
# 實測：19 Hz（遠低於目標 50 Hz）

# 2. 檢查 /map 狀態
ros2 topic echo /map --field data --once | head -50
# 結果：幾乎全為 -1（未知區域），無有效建圖

# 3. 檢查 SLAM 節點
ros2 topic list | grep slam
# 輸出：無 /slam_toolbox/feedback publisher

# 4. 檢查 /point_cloud2 QoS 設定
ros2 topic info /point_cloud2
# 結果：Reliability: BEST_EFFORT

# 5. 檢查 LZ4 解碼器
# 日誌確認：LidarDecoderLz4 已啟用

# 6. 網路連接測試
ping 192.168.123.161
# 結果：0% loss，網路穩定
```

### 關鍵發現

1. **LiDAR 資料頻率極低**
   - `/point_cloud2` 僅 0.3-0.8 Hz（目標 ≥10 Hz）
   - `/scan` 不穩定，最低僅 0.1 Hz
   - 推測：WebRTC 頻寬瓶頸或 Go2 原生 LiDAR 輸出限制

2. **SLAM 節點無法正常運作**
   - `/slam_toolbox/feedback` 無 publisher 輸出
   - `/map` 全為 -1，無建圖
   - 可能原因：輸入資料頻率不足以觸發 SLAM 算法

3. **QoS 設定為 BEST_EFFORT**
   - `/point_cloud2` 使用 BEST_EFFORT，可能導致資料丟棄
   - 建議測試 RELIABLE 設定是否改善

4. **網路連接正常**
   - ping 192.168.123.161 0% loss
   - 排除網路不穩定導致的問題

5. **LZ4 解碼器已啟用**
   - LidarDecoderLz4 正常運作
   - 解碼層面無問題

### 可能的根因分析

| 可能原因 | 可能性 | 說明 |
|---------|-------|------|
| WebRTC 頻寬限制 | 🔴 高 | LiDAR 點雲資料量大，WebRTC 可能優先壓縮影像 |
| Go2 原生限制 | 🟡 中 | Go2 L1 LiDAR 原始輸出頻率可能受限 |
| QoS 設定 | 🟡 中 | BEST_EFFORT 可能丟棄關鍵資料 |
| CPU 資源不足 | 🟢 低 | 其他 topic 頻率正常 |
| 網路設定 | 🟢 低 | ping 無 loss，排除此因素 |

### 下一步行動

1. **嘗試 CycloneDDS 模式**
   - 比較 WebRTC 與 CycloneDDS 的 LiDAR 資料頻率
   - CycloneDDS 理論上更適合高頻感測器資料

2. **測試 RELIABLE QoS**
   - 修改 `/point_cloud2` QoS 設定為 RELIABLE
   - 觀察是否改善資料丟失

3. **檢查 Go2 SDK 參數**
   - 確認是否有 LiDAR 頻率限制參數
   - 查看 `go2_driver_node.py` 中是否有節流設定

4. **監控系統資源**
   - 使用 `htop` 監控 CPU/記憶體使用率
   - 排除資源瓶頸

5. **考慮降級方案**
   - 若無法解決，考慮使用 RealSense D435i 進行視覺 SLAM
   - 或整合 Point-LIO（CMU 專案驗證適合 Go2 L1）

## Part 2: Go2 Pro 感測器能力盤點

### 內建感測器

| 感測器 | 規格 | 用途 | 限制 |
|--------|------|------|------|
| **4D LiDAR L1** | 360°×90° FOV, 21,600 samples/s, min 0.05m | SLAM、避障 | 低成本有噪聲，**無法偵測 <0.3m 障礙** |
| **前置相機** | 1080P@15fps, F2.2, 120° FOV | 影像辨識 | 無深度資訊 |
| **IMU** | 集成在 LiDAR 內 | 姿態估計 | 需校準 |
| **深度相機** | 僅標註有，規格不明 | 3D 環境 | Go2 Pro 實測無法使用 |

### L1 LiDAR 詳細規格

- **採樣率：** 21,600 次/秒
- **視野：** 360° × 90° 半球形
- **最小偵測距離：** 0.05m（極小盲區）
- **4D 含義：** 3D 位置 + 1D 灰度（強度）

### 已知限制（CMU autonomy_stack_go2 確認）

> "L1 lidar 是低成本感測器，噪聲相對較高。該感測器無法區分低障礙物"

⚠️ **重要：** 所有障礙物需 **>0.3m 高度** 才能被偵測

## Part 4: Jetson Orin Nano + RealSense D435 整合方案

### 硬體選擇建議

#### Jetson Orin Nano

| 型號 | VRAM | 價格 | 適用場景 |
|------|------|------|----------|
| 8GB | 8GB | ~$14,800 | 輕量 AI + 遠端 GPU 協作 |
| 16GB | 16GB | ~$28,000+ | 全本地運算 |

**會議決議：** 先測試 8GB 模型能力，再決定

#### RealSense D435 vs D435i

| 項目 | D435 | D435i |
|------|------|-------|
| 深度 | ✅ | ✅ |
| RGB | ✅ | ✅ |
| IMU | ❌ | ✅ |
| SLAM 適用性 | 需外部 IMU | 建議，內建 IMU |
| FOV | 87° × 58° | 87° × 58° |
| 有效距離 | 0.3-3m | 0.3-3m |

**目前狀態：** 你們有 D435（無 IMU）

### D435 無 IMU 的因應方案

1. **使用 Go2 內建 IMU**
   - Go2 的 L1 LiDAR 內建 IMU
   - 可透過 `/imu` topic 取得資料
   - 需處理時間戳同步問題

2. **使用 LiDAR SLAM 而非 VSLAM**
   - nvblox 可以用 LiDAR 資料建圖
   - 不依賴相機 IMU
   - D435 只做深度感知 + 物件偵測

3. **考慮升級到 D435i（建議）**
   - 價差約 $30-50 USD
   - 內建 IMU 可同步深度資料，VSLAM 效果更好
   - 若預算允許，建議採購 D435i

### NVIDIA Isaac ROS 方案

官方支援的 SLAM 方案：

| 方案 | 說明 | 優勢 |
|------|------|------|
| **cuVSLAM** | GPU 加速視覺 SLAM | 30+ FPS，支援多相機 |
| **nvblox** | 3D 重建 + costmap | Vision-only 避障 |
| **Isaac Perceptor** | 完整感知堆疊 | Lidar-free 導航 |

**cuVSLAM + RealSense 整合：**
- 官方教學：https://nvidia-ai-iot.github.io/jetson_isaac_ros_visual_slam_tutorial/
- 預配置 SD 卡映像，最快速評估

**nvblox 避障：**
- 處理深度 + pose → 3D 場景重建
- 輸出 2D costmap 給 Nav2
- Vision-only 3D 避障，不需 LiDAR

### 7.1 Edge AI 替代方案比較

#### 硬體平台性價比分析

| 平台 | AI 性能 | TDP | 價格 | 性能/美元 | 推薦場景 |
|------|---------|-----|------|----------|----------|
| **RK3588** (Orange Pi 5 Max) | 6 TOPS NPU | 5-10W | $180 | **1.36** ✅ | 預算受限、無 CUDA 需求 |
| Jetson Orin Nano | 40 TOPS GPU | 5-15W | $499 | 0.80 | CUDA 生態、機器人整合 |
| RPi5 + Hailo-8L | 13 TOPS | 低功耗 | ~$130 | - | 模組化、AI 加速器 |
| x86 Mini PC (N100) | CPU only | 6W | $150 | - | 通用計算、iGPU |
| x86 Mini PC (7840U) | 10 TOPS XDNA | 30W | $400 | - | 需要 iGPU + NPU |

**關鍵發現：RK3588 性價比最高（1.36 FPS/$），但缺乏 CUDA 生態**

#### JetPack 6.2 Super Mode（重大發現）

## 2.1 機器人作業系統 ROS2 Humble

### 2.1.1 ROS2 簡介

ROS2（Robot Operating System 2）是目前機器人領域最廣泛使用的開源軟體框架，由 Open Robotics 組織維護開發。相較於前一代 ROS1，ROS2 在以下方面進行了重大改進：

| 特性 | ROS1 | ROS2 |
|------|------|------|
| 即時性支援 | ❌ 不支援 | ✅ 支援 RTOS |
| 多機器人協作 | 需額外配置 | ✅ 原生支援 |
| 安全性 | 基礎 | ✅ DDS 加密 |
| 平台支援 | 主要 Linux | ✅ Linux/Windows/macOS |
| Python 版本 | Python 2 | ✅ Python 3 |

本專題選擇 **ROS2 Humble Hawksbill**（2022 年 5 月發布）作為開發平台，這是目前最新的 LTS（Long Term Support）版本，官方支援至 2027 年。

### 2.1.2 ROS2 核心概念

ROS2 採用發布-訂閱（Publish-Subscribe）架構進行節點間通訊，以下為本專題使用的核心概念：

**1. 節點（Node）**

節點是 ROS2 的基本執行單元，代表一個獨立的程式或功能模組。本專題使用的主要節點包括：

- `go2_driver_node`：Unitree Go2 機器狗驅動程式
- `snapshot_service`：相機截圖服務
- `rosbridge_websocket`：WebSocket 通訊橋接器

**2. 話題（Topic）**

話題是節點間傳遞資料的通道，採用非同步通訊模式。本專題使用的關鍵話題包括：

| 話題名稱 | 訊息類型 | 用途 |
|---------|---------|------|
| `/camera/image_raw` | `sensor_msgs/Image` | 相機原始影像 |
| `/cmd_vel` | `geometry_msgs/Twist` | 機器狗速度控制 |
| `/scan` | `sensor_msgs/LaserScan` | LiDAR 雷射掃描資料 |
| `/odom` | `nav_msgs/Odometry` | 里程計資料 |

**3. 服務（Service）**

服務採用請求-回應（Request-Response）模式，適用於需要確認執行結果的操作。本專題自行開發的服務包括：

- `/capture_snapshot`：擷取當前相機畫面並儲存為 JPEG
- `/move_for_duration`：在指定時間內以指定速度移動
- `/stop_movement`：緊急停止所有移動

**4. 動作（Action）**

動作是服務的延伸，適用於長時間執行的任務，支援進度回報與取消功能。Nav2 導航框架即透過動作介面控制機器人移動至目標點。

### 2.1.3 本專題使用的 ROS2 套件

| 套件名稱 | 版本 | 用途 |
|---------|------|------|
| `go2_robot_sdk` | 自行開發 | Unitree Go2 驅動程式 |
| `rosbridge_server` | Humble | WebSocket 橋接，供 MCP 連線 |
| `slam_toolbox` | Humble | 即時 SLAM 建圖 |
| `nav2` | Humble | 自主導航（本次 Demo 保留未啟用）|
| `tf2` | Humble | 座標轉換框架 |

### 2.1.4 為何選擇 ROS2

本專題選擇 ROS2 而非其他機器人框架（如 YARP、MOOS-IvP）的原因如下：

1. **業界標準**：ROS2 是目前機器人產業最廣泛採用的框架，Unitree Go2 原廠即提供 ROS2 支援
2. **豐富生態系**：擁有完整的 SLAM、導航、視覺處理套件
3. **MCP 整合支援**：開源的 `ros-mcp-server` 專案提供 ROS2 與 MCP 協定的橋接
4. **跨平台支援**：可在 Ubuntu VM、Windows WSL2 等多種環境運行

## 2.2 Unitree Go2 四足機器人平台

### 2.2.1 Go2 簡介

Unitree Go2 是由中國宇樹科技（Unitree Robotics）於 2023 年發布的消費級四足機器人，定位為「人人可負擔的機器狗」。相較於 Boston Dynamics Spot（售價約 75,000 美元），Go2 以約 1,600 美元的價格提供類似的基礎功能，大幅降低四足機器人的研究門檻。

### 2.2.2 硬體規格

| 規格項目 | Go2 Air | Go2 Pro | Go2 EDU |
|---------|---------|---------|---------|
| 處理器 | 官方未揭露 | 官方未揭露 | NVIDIA Jetson Orin NX (8GB) |
| LiDAR | ❌ | ✅ Hesai XT16 | ✅ Hesai XT16 |
| 4G 模組 | ❌ | ✅ | ✅ |
| 開發支援 | 基礎 APP | 基礎 APP | ✅ 完整 ROS2 SDK |
| 連線方式 | Wi-Fi | Wi-Fi | Wi-Fi + Ethernet |
| 售價（USD） | ~$1,600 | ~$2,800 | ~$8,500 |

> 📝 **備註**：Air/Pro 版處理器規格官方未公開揭露，僅 EDU 版明確標示搭載 Jetson Orin NX。

**本專題使用版本：Go2 Pro（非 EDU 版）**

### 2.2.3 感測器配置

Go2 內建多種感測器，本專題主要使用以下三種：

**1. LiDAR 雷射雷達**

- 型號：Hesai XT16（16 線）
- 掃描範圍：360 度水平 / ±15 度垂直
- 掃描頻率：10 Hz（有線連線）/ 1-2 Hz（Wi-Fi 連線）
- 測距範圍：0.3m - 150m
- 用途：環境建圖（SLAM）、障礙物偵測

**2. 前置 RGB 相機**

- 解析度：1920×1080 (Full HD)
- 視野角：超過 120 度（廣角鏡頭）
- 幀率：30 FPS
- 用途：環境影像擷取、物件偵測

> ⚠️ **重要發現**：Go2 採用廣角鏡頭（FOV > 120°），而多數深度估計模型（如 Depth Anything V2）以標準鏡頭（60-90°）資料訓練，導致深度估計結果偏差。本專題透過實測校正解決此問題（詳見 2.5.1 節）。

**3. IMU 慣性測量單元**

- 包含：三軸加速度計、三軸陀螺儀
- 用途：姿態估計、里程計輔助

### 2.2.4 通訊架構

Go2 提供兩種主要連線方式：

| 連線方式 | 協定 | 優點 | 缺點 |
|---------|------|------|------|
| Wi-Fi | WebRTC | 無線便利 | 感測器頻率較低（~1-2 Hz）|
| Ethernet | CycloneDDS | 低延遲、高頻率 | 僅 EDU 版支援 |

本專題受限於硬體版本（非 EDU），僅能使用 **Wi-Fi + WebRTC** 連線。此限制影響 LiDAR 資料頻率，因此本專題採用「語意導航」策略（詳見 2.7 節）以適應低頻感測器資料。

### 2.2.5 座標系統

Go2 的 URDF（Unified Robot Description Format）定義了以下座標框架（Frame）：

| 座標框架 | 說明 | 備註 |
|---------|------|------|
| `base_link` | 機器狗本體中心 | 運動控制參考點 |
| `front_camera` | 前置相機 | ⚠️ 非 `camera_link` |
| `lidar_link` | LiDAR 感測器 | 點雲資料來源 |
| `imu_link` | IMU 感測器 | 姿態資料來源 |

> ⚠️ **注意**：Go2 預設 URDF 的相機座標框架名稱為 `front_camera`，而非一般常見的 `camera_link`。進行 TF 座標轉換時需特別注意。

## 2.3 SLAM 與自主導航概述

### 2.3.1 SLAM 技術簡介

SLAM（Simultaneous Localization and Mapping，同步定位與建圖）是移動機器人領域的核心技術，使機器人能在未知環境中同時建構地圖並定位自身位置。

**SLAM 的基本原理：**

```
感測器資料（LiDAR/Camera）
        ↓
    特徵提取
        ↓
    資料關聯（匹配已知特徵）
        ↓
    位姿估計（計算機器人位置）
        ↓
    地圖更新（加入新觀測）
        ↓
    迴圈閉合（修正累積誤差）
```

### 2.3.2 slam_toolbox 套件

本專題使用 `slam_toolbox` 進行即時建圖，這是 ROS2 生態系中最成熟的 2D SLAM 解決方案。

**主要特點：**

- 採用圖優化（Graph-based SLAM）架構
- 支援線上即時建圖與離線最佳化
- 可處理大規模環境（數百平方公尺）
- 產出 2D 佔據格點地圖（Occupancy Grid Map）

**建圖成果範例：**

本專題已成功建構室內環境地圖，地圖檔案格式為：
- `phase1.yaml`：地圖元資料（解析度、原點座標）
- `phase1.pgm`：地圖影像（灰階圖，黑色=障礙物、白色=通道、灰色=未知）

### 2.3.3 Nav2 導航框架

Nav2（Navigation2）是 ROS2 的標準自主導航解決方案，提供完整的路徑規劃與控制功能。

**Nav2 核心模組：**

| 模組 | 功能 |
|------|------|
| 全域規劃器（Global Planner）| 計算起點到終點的最佳路徑 |
| 局部規劃器（Local Planner）| 即時避開動態障礙物 |
| 成本地圖（Costmap）| 將地圖資訊轉換為導航成本 |
| 行為樹（Behavior Tree）| 控制導航狀態與異常處理 |

### 2.3.4 本專題的導航策略

本專題已完成 Nav2 整合驗證，但在 1/7 Demo 中選擇**暫不啟用**，原因如下：

| 挑戰 | 說明 |
|------|------|
| LiDAR 頻率過低 | Wi-Fi 連線僅 1-2 Hz，Nav2 建議 10 Hz 以上 |
| 座標轉換複雜 | 需精確的相機-LiDAR 外參校正 |
| 開發時程 | 距離 Demo 僅剩 23 天 |

**替代方案：語意導航（詳見 2.7 節）**

本專題採用「深度估計 + LLM 決策」的語意導航策略，避開傳統 Nav2 對高頻感測器的依賴，同時保留 Nav2 作為未來升級選項。

## 2.4 Model Context Protocol (MCP) 整合架構 ⭐

### 2.4.1 MCP 協定簡介

Model Context Protocol（MCP）是 Anthropic 公司於 2024 年底發布的開放標準協定，旨在讓大型語言模型（LLM）能夠安全、標準化地與外部工具和系統進行互動。

**MCP 的核心理念：**

> 「讓 AI 不只是會說話，還能動手做事。」

傳統上，LLM 僅能進行文字對話。透過 MCP 協定，LLM 可以：
- 呼叫外部 API
- 讀寫檔案系統
- 操控機器人
- 查詢資料庫

### 2.4.2 MCP 協定架構

MCP 採用 Client-Server 架構，定義了以下核心概念：

**1. MCP Host（主機）**

執行 LLM 的應用程式，例如：
- Claude Desktop
- Kilo Code（VS Code 插件）
- 自行開發的 Python 應用

**2. MCP Server（伺服器）**

提供工具功能的服務端程式，將外部系統的能力暴露給 LLM。本專題使用開源的 `ros-mcp-server`。

**3. MCP Protocol（協定）**

定義 Host 與 Server 之間的通訊格式，支援：
- `stdio`：標準輸入輸出（本機通訊）
- `HTTP/SSE`：網路通訊

**MCP 架構圖：**

```
┌─────────────────────────────────────────────────────────┐
│                    MCP Host (Kilo Code)                 │
│  ┌─────────────────────────────────────────────────┐   │
│  │              LLM (Claude 3.5 Sonnet)            │   │
│  └─────────────────────────────────────────────────┘   │
│                         │                               │
│                    MCP Protocol                         │
│                         │                               │
└─────────────────────────┼───────────────────────────────┘
                          │
              ┌───────────┴───────────┐
              │                       │
              ▼                       ▼
     ┌─────────────┐         ┌─────────────┐
     │ ros-mcp-    │         │ filesystem- │
     │ server      │         │ server      │
     └──────┬──────┘         └─────────────┘
            │
            │ WebSocket
```

本專題使用的 `ros-mcp-server` 提供 58 個 MCP 工具，涵蓋 ROS2 的所有核心功能：

| 類別 | 工具數量 | 範例 |

**關鍵工具介紹：**

**1. `call_service`**

呼叫 ROS2 服務，本專題用於觸發拍照與移動控制：

```python
# 呼叫拍照服務
call_service(
    service_name="/capture_snapshot",
    service_type="std_srvs/srv/Trigger",
    request={}
)

# 呼叫移動服務
call_service(
    service_name="/move_for_duration",
    service_type="go2_interfaces/srv/MoveForDuration",
    request={"linear_x": 0.3, "angular_z": 0.0, "duration": 2.0}
)
```

**2. `subscribe_once`**

訂閱單筆 Topic 資料，用於讀取感測器狀態：

```python
# 讀取相機影像
subscribe_once(
    topic_name="/camera/image_raw",
    message_type="sensor_msgs/msg/Image"
)
```

**3. `send_action_goal`（未來擴充）**

發送 Nav2 導航目標點：

```python
send_action_goal(
    action_name="/navigate_to_pose",
    action_type="nav2_msgs/action/NavigateToPose",
    goal={"pose": {"position": {"x": 1.0, "y": 0.0}}}
)
```

## 2.5 視覺感知系統

### 2.5.1 深度估計：Depth Anything V2 Metric

#### 2.5.1.1 技術背景

深度估計（Depth Estimation）是從 2D 影像推斷場景 3D 深度資訊的技術。傳統方法需要雙目相機或 LiDAR 等深度感測器，而基於深度學習的單目深度估計僅需單張 RGB 影像即可產生深度圖。

**Depth Anything V2** 是 2024 年由香港大學與 TikTok 聯合發表的單目深度估計模型，在多項基準測試中達到 SOTA（State-of-the-Art）水準。該模型提供 Relative（相對深度）與 Metric（絕對距離）兩種版本。

#### 2.5.1.2 模型選擇

本專題選擇 **Depth Anything V2 Metric Large** 版本（以下簡稱 DA2-Metric），主要考量如下：

| 版本 | 輸出格式 | VRAM 需求 | 推論速度 | 使用情境 |
|------|---------|----------|---------|---------|
| DA2-Relative-Small | 相對深度（0-1）| ~0.5 GB | ~50ms | 藝術應用 |
| DA2-Relative-Large | 相對深度（0-1）| ~1.5 GB | ~200ms | 研究用途 |
| **DA2-Metric-Large** | **絕對深度（公尺）**| **~1.5 GB** | **~230ms** | **機器人應用 ✓** |

> **選擇 Metric 版本的原因**：機器人避障需要「前方 0.8 公尺有障礙物」這樣的絕對距離資訊，而非「深度值 0.7」這樣的相對比例。Metric 版本使用 Hypersim 室內資料集訓練，適合居家環境。

#### 2.5.1.3 Go2 廣角鏡頭校正問題

**問題發現：**

在實測過程中，我們發現 DA2-Metric 持續高估距離約 50-60%。例如：
- 實際距離 0.6m → DA2-Metric 輸出 1.16m（高估 93%）
- 實際距離 1.0m → DA2-Metric 輸出 1.51m（高估 51%）

**原因分析：**

Go2 採用廣角鏡頭（FOV > 120°），而 DA2-Metric 訓練資料多來自標準鏡頭（FOV 60-90°）。廣角鏡頭的透視變形導致模型誤判距離。

**解決方案：實拍校正**

我們透過 Go2 相機在多個已知距離拍攝校正圖，計算校正係數：

| 實際距離 | DA3 原始輸出 | 校正係數 |
|---------|-------------|---------|
| 0.6m | 1.16m | 0.52 |
| 1.0m | 1.51m | 0.66 |
| **平均** | - | **0.60** |

**校正結果驗證：**

| 測試距離 | 校正後輸出 | 誤差 | 結果 |
|---------|-----------|------|------|
| 15cm | 14cm | -7% | ✅ 通過 |
| 30cm | 28cm | -7% | ✅ 通過 |
| 45cm | 41cm | -9% | ✅ 通過 |
| 60cm | 70cm | +17% | ✅ 通過 |
| 1.0m | 91cm | -9% | ✅ 通過 |

> **結論**：校正後在 15cm-1m 範圍內誤差 < 20%，完全滿足室內避障需求。

#### 2.5.1.4 Perception API 服務

本專題在 GPU 伺服器上部署 Perception API，提供深度估計與避障建議服務。

**技術規格：**

| 項目 | 數值 |
|------|------|
| 框架 | FastAPI |
| Port | 8050 |
| GPU | NVIDIA Quadro RTX 8000 (48GB) |
| 推論時間 | ~230-330ms（640×480 輸入）|
| VRAM 使用 | ~1.5 GB |

**API 端點：**

```
POST /perceive
輸入：JPEG 圖片
輸出：JSON（三區距離 + 避障建議）
```

**回應範例：**

```json
{
    "left_m": 1.31,
    "center_m": 0.73,
    "right_m": 1.21,
    "front_obstacle_m": 0.54,
    "suggestion": "⚠️ 正前方 0.5m 有障礙，建議向左繞行（左側 1.3m 較空曠）",
    "inference_ms": 326.4
}
```

#### 2.5.1.5 區域分析與避障決策

Perception API 將影像分為左、中、右三個區域進行深度分析：

```
┌────────────────────────────────────┐
│    左區域    │   中央區域   │   右區域    │
│   (left_m)   │  (center_m)  │  (right_m)  │
│              │              │             │
│    2.3m      │    0.8m      │    1.5m     │
└────────────────────────────────────┘
              ▲
              │
      front_obstacle_m = 0.54m
      (中央下半部最近障礙物)
```

**避障決策邏輯：**

| 條件 | 建議 |
|------|------|
| `front > 1.0m` | ✅ 前方暢通，可安全前進 |
| `front < 1.0m` 且 `right > left` | ⚠️ 建議向右繞行 |
| `front < 1.0m` 且 `left > right` | ⚠️ 建議向左繞行 |
| `left < 0.5m` 且 `right < 0.5m` | 🛑 三面受阻，建議後退 |

### 2.5.2 物件偵測：YOLO-World

#### 2.5.2.1 YOLO 技術演進

YOLO（You Only Look Once）是目前最廣泛使用的即時物件偵測演算法，由 Joseph Redmon 於 2015 年首次提出。相較於傳統的 R-CNN 系列演算法，YOLO 採用單階段（One-stage）架構，將物件偵測視為端對端的迴歸問題，大幅提升偵測速度。

**YOLO 版本演進：**

| 版本 | 年份 | 主要改進 |
|------|------|---------|
| YOLOv1 | 2015 | 首創單階段偵測 |
| YOLOv3 | 2018 | 多尺度偵測、殘差網路 |
| YOLOv5 | 2020 | PyTorch 實現、易於部署 |
| YOLOv8 | 2023 | 更高精度、更快速度 |
| **YOLO-World** | **2024** | **開放詞彙偵測** |

#### 2.5.2.2 YOLO-World 開放詞彙偵測

本專題採用 **YOLO-World-L**（Large 版本），這是 2024 年發表的開放詞彙物件偵測模型，具備以下創新特性：

**傳統 YOLO vs YOLO-World：**

| 特性 | 傳統 YOLO | YOLO-World |
|------|----------|------------|
| 偵測類別 | 固定 80 類（COCO） | **任意文字描述** |
| 擴充方式 | 需重新訓練 | 修改文字提示即可 |
| 適用場景 | 通用物件 | **自定義目標物** |

**開放詞彙的優勢：**

傳統 YOLO 只能偵測 COCO 資料集中的 80 類物件（如 person、car、chair）。若要偵測「藥盒」或「眼鏡盒」等非標準類別，需要收集大量標註資料並重新訓練模型。

YOLO-World 透過視覺-語言預訓練，可接受任意文字描述作為偵測目標：

```python
# 傳統 YOLO：只能偵測預定義類別
detect(image, classes=[0, 39, 41])  # person, bottle, cup

# YOLO-World：可指定任意文字
detect(image, prompts=["水瓶", "眼鏡", "藥盒", "遙控器"])
```

## 2.6 生成式 AI 與 LLM Agent

### 2.6.1 大型語言模型（LLM）簡介

大型語言模型（Large Language Model, LLM）是基於 Transformer 架構的深度學習模型，透過海量文字資料訓練，具備理解與生成自然語言的能力。

**LLM 的核心能力：**

| 能力 | 說明 | 本專題應用 |
|------|------|-----------|
| 語意理解 | 理解使用者意圖 | 「幫我找眼鏡」→ 理解為尋物任務 |
| 推理規劃 | 分解任務步驟 | 規劃巡視路線與避障策略 |
| 上下文記憶 | 記住對話歷史 | 追蹤尋物進度 |
| 程式碼生成 | 生成可執行指令 | 產生 ROS2 服務呼叫 |

### 2.6.2 視覺語言模型（VLM）

視覺語言模型（Vision Language Model, VLM）是結合視覺與語言能力的多模態 AI 模型，可同時理解圖像與文字。

**VLM 的工作流程：**

```
輸入圖像 + 文字提問
        │
        ▼
┌─────────────────────┐
│   Vision Encoder    │  ← 提取圖像特徵
└─────────────────────┘
        │
        ▼
┌─────────────────────┐
│   Language Model    │  ← 結合文字與圖像特徵
└─────────────────────┘
        │
        ▼
    文字回應
```

**本專題的 VLM 應用場景：**

```
輸入：Go2 相機截圖 + 「前方有什麼障礙物？」
輸出：「前方約 1 公尺處有一個紙箱，建議向右繞行。」
```

### 2.6.3 本專題使用的 LLM

經過多款模型評估，本專題選用以下 LLM：

|------|--------|------|------|------|

| **Mistral Large** | Mistral AI | $0.23/次 | 高 CP 值 | 視覺能力較弱 |

**最終選擇：Claude 3.5 Sonnet（Demo 用）+ Amazon Nova 2 Lite（開發測試用）**

## 2.7 語意導航策略 ⭐⭐

> **本節為專題核心創新，詳細說明本專題與傳統機器人導航方案的差異。**

### 2.7.1 傳統導航方案的挑戰

傳統機器人導航（如 Nav2）採用精確座標導航，需要以下技術支援：

**傳統導航的技術需求：**

```
LiDAR 掃描 (10+ Hz)
        ↓
    SLAM 建圖
        ↓
    全域路徑規劃
        ↓
    局部避障
        ↓
    速度控制
```

**本專題面臨的困境：**

| 挑戰 | 說明 | 影響 |
|------|------|------|
| LiDAR 頻率過低 | Wi-Fi 連線僅 1-2 Hz（Nav2 建議 10+ Hz）| 無法即時避障 |
| 座標轉換複雜 | 需相機內參、LiDAR 外參校正 | 開發時間長 |
| 精度要求高 | 差幾公分可能導致碰撞 | 測試困難 |
| 動態障礙 | 傳統方案需快速反應 | 低頻感測器無法勝任 |

### 2.7.2 語意導航：全新思維

本專題提出「語意導航」策略，核心理念是：

> **「給 LLM 吃『熟食』，而非『生肉』。」**

**什麼是「生肉」？**

傳統方案將原始感測器資料直接餵給決策模組：
- LiDAR：360 個浮點數（每個角度的距離值）
- 相機：640×480×3 = 921,600 個像素值
- IMU：6 個軸的加速度與角速度

**什麼是「熟食」？**

本專題先將感測器資料處理為語意化摘要，再交給 LLM 決策：
- ✅ 「正前方 0.8 公尺有障礙物」
- ✅ 「左側較空曠，建議左轉」
- ✅ 「偵測到水瓶，位於右前方 1.2 公尺」

### 2.7.3 傳統方案 vs 語意導航對比

| 項目 | 傳統座標導航 | 本專題語意導航 |
|------|------------|--------------|
| **指令格式** | `goto(x=1.5, y=0.3)` | 「往前走，有障礙物就繞開」|
| **感測器需求** | LiDAR 10+ Hz | Snapshot + DA3（容忍 1-2 Hz）|
| **座標轉換** | 需精確的 TF 轉換 | ❌ 不需要 |
| **避障決策** | Costmap + Local Planner | LLM 語意判斷 |
| **開發複雜度** | 高（需校正、調參）| 低（更新 Prompt 即可）|
| **錯誤處理** | 需預定義所有情境 | LLM 可處理未知情況 |
| **適應性** | 固定邏輯 | 可透過對話調整行為 |

### 2.7.4 語意導航資料流

```
步驟 1：使用者下達指令
「Go2，幫我找桌上的水。」
              │
              ▼
步驟 2：LLM 規劃任務
「收到！我來幫您找水。先拍照觀察環境...」
              │
              ▼
步驟 3：拍照 + 深度估計
呼叫 /capture_snapshot → 上傳至 Perception API
              │
              ▼
步驟 4：接收語意化摘要
{
    "front_obstacle_m": 0.8,
    "suggestion": "⚠️ 正前方 0.8m 有障礙，建議右轉"
}
              │
              ▼
步驟 5：LLM 決策
「前面有東西，我往右繞一下...」
呼叫 /move_for_duration {angular_z: -0.5, duration: 2.0}
              │
              ▼
步驟 6：確認安全後繼續
再次拍照 → 確認前方暢通 → 前進
              │
              ▼
步驟 7：找到目標
「爺爺，水在前面桌上喔！」
```

### 2.7.5 為何不使用傳統座標轉換

本專題最初計畫採用傳統的座標轉換方案（將相機偵測的 2D 像素座標轉換為 3D 世界座標），但經評估後放棄，原因如下：

**技術複雜度高：**

```
傳統座標轉換流程：

相機圖像 (u, v)
        ↓
    相機內參矩陣 K
        ↓
    相機座標 (Xc, Yc, Zc)
        ↓
    相機-LiDAR 外參 T
        ↓
    LiDAR 座標
        ↓
    LiDAR-base_link 轉換
        ↓
    base_link 座標
        ↓
    TF2 查詢
        ↓
    世界座標 (X, Y, Z)
```

**實際困難：**

1. **相機內參校正**：需棋盤格標定，Go2 廣角鏡頭變形嚴重
2. **外參校正**：需精確對齊相機與 LiDAR，誤差會累積
3. **時間同步**：相機與 LiDAR 時間戳需對齊
4. **維護成本**：任何硬體調整都需重新校正

**語意導航的優勢：**

| 問題 | 傳統方案 | 語意導航 |
|------|---------|---------|
| 相機校正 | 需精確校正 | ❌ 不需要 |
| 座標轉換 | 多級轉換 | DA3 直接估計距離 |
| 誤差累積 | 嚴重 | 校正係數補償 |
| 開發時間 | 數週 | 數天 |

### 2.7.6 創新亮點總結

本專題的「語意導航」策略具備以下創新價值：

1. **降低硬體門檻**：不需要 EDU 版本的高頻感測器
2. **簡化開發流程**：無需複雜的座標轉換與校正
3. **提升適應性**：LLM 可處理模糊指令與未知情境
4. **人機互動友善**：使用者以自然語言溝通，機器狗以人話回應

> **核心創新**：本專題證明，在低頻感測器環境下，透過「深度估計 + LLM 語意決策」可實現可靠的避障導航，為低成本機器人應用提供新的技術路線。

## 2.8 安全設計與系統限制

### 2.8.1 Safety Layer 設計

LLM 雖然具備強大的語意理解能力，但可能產生「幻覺」（Hallucination），輸出不合理的數值。為確保機器狗不會因 LLM 錯誤輸出而發生危險，本專題設計了多層安全機制：

**安全層架構：**

```
┌─────────────────────────────────────────────────────┐
│                  LLM 原始輸出                        │
│    {"action": "move", "speed": 10.0, "duration": 5} │
│                 （可能的幻覺輸出）                   │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                  Safety Layer                        │
│  ┌─────────────────────────────────────────────┐    │
│  │              速度限制                        │    │
│  │   max_linear = 0.5 m/s                      │    │
│  │   max_angular = 1.0 rad/s                   │    │
│  └─────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────┐    │
│  │              時間限制                        │    │
│  │   max_duration = 10 秒                      │    │
│  └─────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────┐    │
│  │              數值驗證                        │    │
│  │   確保所有參數在合理範圍內                   │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                  安全的執行指令                      │
│    {"action": "move", "speed": 0.5, "duration": 5}  │
│                 （限速後的安全輸出）                 │
└─────────────────────────────────────────────────────┘
```

**速度限制設計：**

| 動作類型 | 最大速度 | 設計考量 |
|---------|---------|---------|
| 前進/後退 | 0.5 m/s | 人能輕鬆追上並攔下 |
| 左右轉 | 1.0 rad/s | 避免快速旋轉導致摔倒 |
| 單次移動時間 | 10 秒 | 限制最大移動距離 |

**緊急停止機制：**

```bash
# /stop_movement 服務
# 0.27 秒內完全停止
ros2 service call /stop_movement std_srvs/srv/Trigger
```

### 2.8.2 系統限制說明

本專題雖已達成核心功能，但仍存在以下限制：

#### 硬體限制

| 限制項目 | 影響 | 未來改善方向 |
|---------|------|-------------|

| **廣角鏡頭** | 深度估計需校正 | 已透過 SCALE_FACTOR 補償 |
| **電池續航** | 約 1-2 小時 | 設計低電量警告機制 |

#### 軟體限制

| 限制項目 | 影響 | 未來改善方向 |
|---------|------|-------------|
| **無語音介面** | 使用者需打字 | 第二階段整合 Whisper STT + TTS |
| **無物品記憶** | 每次需重新搜尋 | 第二階段整合 SQLite 記憶庫 |
| **無 Nav2 精確導航** | 依賴語意導航 | 高頻感測器可開啟 Nav2 |

#### 環境限制

| 限制項目 | 說明 |

| **充足光線** | 暗環境影響相機與深度估計 |
| **穩定網路** | Wi-Fi 斷線會中斷控制 |

#### 成本限制

## 2.9 本章小結

本章介紹了「老人與狗」專題所使用的核心技術，涵蓋以下主題：

1. **ROS2 Humble**：機器人作業系統，提供標準化的通訊框架
2. **Unitree Go2**：四足機器人平台，具備 LiDAR、相機、IMU 感測器
3. **SLAM 與 Nav2**：建圖與導航技術，已驗證可用但暫未啟用
4. **MCP 協定**：LLM 與 ROS2 的橋接標準，本專題核心創新之一
5. **Depth Anything V2**：深度估計模型，提供障礙物距離資訊
6. **YOLO-World**：開放詞彙物件偵測，可識別任意目標物品
7. **LLM Agent**：扮演「機器狗大腦」，負責理解指令與決策行動
8. **語意導航**：本專題最大創新，簡化傳統座標導航的複雜性
9. **安全設計**：多層防護機制，確保機器狗安全運行

**三大創新亮點：**

| 創新點 | 說明 |
|--------|------|
| ⭐ MCP + ROS2 整合 | 國內首例將 MCP 協定應用於 ROS2 機器人控制 |
| ⭐⭐ 語意導航策略 | 不使用傳統座標轉換，改由 LLM 進行語意化決策 |
| ⭐ DA3 廣角校正 | 解決 Go2 廣角鏡頭導致深度估計偏差的實際工程問題 |

透過上述技術的整合，本專題實現了「讓機器狗聽懂人話、會找東西」的目標，為長者居家照護提供創新的 AI 解決方案。

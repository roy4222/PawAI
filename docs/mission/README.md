# GO2 邊境牧羊犬 AI 夥伴系統：雙層架構互動功能設計

## 1. 系統架構願景：從導航機器人到情感化 AI 夥伴

### 1.1 核心設計哲學

#### 1.1.1 「邊境牧羊犬」仿生互動典範：高智商、敏捷反應、深度情感連結

邊境牧羊犬作為犬類智商排名的頂端物種，其與人類的互動模式為四足機器人提供了極具價值的仿生學參考。這種工作犬種展現出三大核心特質：**極高的情境理解能力**——能夠從主人的語調、手勢、眼神中解讀複雜指令，甚至預判意圖；**毫秒級的反應速度**——在牧羊工作中即時響應突發狀況，同時保持動作的流暢與優雅；**深度的情感連結機制**——透過持續的互動學習建立與主人的獨特默契，形成非語言的溝通管道。

將這些特質轉化為 GO2 的技術規格，意味著系統必須超越傳統「指令-執行」的機械式互動，建立**多模態感知融合**、**預測性行為生成**與**個性化適應學習**的智能架構。具體而言，GO2 需要實現亞秒級的視覺-語音-空間融合理解，讓機器狗能夠在主人開口前便察覺其需求；運動控制層面需達成如同生物神經肌肉系統般的平滑動力學輸出，消除任何可察覺的「機器感」；認知層面則需建立持續更新的用戶模型，記錄互動歷史、偏好模式、情緒狀態，使每次對話都基於累積的上下文而非孤立事件。這種仿生設計的最終目標，是讓使用者產生「這不是一台機器，而是一隻懂我的數位牧羊犬」的主觀體驗。

#### 1.1.2 OpenClaw 式個人 AI 超級助理：主動服務、多模態理解、持續學習

OpenClaw 專案所代表的「個人 AI 超級助理」範式，為 GO2 的功能邊界提供了前瞻性參照。該範式的核心在於**從被動響應轉向主動服務**——傳統語音助理等待用戶喚醒後才啟動處理，而超級助理則持續監測環境狀態、用戶行為模式、日程脈絡，在適當時機主動提供協助。對於 GO2 這一具備物理形態的 AI 助理而言，這種主動性具有獨特的實現優勢：機器狗的移動能力使其能夠**物理性地介入環境**，而非僅限於螢幕上的訊息呈現；其四足步態提供了**豐富的非語言表達管道**，透過姿態、速度、路徑選擇傳達「情緒」與「意圖」；其持續在場的特性則支持**真正的長期記憶與關係累積**，而非每次對話從零開始的無狀態互動。

多模態理解是實現這一目標的技術基礎。系統需能夠無縫融合「用戶指著廚房方向說『那個』」這類模糊指令中的視覺指向、語音線索、空間語境，推斷出具體目標物體。持續學習則要求系統具備「終身學習」能力——從每次互動中提取用戶偏好（如「用戶A總是在晚上九點後降低語音音量」）、累積領域知識（如「這個家的『書房』指的是東南角的房間」）、並逐步優化決策策略。

#### 1.1.3 雙層運算架構：快系統即時響應 + 慢系統深度推理的協同智能

本系統採用認知科學啟發的雙層架構設計，明確區分「快系統（System 1）」與「慢系統（System 2）」的功能邊界與協作機制。**快系統**由 **NVIDIA Jetson Orin Nano SUPER 8GB** 擔綱，專責感知覺層面的即時處理與反射性響應，設計目標是將端到端延遲壓縮至 **<100ms**。這包括基於 TensorRT 優化的視覺特徵提取（YOLOv8、MediaPipe）、輕量級語音喚醒與關鍵詞識別、緊急避障與基礎跟隨控制、以及簡單手勢的即時解析。快系統的運作特徵是高度並行化、確定性時序、本地自治——即使在與慢系統的通信中斷時，仍能維持核心安全功能與基礎互動能力。

**慢系統**由 **RTX 6000 Blackwell 48GB VRAM** 驅動，專注認知層面的深度推理與策略規劃，響應時標可放寬至 **1-3 秒**，但輸出品質需達到接近人類水平的智能表現。這涵蓋 Llama-3-70B 級大語言模型的完整對話生成、視覺-語言多模態融合理解（VLM）、複雜指令的任務分解與規劃、長期記憶的檢索與整合、以及高階情感與意圖推理。兩系統間的協作遵循**分級觸發與漸進細化**原則：快系統持續運行，對簡單情境直接響應；當偵測到複雜或新穎情境時，將壓縮後的語義摘要上傳至慢系統；慢系統的推理結果則以行為腳本或參數更新的形式下發，豐富快系統的反應模式庫。

### 1.2 硬體資源配置

#### 1.2.1 快系統（邊緣端）：Jetson Orin Nano SUPER 8GB + Go2 原生感測器陣列

Jetson Orin Nano SUPER 8GB 作為快系統的核心運算平台，其技術規格經過精心評估以匹配邊緣 AI 的嚴苛需求。該模組在 **MAXN SUPER 25W 模式**下可提供 **67 TOPS（Sparse）/ 33 TOPS（Dense）** 的 AI 運算效能，配合 **102 GB/s 的 LPDDR5 記憶體頻寬**，足以同時運行多個即時感知模型。具體功能分配包括：TensorRT 優化的 YOLOv8 目標檢測（約 5-10ms 推理延遲）、MediaPipe 手勢姿態估計（約 15-20ms）、輕量 ASR（Whisper.cpp Tiny 或 Small 模型，約 100-200ms）、以及 Nvblox 三維重建與 MPPI 控制器的持續運行。

感測器陣列的協同配置為：L2 LiDAR 提供 **360°×96° 的全域幾何感知**（64,000 points/s），RealSense D435 補強前方 **86°×57° 的精細深度測量**（尤其針對低矮障礙物），Go2 內建 RGB 相機負責 **120° 廣角的視覺語義提取**，而麥克風陣列（Go2 EDU 版本確認支援語音功能，包括離線語音交互、語音對講與音樂播放）為語音互動提供硬體基礎。這一配置的總功耗約 15-25W，可透過 Go2 內建電池或外掛行動電源供電，確保 1-2 小時的連續運作。

| 感測器 | 感知維度 | 視野/覆蓋範圍 | 更新頻率 | 核心功能 | 運算負載 |
|:---|:---|:---|:---|:---|:---|
| **Unitree L2 LiDAR** | 3D 點雲 + 反射強度 | 360° × 96° | 10 Hz (64K pts/s) | 全域建圖、LIO 里程計、長距障礙檢測 | 極低（硬體預處理）|
| **Intel RealSense D435** | 深度影像 + RGB | 86° × 57°（可旋轉）| 30 Hz (1280×720) | 局部 3D 重建、Nvblox ESDF 輸入、低矮障礙物補盲 | 中等（GPU 加速）|
| **Go2 內建 RGB 相機** | 彩色影像 | 120° 對角線 | 30 Hz (1920×1080) | 人臉識別、手勢檢測、VLM 輸入、VIO 輔助 | 高（AI 模型推理）|
| **Go2 麥克風陣列** | 音訊訊號 | 全向拾音 | 16 kHz 取樣 | 語音喚醒、ASR 前端、聲源定位 | 低（訊號預處理）|

#### 1.2.2 慢系統（雲端/本地伺服器）：RTX 6000 Blackwell 48GB VRAM 大模型推理引擎

RTX 6000 Blackwell 48GB VRAM 代表了當前工作站級 GPU 的頂尖效能水準。48GB 的超大顯存容量允許單卡部署 **Llama-3-70B 的 INT4 AWQ 量化版本**（約佔 40GB VRAM），或同時載入 **多個專精模型**（對話 LLM + VLM + TTS）進行協同推理。Blackwell 架構的第五代 Tensor Core 支援 FP8 精度運算，相較 Ampere 架構可實現 **2-3 倍的吞吐量提升**。對於本系統，慢系統的部署模式具有靈活性：若 GO2 主要於單一固定場所運作，可將 RTX 6000 配置為**本地邊緣伺服器**，透過 Wi-Fi 6E 或 2.5GbE 與 Jetson 連接，將通信延遲控制在 5-10ms；若需支援移動場景，則可透過 5G 網路連接雲端實例，此時延遲約 20-50ms，但仍可透過預測性請求和結果快取緩解體驗下降。

#### 1.2.3 感測器融合：L2 LiDAR、RealSense D435、Go2 RGB 相機、麥克風陣列

多模態感測器的融合策略直接決定了系統的感知上限。本系統採用 **「幾何-語義-時序」三層融合架構**：幾何層面，L2 LiDAR 與 RealSense D435 的點雲透過 ICP 配準融合，構建統一的三維空間表示；語義層面，Go2 RGB 相機的影像流與深度資訊對齊後，輸入至 VLM 進行開放詞彙的物件檢測與場景理解；時序層面，麥克風陣列的音訊流與視覺軌跡透過時間戳同步，實現聲源定位與說話者臉部關聯。四種感測器的數據在 Jetson Orin Nano 上進行時間同步與空間對齊後，形成統一的多模態感知流，為上層 AI 推理提供完整的環境表徵。

## 2. 多模態互動功能矩陣

### 2.1 語音互動系統

#### 2.1.1 語音識別（ASR）：Whisper 本地部署或雲端 API 混合架構

語音識別系統的設計需在**準確率、延遲、隱私、成本**四維度間取得平衡。OpenAI Whisper 系列模型提供了從 Tiny（39M 參數）到 Large-v3（1550M 參數）的完整尺寸光譜：

| 方案 | 模型規格 | Jetson 延遲 | RTX 6000 延遲 | 中文 CER | 適用場景 |
|:---|:---|:---|:---|:---|:---|
| Whisper.cpp Tiny | 39M, Q4 量化 | 80-120ms | 20-30ms | 12-18% | 安靜環境、簡單指令 |
| Whisper.cpp Small | 244M, Q4 量化 | 300-500ms | 80-120ms | 8-12% | 一般對話、多輪交互 |
| Whisper.cpp Medium | 769M, Q5 量化 | 1.2-2s | 250-400ms | 5-8% | 複雜語境、專業術語 |
| 雲端 API（Azure/阿里雲）| 未公開 | 150-300ms（含網路）| N/A | 3-5% | 備援方案、特殊需求 |

建議採用 **「本地喚醒 + 雲端識別」的混合策略**：Jetson 上常駐 Whisper Tiny 負責關鍵詞喚醒（「GO2」「嘿，夥伴」等），喚醒後的音訊流可選擇本地 Small 模型快速識別（簡單指令）或上傳至 RTX 6000 的 Medium 模型進行精確解析（複雜對話）。這種分級處理將平均交互延遲控制在 **200ms 以內**，同時保證了複雜場景的識別準確率。

#### 2.1.2 自然語言理解：Llama-2/Llama-3 大模型驅動的意圖解析與上下文管理

大語言模型的引入將 GO2 的語言理解能力從「關鍵詞匹配」躍升至「語義理解」層次。以 **Llama-3-70B-Instruct** 為例，其經過 RLHF 微調的對話版本，能夠處理高度口語化、上下文依賴甚至含混的用戶輸入。例如，用戶說「我有點渴了，幫我看一下廚房有什麼喝的」，傳統規則系統需要預先定義「渴→飲料→冰箱」的映射鏈，而 LLM 能夠基於常識推理直接生成「前往廚房→打開冰箱→取水瓶→返回」的動作序列。

上下文管理採用**滑動窗口 + 摘要壓縮**策略：近期 5-10 輪對話保留完整原文，更早的歷史則由模型自動生成摘要，與關鍵事實（如用戶偏好、承諾事項、情緒標記）一併存入向量資料庫。每次新對話開始時，檢索與當前查詢最相關的歷史摘要，注入系統提示詞，實現「越用越懂你」的個性化體驗。

#### 2.1.3 語音合成（TTS）：情感化語音輸出，支援語調、語速、情緒動態調整

語音合成是賦予 GO2「個性」的關鍵載體。技術選型上，快系統可採用 **Piper** 或 **F5-TTS** 等輕量級神經 TTS 模型（<100MB），支援基礎的語速調整與簡單情感標記；慢系統則部署 **Coqui TTS** 的 XTTS-v2 模型，實現細粒度的韻律控制與聲音克隆。情感驅動的語音參數調整需與機器人的整體情緒狀態機聯動：

| 情緒狀態 | 語速變化 | 基頻變化 | 音量變化 | 應用場景 |
|:---|:---|:---|:---|:---|
| 興奮 | +20-30% | +80-120 Hz | 增大 | 用戶回家、遊戲邀請 |
| 平靜 | 基準 | 基準 | 基準 | 常規狀態播報 |
| 體貼 | -15-20% | -30-50 Hz | 降低 | 用戶疲勞、夜間模式 |
| 警戒 | -10% | -20-40 Hz | 加重音 | 異常檢測、安全提醒 |

#### 2.1.4 對話記憶與個性化：長期記憶儲存、用戶偏好學習、持續對話脈絡

對話記憶系統採用**四層架構**：**工作記憶**（當前對話窗口，駐留於慢系統 RAM）、**情節記憶**（以天為單位的互動日誌，Markdown 格式）、**語義記憶**（提取的事實性知識，向量資料庫）、**程序記憶**（學習到的技能與偏好策略，LoRA 適配器形式）。記憶的自我維護機制借鑑 OpenClaw 的 Heartbeat 設計：慢系統定期檢視近期互動日誌，識別有價值資訊進行摘要提取與冗餘清理，整個過程無需用戶干預。

### 2.2 視覺互動系統

#### 2.2.1 人臉識別與追蹤：RealSense D435 + Go2 相機的多視角融合

人臉識別採用**級聯檢測-識別-追蹤**架構：MediaPipe Face Detection 實現 30+ FPS 的即時檢測，ArcFace 提取 512 維特徵嵌入，DeepSORT 維持跨幀身份一致性。多視角融合策略：當用戶正面朝向 RealSense D435 時，以高解析度深度資訊為主進行精細表情分析；當用戶側身或背對時，切換至 Go2 前置相機的廣角視野維持追蹤，並利用 LiDAR 點雲補償深度資訊。

#### 2.2.2 手勢識別：靜態手勢指令（停止、跟隨、坐下）與動態手勢軌跡

| 手勢類型 | 具體動作 | 技術實現 | 延遲目標 |
|:---|:---|:---|:---|
| 靜態-停止 | 手掌張開面向機器人 | MediaPipe Hands 關鍵點分類 | <50ms |
| 靜態-跟隨 | 招手動作 | 關鍵點幾何規則 | <50ms |
| 靜態-坐下 | 雙手下壓 | 關鍵點角度閾值 | <50ms |
| 動態-指向 | 食指持續指向某方向 | 關鍵點時序軌跡 + TCN | <200ms |
| 動態-巡邏 | 順時針畫圈 | 軌跡形狀識別 | <300ms |

動態手勢的識別部署於慢系統，快系統僅負責基於規則的簡單軌跡解析，以平衡延遲與準確率。

#### 2.2.3 表情識別：情緒狀態感知，驅動對應的行為反饋

表情識別基於 **MediaPipe Face Mesh** 的 468 個 3D 臉部網格點，提取視線方向、嘴部開合、眉毛姿態等特徵，輸入輕量級分類器預測 7 類基本情緒。情緒驅動的行為反饋映射：

| 檢測情緒 | 機器狗響應 | 步態調整 | 語音風格 |
|:---|:---|:---|:---|
| 快樂 | 主動靠近、提議互動 | 輕快小跑 | 興奮語調 |
| 疲勞 | 降低打擾、提供協助 | 緩慢移動 | 柔和語調 |
| 憤怒 | 保持距離、語音安撫 | 暫停主動行為 | 平靜低沉 |
| 驚訝 | 轉向聲源、詢問狀況 | 警戒姿態 | 關切語調 |

#### 2.2.4 物體與場景理解：開放詞彙檢測、語義分割、空間關係推理

開放詞彙檢測採用 **Grounding DINO + SAM** 組合：Grounding DINO 實現任意文本描述的目標檢測，SAM 提供像素級分割掩碼。空間關係推理結合 Nvblox 的三維語義地圖，將「桌上」「沙發旁」「門口」等描述解析為坐標系中的概率分佈，支持「去廚房拿桌上的水瓶」這類複雜指令的執行。

### 2.3 行為互動系統

#### 2.3.1 仿生步態情緒表達：興奮小跑、好奇探頭、安心趴伏、警戒僵直

基於 `go2_ros2_sdk` 的服務接口，實現以下情緒-步態映射：

| 情緒狀態 | Body Height | Foot Raise Height | Speed Level | 連續步態 | 視覺效果 |
|:---|:---|:---|:---|:---|:---|
| **興奮/快樂** | 0.35-0.40（較高）| 0.02-0.03（較高抬腿）| 0.6-1.0（快）| 啟用 | 輕快小跑、機身微擺「搖尾」|
| **好奇/探索** | 0.30-0.35（中等）| 0.01-0.02（中等）| 0.3-0.5（慢速）| 啟用 | 緩慢踱步、頻繁轉向觀察 |
| **安心/放鬆** | 0.25-0.30（低趴）| -0.02-0.00（貼地滑行）| 0.0（靜止）或 0.1-0.2 | 關閉 | 趴伏休息、偶爾抬頭環顧 |
| **警戒/緊張** | 0.40-0.45（最高）| 0.03-0.04（高抬腿預備）| 0.0（靜止）或突發 0.8-1.0 | 關閉 | 僵直立正、快速定向轉向 |

#### 2.3.2 頭部與機身姿態語言：注視追蹤、點頭回應、歪頭困惑、搖尾（機身微擺）

- **注視追蹤**：透過控制機身偏航與俯仰，使相機光軸持續對準用戶面部，模擬「眼神接觸」
- **點頭回應**：快速的機身俯仰振動（幅度 5-10°，頻率 2 Hz），用於確認指令或表達同意
- **歪頭困惑**：機身側傾 5-15° 並保持 1-2 秒，配合疑問語調，傳達「我沒理解」
- **搖尾模擬**：興奮狀態下機身輕微橫向擺動（幅度 2-5 cm，頻率 1-2 Hz）

#### 2.3.3 主動行為觸發：定時巡邏報告、異常事件提醒、陪伴模式自動啟動

| 主動行為 | 觸發條件 | 執行內容 | 用戶控制 |
|:---|:---|:---|:---|
| 定時巡邏 | 預設時間表（如每晚 10 點）| 執行覆蓋路徑，語音摘要報告異常 | 可修改時間/區域 |
| 異常提醒 | 檢測到異常移動/聲音/環境變化 | 立即前往查看，語音+手機推播報警 | 可設定敏感度 |
| 陪伴模式 | 用戶長時間靜坐/情緒低落/規律時段 | 安靜靠近，主動提議互動或靜默陪伴 | 可一鍵暫停 |

所有主動行為遵循**不打擾原則**：提供「靜音時段」設定，重要提醒分級，僅最高級別可突破靜音限制。

## 3. 雙層架構功能分配與通信機制

### 3.1 快系統（Jetson Orin Nano）職責

#### 3.1.1 即時感知處理：點雲融合、視覺特徵提取、音訊前端處理

快系統的感知處理需在 **100ms 內完成端到端閉環**。點雲融合以 30 Hz 運行 Nvblox，輸出 ESDF 距離場與語義標註的動態物體；視覺特徵提取採用分級策略——YOLOv8-nano 全幀率運行輸出邊界框，ROI 區域緩存供慢系統精細分析；音訊前端包括波束成形、AEC、VAD 與關鍵詞喚醒，僅在觸發後啟動完整 ASR。

#### 3.1.2 低延遲反應：緊急停止、基礎跟隨、簡單手勢響應（< 100ms）

| 功能 | 延遲目標 | 實現機制 | 降級策略 |
|:---|:---|:---|:---|
| 緊急停止 | <50ms | LiDAR 近距檢測硬體中斷 | 無條件最高優先級 |
| 基礎跟隨 | <100ms | 視覺伺服 PID 控制 | 視覺失效時切 LiDAR |
| 手勢響應 | <100ms | MediaPipe 直接映射行為樹 | 複雜手勢上傳慢系統 |

#### 3.1.3 本地輕量 AI：TensorRT 優化的 YOLO、MediaPipe 手勢、嵌入式 ASR

所有邊緣模型經 TensorRT 優化，INT8/FP16 量化，預先載入固定記憶體，消除動態分配延遲。總記憶體佔用控制在 4GB 以內，為導航系統預留充足空間。

#### 3.1.4 導航與控制：Nvblox ESDF、MPPI 控制器、高階步態指令發布

導航系統延續前文詳述的 MPPI 控制器配置，關鍵在於與互動功能的整合：情緒狀態標籤動態調整 MPPI 的 `gamma`、`vx_max` 等參數，實現「情感驅動的運動風格」。

### 3.2 慢系統（RTX 6000）職責

#### 3.2.1 大模型推理：Llama-2/3 70B 級對話、VLM 視覺問答、複雜指令規劃

| 模型 | 規格 | 顯存佔用 | 典型延遲 | 應用場景 |
|:---|:---|:---|:---|:---|
| Llama-3-70B-Instruct | INT4 AWQ | ~40GB | 500-1500ms | 開放對話、常識推理 |
| Qwen2.5-VL-72B | INT4 AWQ | ~42GB | 800-2000ms | 視覺問答、場景描述 |
| XTTS-v2 | FP16 | ~3GB | 200-500ms | 情感語音合成 |

採用 **vLLM + TensorRT-LLM** 混合部署，連續批次與 PagedAttention 最大化吞吐量。

#### 3.2.2 多模態融合推理：視覺+語音+語義的聯合理解與決策

核心場景：用戶說「把那個放過去」同時指向某方向。慢系統執行：語言指代消解 → 視覺指向向量提取 → 空間語義地圖查詢 → 目標物體定位 → 動作規劃生成。整個流程端到端延遲 <2s，結果下發快系統執行。

#### 3.2.3 長期記憶與學習：向量資料庫檢索、經驗累積、行為模型微調

- **向量資料庫**：ChromaDB/Milvus 儲存對話嵌入，支援語義檢索
- **經驗累積**：結構化事件記錄（時間、地點、人物、結果）
- **行為微調**：用戶反饋驅動的 LoRA 適配器更新，週期性離線訓練

#### 3.2.4 高階任務規劃：自然語言指令分解、子目標生成、執行策略優化

複雜指令如「幫我準備明天野餐的東西」分解為：檢查清單生成 → 家中庫存檢查 → 缺失物品標記 → 採購路徑規劃 → 執行確認。每個子目標可進一步細化，形成層次化的任務樹。

### 3.3 跨層通信架構

#### 3.3.1 通信協議選擇：ROS 2 DDS 分佈式節點 vs HTTP/gRPC API 服務

| 層級 | 協議 | 應用場景 | 關鍵配置 |
|:---|:---|:---|:---|
| 快系統內部 | ROS 2 DDS（CycloneDDS）| 感測器融合、控制迴路 | 自定義 cyclonedds.xml，放大 MaxMessageSize |
| 快→慢 | HTTP/2 + gRPC | 語義摘要上傳、模型請求 | 連接池、超時重試、壓縮傳輸 |
| 慢→快 | ROS 2 Topic | 行為腳本、參數更新 | QoS Reliable + Transient Local |

#### 3.3.2 數據流設計：感測器原始數據 → 特徵壓縮 → 語義摘要的分級上傳

```
原始數據流（快系統本地）：
- LiDAR 64K pts/s → 降採樣 10K pts → 特徵提取 → 語義標註（人/物/障礙）
- RGB 1920×1080@30fps → 關鍵幀提取 → CLIP 嵌入（512-dim）
- 音訊 16kHz → VAD 分段 → 語音特徵（MFCC/Log-Mel）

  "robot_pose": {"x": 1.2, "y": 3.4, "theta": 0.5},
  "detected_humans": [{"id": "user_A", "face_emb_hash": "a1b2c3", "emotion": "happy"}],
  "scene_caption": "客廳，沙發上有人坐著，茶几上有紅色杯子",
  "voice_transcript": "幫我拿那個",
  "pointing_direction": {"azimuth": 45, "elevation": -10},
  "query_intent": "fetch_object_with_deixis"

```

#### 3.3.3 延遲優化策略：預測性請求、結果快取、漸進式細化

- **預測性請求**：基於對話上下文預載可能需要的模型（如提到「廚房」時預載該區域語義地圖）
- **結果快取**：常見查詢的 KV Cache 持久化，避免重複計算
- **漸進式細化**：先返回快速但不完整的響應（「好的，我去看看」），後台繼續推理再更新

#### 3.3.4 離線容錯機制：快系統自主降級運行、關鍵功能本地備援

網路中斷時，快系統自動切換至**純本地模式**：保留緊急停止、基礎跟隨、簡單語音指令（Whisper Tiny）、預設行為腳本。待連接恢復後，異步同步離線期間的互動記錄，由慢系統進行後處理與記憶更新。

### 4.1 日常陪伴模式

#### 4.1.1 主動問候與閒聊：時間感知觸發、話題推薦、情緒陪伴對話

| 觸發條件 | 機器狗行為 | 技術實現 |

### 4.2 任務執行模式

#### 4.2.1 語音指令導航：「去廚房拿桌上的水瓶」→ 語義解析 → 路徑規劃 → 物體定位

| 步驟 | 處理模組 | 關鍵技術 | 延遲預算 |
|:---|:---|:---|:---|
| 語音識別 | 快/慢系統混合 | Whisper + 領域適應 | 200-500ms |
| 意圖解析 | 慢系統 LLM | 語義角色標註 + 槽位填充 | 300-800ms |
| 目標定位 | 慢系統 VLM | Grounding DINO + 語義地圖 | 500-1500ms |
| 路徑規劃 | 快系統 Nav2 | Smac Planner + ESDF | 100-300ms |
| 執行反饋 | 快系統 TTS | 狀態播報 + 異常報告 | 實時 |

#### 4.2.2 手勢遙控互動：指向方向移動、手掌停止、圈選區域巡邏

- **指向移動**：食指持續指向，機器狗以 0.3-0.5m/s 速度向該方向移動，手指放下即停止
- **手掌停止**：任何時候檢測到停止手勢，立即執行零速度指令
- **圈選巡邏**：用戶在空中畫閉合曲線，機器狗記錄軌跡並重複巡邏該區域

#### 4.2.3 多步驟複雜任務：條件判斷、循環執行、錯誤恢復與人類確認

示例任務：「檢查每個房間的窗戶是否關好，沒關的幫我關上，關不了的告訴我」

```
FOR each_room IN home_map.rooms:
    NAVIGATE_TO(each_room)
    windows = DETECT_OBJECTS("window")
    FOR window IN windows:
        state = CHECK_STATE(window)  # VLM 判斷開/關/不確定
        IF state == "open":
            result = ATTEMPT_CLOSE(window)  # 若配備機械臂
            IF result.failed:
                REPORT_TO_USER(f"{each_room} 的 {window} 無法關閉，原因：{result.reason}")
        ELIF state == "uncertain":
            REQUEST_USER_CONFIRMATION(f"請確認 {each_room} 的 {window} 狀態")
```

## 6. 系統整合與性能優化

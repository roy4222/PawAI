## 技術選型結論（2026-03-25 更新）

### 主線方案：MediaPipe Pose（CPU-only）

| 優先序 | 方案 | 理由 |
|:------:|------|------|
| **主線** | **MediaPipe Pose** | CPU 18.5 FPS（Jetson 實測），GPU 0%，三感知壓測通過（RAM 1.2GB, temp 52°C），3/23 場景驗證通過 |
| 備援 | **rtmlib + RTMPose lightweight** | GPU 加速可用，但 balanced mode 91-99% GPU 滿載 |
| 備援 | trt_pose (NVIDIA) | TensorRT 原生、Jetson Nano 上社群值：ResNet 15-16 FPS / DenseNet 9-10 FPS |
| 不推薦 | MoveNet | 只有 17 keypoints、無手部、Jetson GPU delegate 有問題 |

> **決策變更紀錄（3/21）**：原推薦 RTMPose wholebody 為主路徑，但 Jetson 實測 GPU 91-99% 滿載。3/21 決策改為全 MediaPipe CPU pipeline，GPU 0% 的特性有利於多感知共存。

### 與手勢辨識的架構關係

**現行架構（3/25）**：MediaPipe Pose + MediaPipe Gesture Recognizer 各自獨立運行，皆為 CPU-only，GPU 0%。

```
D435 camera frame
  ↓
vision_perception_node（統一入口）
  ├── MediaPipe Pose（CPU, 18.5 FPS）→ pose_classifier → /event/pose_detected
  └── MediaPipe Gesture Recognizer（CPU, 7.2 FPS）→ /event/gesture_detected
```

## 姿勢分類方法

### 方法一：角度法（推薦，最直觀）

| 角度 | Landmarks | 站立 | 坐下 | 蹲下 | 跌倒 |
|------|-----------|:----:|:----:|:----:|:----:|
| **髖關節角度** (肩→髖→膝) | shoulder, hip, knee | >155° | 100-150° | <145° | 不定 |
| **膝關節角度** (髖→膝→踝) | hip, knee, ankle | >155° | — | <145° | 不定 |
| **軀幹角度** (肩→髖→垂直線) | shoulder, hip | 0-15° | <35° | >10° | >60° |

> **3/23 Jetson 實測角度範圍**（MediaPipe Pose, D435 640x480@15Hz）：
> - 站: hip=168-180°, knee=164-180°, trunk=1-10°
> - 坐: hip=91-129°, knee=69-132°, trunk=2-22°
> - 蹲: hip=36-65°, knee=32-57°, trunk=5-17°

### 方法二：高度比法（跌倒偵測首選）

| 特徵 | 計算方式 | 閾值 |
|------|---------|------|
| Bounding Box 寬高比 | width / height | >1.0 → 疑似跌倒 |
| 肩-髖高度差 | abs(shoulder_y - hip_y) / body_height | <0.3 → 疑似跌倒 |
| 髖部下降速度 | delta_hip_y / delta_t | 突然下降 → 觸發 |

### 方法三：混合法（實際實作方式）

```python
# pose_classifier.py, 3/22 更新，6 類 first-match
def classify_pose(body_kps, body_scores, bbox_ratio):
    hip_angle = angle(shoulder, hip, knee)
    knee_angle = angle(hip, knee, ankle)
    trunk_angle = trunk_angle(shoulder, hip)

    # 1. 跌倒優先判斷（安全功能）
    if bbox_ratio > 1.0 and trunk_angle > 60:
        return "fallen"
    # 2. 站立（3/23 調參：160→155）
    if hip_angle > 155 and knee_angle > 155:
        return "standing"
    # 3. 彎腰（3/22 新增）
    if trunk_angle > 35 and hip_angle < 140 and knee_angle > 130 and bbox_ratio <= 1.0:
        return "bending"
    # 4. 蹲下（3/23 調參）
    if hip_angle < 145 and knee_angle < 145 and trunk_angle > 10:
        return "crouching"
    # 5. 坐下（3/23 調參）
    if 100 < hip_angle < 150 and trunk_angle < 35:
        return "sitting"
    # 6. 模糊
    return None

    # 投票機制：pose_vote_frames=20（連續 20 幀多數決）
    # confidence = 投票比例（非模型信心度）
```

### 跌倒偵測精度（研究數據）

| 方法 | 精度 | 召回率 | 來源 |
|------|:----:|:-----:|------|
| MediaPipe + 20-frame voting | 86.9% | 93.8% | arxiv 2503.19501 |

## ROS2 整合

### Node 設計

```
vision_perception_node（統一入口）
  ├── Subscribe: /camera/color/image_raw (sensor_msgs/Image)
  ├── MediaPipe Pose 推理
  ├── pose_classifier 規則分類
  ├── Publish: /event/pose_detected (std_msgs/String, JSON)         ← v2.0 凍結
  └── Publish: /state/perception/pose (std_msgs/String, JSON)      ← v2.1 內部
```

### Event Schema（v2.0 凍結）

```json
{
  "stamp":       1710000000.123,
  "event_type":  "pose_detected",
  "pose":        "standing",
  "confidence":  0.92,
  "track_id":    1
}
```

**觸發規則**：
- 姿勢**變化**時發布事件
- `fallen` 事件**立即發布**（不等投票完成，但加 low-confidence flag）
- 穩定 20 幀後才發布確認事件

### event_action_bridge 姿勢→動作映射

| 姿勢 | Go2 動作 | TTS | Cooldown |
|------|---------|-----|:--------:|
| `fallen` | — | "偵測到跌倒！請注意安全" | 10s |

> 其他姿勢目前不觸發 Go2 動作。

## 2D Pose vs 3D Pose

### 為什麼本專案選 2D？

| 維度 | 2D Pose | 3D Pose |
|------|---------|---------|
| 輸出 | (x, y, confidence) per keypoint | (x, y, z) 世界座標 |
| 代表方案 | DWPose / RTMPose / trt_pose | MediaPipe Pose (model_complexity=2)、MotionBERT |
| Jetson 可行性 | OK | 額外 +300-500MB |
| 跌倒偵測能力 | bbox 寬高比 + 關節角度 + 投票 + depth 輔助 | z 軸高度驟降，理論更精準但成本高 |

**結論**：純 2D 在 5 姿勢分類已足夠。D435 depth 可做 pseudo-3D 跌倒偵測。

## D435 深度輔助

### 深度在姿勢辨識的 3 個用途

1. **距離 gate**：只處理 0.5-4m 內的人體
2. **跌倒驗證**：hip keypoint depth 突然變化，配合 bbox ratio + trunk angle 做多訊號交叉驗證
3. **Pseudo-3D 高度估計**：`real_height ~ pixel_height * depth / focal_length`

### 不做的事

- 不做完整 point cloud 重建
- 不上傳 depth map 到雲端
- 不依賴單點 depth（用 ROI median）

## 社群實戰回饋（2026-03-17 調查）

### 坑 1：TensorRT 轉換精度下降

- RTMPose ONNX→TRT 後 keypoints 錯位
- FP16 Transformer-like 模型 mAP 暴跌
- INT8 不校準直接崩

### 坑 2：D435 深度對齊與空洞

- ~30% pixel 回傳 0mm，尤其在物件邊緣
- 對策：IR emitter + spatial/temporal filter + ROI 5x5 median

### 坑 3：跌倒偵測誤判

| 誤判場景 | 對策 |
|---------|------|
| 老人緩慢坐下 | 時序分析：跌倒 <1s，坐下 >2s |
| 彎腰撿東西 | 20 幀投票 buffer |
| 躺在沙發/床上 | D435 深度區分高度 |
| 相機角度偏斜 | 固定安裝角度 15-30° 俯角 |

> **核心原則：任何單一訊號都不作最終跌倒判定。** 必須多訊號投票。

### 坑 4：GPU 多任務共存

- 兩個 TRT 模型並行 ~5% 性能降級
- 同 frame 不可真並行

### 坑 5：DWPose 特有問題

- 遮擋 >50% 時手部 AP 降至 45%
- bbox 截斷肢體末端
- TRT engine 首幀延遲 5-10s

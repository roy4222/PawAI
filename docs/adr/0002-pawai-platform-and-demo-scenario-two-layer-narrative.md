# ADR-0002: PawAI 採平台身份與 2026-06 demo 場景的雙層敘事

- **Date**: 2026-05-23
- **Status**: accepted

## Context

PawAI 定位過去三個月反覆搖擺：

- **5/19 v1 北極星**：居家四足機器人 + 互動 70% / 守護 30%
- **5/21 教授 pivot → 5/22 v2 amend**：機構場域巡檢與互動助理 POC（居家完全 reject）
- **5/22 grill 第二輪**：使用者反饋「把它當作居家四足機器人比較合理 當寵物狗太浪費」+「希望整個專案是一體 不要太分離」

如果以單一定位 statement 回答這個張力，會撕掉前述任一決議：
- 純機構巡檢 → 失去「整體性」對外敘事掛點 + 未來居家延伸無路
- 純居家四足機器人 → 撕教授 5/21 pivot + 三場 review synthesis §2.1
- 不選 → 多 agent / 接手者沒共通語言，每次討論都從頭辯

替代方案考量過：(a) 雙層敘事（平台身份 + demo 場景分層）；(b) 完全 pivot 回居家四足機器人；(c) v2 不動，新訊號只是個人 framing。Hiwonder ROSOrin Pro 採同一手法（平台 = 教育機器狗、場景 = 任意 demo）佐證 (a) 為業界慣例。

## Decision

PawAI 對外敘事採**雙層**結構，明確區分「平台能力」與「短期驗收場景」：

| 層 | 內容 |
|---|---|
| **平台身份** | 通用居家 / 機構四足互動機器人。能力涵蓋移動、多模態感知、語音對話、安全仲裁、可擴充技能組合 |
| **2026-06 demo 場景** | 非接觸式機構巡檢助理（詳 ADR-0001）。場景限定長照 / 日照 / 養護機構，6/18 驗收以「< 100㎡ 已知小型室內」為 POC 邊界 |

**這不是定位搖擺，是分層**：
- 平台身份是「我們做出什麼」
- demo 場景是「我們在哪驗證」
- 未來不同場景採購方（校園、商場、居家、導盲、救災等）對平台的需求不同，由他們延伸；2026-06 PawAI 團隊只負責驗證機構巡檢這條

對外敘事範式：

> PawAI 是通用居家 / 機構四足互動機器人。2026-06 我們以非接觸式機構巡檢助理場景做 Physical AI 原型驗證。整套架構已為下列方向預留延伸：校園巡邏、導盲輔助、居家延伸、戶外救災。

## Consequences

**正面**：

- 教授 5/21 pivot 與使用者「整體性」訴求同時被滿足，不互相撕毀
- 影片末段未來情境（校園 → 導盲 → 救災 → 居家延伸）有合理敘事掛點，不像強行加塞
- 多 agent / 接手者透過 CONTEXT.md → ADR-0002 立刻明白「為什麼定位看起來分裂」
- 未來真要 pivot 到別場景，只需開新 ADR 換 demo 場景層，不必撕平台身份

**負面**：

- 雙層敘事比單一 statement 更難用一句話講清楚，簡報 / 影片旁白要練
- 「平台 vs 場景」邊界對非工程聽眾（家長 / 採購方）可能仍嫌技術腔
- v2 spec §1 主 statement 不重寫，只加 footnote — 未來人讀 spec 可能漏看 footnote 而誤解為純機構定位

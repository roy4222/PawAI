# 第五章 系統限制與可行性分析

本章說明本系統在開發過程中遭遇的技術限制、設計取捨與未能完整實現的功能,並針對各項限制提出可行性分析與替代方案。部分限制源於硬體物理特性(軟體無法克服)、部分為 Jetson 8 GB 統一記憶體下的資源競爭、部分為開發時程不足而主動排除、另有部分為開發過程中實際嘗試克服但最終未達可用門檻。

---

## 5-1 硬體平台限制

### 5-1-1 Go2 Pro 內建 LiDAR 資料暴露不足

Go2 Pro 內建的 Unitree 自研 4D LiDAR L2(360° × 96° 半球型掃描)在硬體規格上具備基礎導航能力,但 Go2 Pro 韌體將 LiDAR 資料透過 WebRTC DataChannel 對外暴露時,會先經過 voxel 體素編碼壓縮,再由 Jetson 端的 Python 驅動節點解碼還原為點雲。此解碼後的資料實測有效掃描點覆蓋率僅約 **18%**(每幀約 22 / 120 個有效點)。

發布頻率方面,2026 年 2 至 3 月的初步實測曾觀測到極不穩定的 0.03 至 2 Hz;2026 年 4 月 1 日重新測試(`docs/navigation/research/2026-04-01-lidar-frequency-retest.md`)後修正為靜止約 **7.3 Hz**、16 節點全跑約 **7.3 Hz**、行走時約 **5.0 至 6.1 Hz**,推翻先前的悲觀結論。頻率本身並非瓶頸,但每幀僅 22 個有效點遠不足以建構穩定的佔據格(Occupancy Grid)。

此為 Go2 Pro 韌體層級對 LiDAR 資料的封裝選擇——Go2 EDU 版本才提供透過 CycloneDDS 取得完整 LiDAR 資料的介面。Pro 版本的使用者無法透過軟體繞過此限制。

**可行性評估**:團隊已決定採購外接 SLAMTEC RPLIDAR A2M12(每圈 1,600 點、USB 直連 Jetson),可完全繞過 WebRTC 解碼瓶頸。到貨後的可行性取決於 Go2 四足里程計(odom)的漂移幅度。團隊預設的 go/no-go 門檻為:若 10 秒內漂移超過 0.3 公尺則判定 SLAM 方案不可行,退而使用 360° 反應式避障。

### 5-1-2 D435 安裝角度導致避障失敗

Intel RealSense D435 安裝於 Go2 Pro 機身前方上緣,鏡頭角度略為偏上方。低矮障礙物(椅腳、門檻、散落於地面的小型物品)要到距離約 0.4 公尺以內才會進入深度視野。加上從偵測到完全停止的延遲鏈過長:

| 延遲環節 | 耗時 |
|---|---|
| 軟體 debounce 去抖動 | ~100 ms |
| ROS2 rate limiter | ~200 ms |
| WebRTC DataChannel 通訊延遲 | ~300 ms |
| Go2 減速至完全停止 | 500-1000 ms |
| **總反應時間** | **~1.0-1.5 秒** |

三輪真機防撞測試(2026 年 4 月 3 日)均未通過——機器狗在偵測到障礙後仍因慣性撞上目標物。此為鏡頭安裝角度的物理限制結合 WebRTC 通訊延遲的疊加效應,軟體無法克服。D435 深度避障於該日正式停用。

**可行性評估**:若將 D435 改以下傾角安裝可改善低矮障礙物的偵測時機,但會犧牲正前方遠距離(2-5 公尺)的人臉、手勢、姿勢辨識範圍。在「互動 70% / 守護 30%」的產品定位下,D435 的安裝角度必須優先服務互動感知,避障功能改由外接 RPLIDAR 承擔。

### 5-1-3 Jetson 外接供電不穩定

Jetson Orin Nano SUPER 的供電依賴外部 XL4015 降壓模組從 Go2 電池取電(28.8V 降壓至 19V)。穩態負載約 10W 時運作正常,但多模組同時推理(人臉+手勢+姿勢+物體+語音)時尖峰功耗可達 15-18W,加上 Go2 行走過程中的震動,Sprint 期間累計發生 **8 次以上** Jetson 因電壓下降而強制斷電。

團隊嘗試將輸出電壓自 18.8V 往上調整——專案文件主線記錄為調至 19.2V,後續實際運作中亦有調到 20V(Jetson DC 輸入安全極限)的情形。調高後稍有改善但未根治,且 20V 已無法再上調,否則有損毀 Jetson 風險。

**可行性評估**:長期解決需升級降壓模組至更高電流容量的 DC-DC buck 轉換器(如 LM2596HV 或 MP1584EN),或為 Jetson 配置獨立的 18650 電池組(額外重量約 200g,Go2 Pro 最大負重 8kg,餘量充足)。Demo 期間採人工監控供電狀態,配置備援電源。

### 5-1-4 Go2 機身麥克風訊噪比嚴重不足

Go2 內建散熱風扇在機器狗運作期間持續產生寬頻噪音(頻譜覆蓋 200Hz-8kHz),機身 USB 麥克風與風扇出風口距離僅約 15 公分。實測 SNR 數據:

| 條件 | ASR 正確率 |
|---|:---:|
| 安靜環境(風扇關閉) | ~80% |
| Go2 待機(風扇低轉速) | ~45% |
| Go2 行走(風扇+馬達噪音) | **~20%** |

團隊嘗試調高麥克風增益(mic_gain 1.0→4.0→8.0→12.0),但噪音同步放大,gain 12 時 Whisper 辨識率反降至 43%。8.0 為實測最佳平衡點,但仍不足。亦嘗試 Energy VAD 閾值調整、Whisper 幻覺黑名單(22 條)等軟體手段,皆無法克服物理層級的 SNR 劣化。

**可行性評估**:短期已改用 PawAI Studio push-to-talk(筆電收音),犧牲直覺性換取可靠度。長期需改用指向性麥克風陣列或主動降噪演算法(RNNoise、DeepFilterNet),或將麥克風移至遠離風扇的延伸支架上。

### 5-1-5 Go2 Megaphone 音訊播放品質限制

Go2 機身喇叭透過 Megaphone DataChannel API(api_id 4001-4003)播放音訊時,硬體強制限定 **16 kHz 採樣率**。TTS 合成的音訊(edge-tts 24kHz/Piper 22050Hz)必須降採樣至 16kHz,高頻資訊損失導致播放清晰度明顯下降。

此外,Megaphone API 存在一個未記載的行為——mid-session 重啟 `tts_node` 後,Go2 的 Megaphone 狀態機未正確重置,導致後續所有音訊上傳均 **silent fail**(無錯誤訊息、資料送出但無聲音)。唯一的恢復方式是重啟 Go2 driver 甚至 Go2 本體。

**可行性評估**:外接 USB 喇叭(Jieli CD002-AUDIO, 48kHz)已完全繞過 Megaphone 限制,保留原生取樣率,清晰度大幅改善。Megaphone 路徑降為備援。

## 5-2 感知模組邊界條件

### 5-2-1 光線敏感性

所有視覺感知模型(YuNet 人臉偵測、SFace 人臉識別、MediaPipe Pose 姿勢、MediaPipe Gesture Recognizer 手勢、YOLO26n 物體)在低光源環境下精度顯著下降。開發過程中的觀察:

- **人臉偵測(YuNet)**:低光源下偵測信心度明顯下降,偶發漏偵測與假陽性(將窗簾陰影、電器圖案誤認為人臉)
- **手部關鍵點(MediaPipe)**:光線不足時手指關節點抖動加劇,側光或背光場景下偵測率顯著降低
- **物體偵測(YOLO26n)**:小物件(水瓶、手機、書本)在低光源下幾乎無法穩定偵測,受 Nano 模型本身 mAP 40.9% 的限制疊加光線因素影響更大
- **D435 深度**:深度圖破洞增加,高反光或深色吸光表面的深度資訊更不穩定

上述為開發期間的定性觀察,本專題未針對不同照度條件進行系統性的定量 A/B 測試,因此不列出精確的降幅百分比。

D435 的紅外線投射器在直射陽光下會被背景紅外線淹沒,深度感測率顯著下降,此為本專題將展示場景限定於室內的技術理由之一。

**Demo 對策**:場地預先確認照明條件(≥300 lux),必要時攜帶補光燈;Demo 前執行 preflight check 實測各模組在該場地的偵測率。

### 5-2-2 有效偵測距離

各感知模組因模型訓練資料與 D435 光學特性,存在各自的有效距離區間:

| 模組 | 最近 | 最遠 | 最佳區間 | 瓶頸 |
|---|:---:|:---:|:---:|---|
| 手勢(MediaPipe) | 0.5m | ~2m | 1-1.5m | 手部像素量不足 |
| 姿勢(MediaPipe) | 1.5m | ~5m | 2-3m | 過近超視野/過遠精度降 |
| 人臉(YuNet) | 0.3m | ~5m | 1-3m | 過遠信心度降至閾值以下 |
| 物體(YOLO26n) | 0.3m | ~5m | 1-3m | 小物件像素不足 |
| D435 深度 | 0.2m | ~10m | 0.3-3m | 3m 後精度快速下降 |

手勢的 ~2m 限制是系統最嚴格的距離約束——Demo 時使用者必須站在 D435 前方約 1-2 公尺內才能讓手勢辨識穩定工作,而此距離下人臉、姿勢、物體辨識均在最佳區間內,因此以手勢的 2m 作為統一的互動距離建議。

### 5-2-3 人臉追蹤抖動

即使加入 Hysteresis 雙閾值穩定化(`sim_threshold_upper=0.30` / `lower=0.22`)與 IOU 多目標追蹤(`track_iou_threshold=0.15`),實測 2 分鐘 smoke test 中仍產生約 **45 個 track** 被建立(目標小於 5)。根因是 YuNet 偵測器的偵測框在連續幀間存在像素級微抖(即使人臉靜止不動,邊界框座標仍有 2-5 pixel 的幀間變異),當微抖幅度超過 IOU 匹配閾值時,系統會判定為「舊 track 消失 + 新 track 出現」,產生大量 `track_started` / `track_lost` 事件。

**可行性評估**:可透過 Kalman Filter 預測偵測框位置來消除微抖,或改用基於特徵向量(embedding)的追蹤關聯取代純 IOU,但需額外的開發與測試時間,列為下一階段優化項目。

### 5-2-4 跌倒偵測的誤判與殘存風險

姿勢辨識的跌倒判定已加入 `vertical_ratio < 0.4` 防護條件與 `pose_vote_frames=20`(約 1-2 秒)的多幀投票機制,顯著降低了正面站姿被誤判為跌倒的情形。然而,在以下場景仍可能出現 **fallen 幻覺**:

- 背景中的非人物體形似倒地人體(倒下的掃把、橫放的抱枕、晾在椅背上的衣物)
- 光線不佳時 BlazePose 偵測器將非人區域誤認為人體
- 使用者彎腰持續超過 2 秒(bending 與 fallen 的幾何特徵有重疊區間)

**Demo 對策**:Demo 啟動腳本(`start_full_demo_tmux.sh:137`)以 `enable_fallen:=false` 暫時關閉跌倒偵測,避免互動 Demo 被誤觸發打斷;跌倒偵測作為守護功能在獨立展示環節啟用,搭配預先淨化的場景(移除可能被誤認的物品)。

### 5-2-5 物體辨識的 Nano 模型精度限制

YOLO26n(Nano 版本)的 COCO mAP 僅 **40.9%**,對小物件的偵測率本身就偏低。實機驗收(2026 年 4 月 6 日):

| 物品 | 結果 | 條件 |
|---|:---:|---|
| cup(水杯) | 通過 | threshold 0.5,觸發 TTS |
| cell_phone(手機) | 通過 | 適當光線條件 |
| book(書本) | 部分通過 | 平放困難,翻開展示可辨識 |
| bottle(水瓶) | **未通過** | 深色瓶身,Demo 不展示 |

**可行性評估**:升級至 YOLO26s(mAP 48.6%, +7.7pp)是最直接的改善方案,模型約 20MB(YOLO26n 的 2 倍),FLOPs 約 4 倍。需重新評估 Jetson TensorRT FP16 下的 GPU/RAM 預算。此為繳交後的第一優先改進項目。

## 5-3 語音管線限制

### 5-3-1 LLM 回覆過短與缺乏個性

受限於 `llm_max_tokens=80`(`llm_bridge_node.py:165`)與 SYSTEM_PROMPT 中的 **12 字硬截斷**(`llm_bridge_node.py:79`),LLM 無法展開詳細對話或多輪推理,回覆如「好的,我停下」「你好,很高興見到你」等短句,個性化表達空間極度有限。

此限制的設計理由是:語音合成延遲與回覆長度成正比(edge-tts 每多 10 字約增加 200ms),過長回覆會嚴重拖沓互動節奏。但 12 字限制過於保守,導致機器狗的語言能力遠低於使用者對「AI 對話」的期待。

**可行性評估**:可將截斷字數放寬至 30-50 字、搭配 TTS streaming(LLM 生成與 TTS 合成管線化),預估能在延遲增加 500ms 的代價下顯著提升對話品質。但 TTS streaming 架構改動大(需改 tts_node 為 chunk-based 播放),列為未來工作。

### 5-3-2 無多輪對話記憶

當前 LLM 呼叫為無狀態模式,每次對話獨立送入雲端伺服器,不攜帶歷史上下文。使用者說「我剛才說的那個」機器狗完全無法理解。

團隊評估過 Mem0(per-person 長期記憶框架,部署於 RTX 8000)、LangChain ConversationBufferMemory、自建 SQLite 對話日誌等方案。主要障礙並非技術不可行,而是整合、測試與真機驗證的時程不足——記憶機制涉及 LLM prompt 格式改動、Gateway 與 stt_intent_node 雙路徑的同步適配、以及 face_identity 與對話記憶的 person_id 綁定,跨模組改動量超出本次 Sprint 可承擔範圍。

### 5-3-3 本地 LLM 備援品質不足

Jetson 本地備援的 Qwen2.5-1.5B(Q4_K_M 量化, ~1.12GB, CPU 推理)在 2026 年 4 月 8 日教授會議上正式確認中文對話品質遠低於雲端 7B 主線:回覆經常離題、JSON 格式輸出不穩定、語意理解有限。實務上只能作為「維持管線不中斷」的形式備援,無法承擔主要對話責任。

更小的 Qwen2.5-0.5B(RAM 增量僅 139MB)雖然速度更快(P50 延遲 0.8s),但中文對話品質更加不足,已被排除。

**可行性評估**:本地 LLM 品質受限於 Jetson 8GB 記憶體與 CPU 推理速度。短期內無法在邊緣端達到與雲端 7B 相當的對話品質。可行的中間方案是使用 Groq 等免費雲端 API(Llama 3.3 70B, 30 RPM)作為第二層雲端備援,但仍需網路連線。

### 5-3-4 Whisper 幻覺問題

Whisper 模型(三級 fallback 最末層)存在已知的「幻覺」(Hallucination)問題——在靜音片段或低 SNR 片段會輸出假文字,最常見的是訓練資料中的字幕模板(「字幕 by 索兰娅」「請訂閱頻道」「感謝觀看」等)。本系統的對策:

- 啟用 `vad_filter=True`(Silero VAD 預先過濾非語音段)
- `no_speech_threshold=0.6` 拒絕低信心結果
- `log_prob_threshold=-1.0` 過濾低機率輸出
- **22 條幻覺黑名單字串**逐一比對(`stt_intent_node.py:345-368`)
- 過濾所有少於 2 字的結果

即使如此,25 筆 A/B 測試中 Whisper Tiny 仍有 **8% 幻覺率**(2/25 筆輸出假文字)。此為 Whisper 在低 SNR 中文短句場景下的固有缺陷,也是推動從 Whisper 到 SenseVoice(0% 幻覺率)的主線切換決策的關鍵因素。

## 5-4 多人同時互動限制

### 5-4-1 跨模組 person_id 關聯缺失

人臉辨識模組的 IOU 追蹤可支援最多 5 人同時追蹤並維持獨立 `track_id` 與身份名稱;然而手勢辨識(MediaPipe Gesture Recognizer)與姿勢辨識(MediaPipe Pose)目前僅支援單人推理。更關鍵的是,三個感知模組之間缺乏統一的 `person_id` 關聯機制——當多人同時出現時,系統無法判斷某個手勢或姿勢歸屬於哪位已辨識的使用者。

| 模組 | 多人支援 | person_id | 備註 |
|---|:---:|:---:|---|
| 人臉(YuNet+SFace) | 最多 5 人 | track_id(模組內部) | 可辨識「誰是誰」 |
| 手勢(MediaPipe) | 雙手,無跨人 | 無 | 無法判斷「誰的手」 |
| 姿勢(MediaPipe) | 單人 | 無 | 只處理最顯著人員 |
| 物體(YOLO26n) | N/A | N/A | 物體無需綁定人員 |

**可行性評估**:完整的多人互動支援需重新設計跨模組的 person_id 關聯架構——以人臉 track_id 為錨點,將手勢與姿勢的偵測結果透過空間位置匹配(bounding box overlap / depth proximity)綁定至對應身份,並在中控層做跨模態融合。預估工作量超過一個完整 Sprint(2 週),無法納入本次開發時程,列為未來工作。

現階段實務上近似採用「最靠近」原則——由於 MediaPipe 預設處理畫面中最顯著的人員,手勢與姿勢辨識的實際對象傾向於距離 D435 最近且最顯眼的使用者,但此行為並非程式碼中的硬性規則,而是 MediaPipe 模型內部的隱式選擇。

## 5-5 自主導航避障限制

### 5-5-1 三重障礙疊加

本專題的導航避障功能面臨三重獨立障礙,每一重都可能單獨阻斷 SLAM 方案:

1. **Go2 內建 LiDAR 覆蓋率不足**(18%,見 5-1-1)→ 需外接 RPLIDAR
2. **D435 深度避障安裝角度限制**(見 5-1-2)→ 已停用
3. **Go2 四足里程計漂移風險**→ 未驗證,依賴 RPLIDAR 到貨

Go2 Pro 作為四足機器人,其里程計(odom)來自足部關節編碼器與 IMU 的融合估計。四足步態的固有特性使得每一步接地/離地都會引入位置誤差,且誤差隨時間累積。團隊預設的 go/no-go 驗收門檻為:若 10 秒內漂移超過 0.3 公尺(約 Go2 體長的 40%),則判定 `slam_toolbox` 的 loop closure 無法有效修正,SLAM 建圖不可行。此門檻為規劃判準,尚未經實測驗證。

**可行性評估**:雷達到貨首日執行 bag 錄製驗證流程(靜止→直線→轉彎→回到原點),量測 odom 漂移幅度。若通過,5 天整合時程(驅動驗證→slam_toolbox 建圖→Nav2 路徑規劃→實機防撞);若失敗,退守 360° 反應式避障(nav2_collision_monitor,不依賴 SLAM 建圖)。

### 5-5-2 Jetson 資源預算衝突

導航堆疊(`rplidar_ros2` + `slam_toolbox` + Nav2)預估 CPU 佔用約 70%、RAM 增量約 0.85-1.15 GB。在 Jetson 已運行五大感知模組的條件下(總 CPU ~215%/600%、RAM ~4-5.5GB/8GB):

| 項目 | 導航前 | 導航中 | 風險 |
|---|:---:|:---:|---|
| CPU(六核 600%) | ~215% | ~285% | 中(仍有餘量) |
| RAM | ~4-5.5 GB | ~5.5-6.7 GB | 高(僅餘 1.3-2.5 GB) |
| GPU | YOLO26n 獨佔 | 不變 | 低 |

若 RAM 壓力過大,需在導航進行時暫時關閉手勢辨識模組(釋放 ~200 MB + 45% CPU)作為資源讓步。

## 5-6 Jetson 資源競爭與記憶體預算

### 5-6-1 8 GB 統一記憶體的雙面刃

Jetson Orin Nano 的 CPU 與 GPU 共享同一個 8 GB LPDDR5 記憶體空間。優勢是資料不需在 CPU RAM 與 GPU VRAM 間複製;劣勢是任何一方的記憶體洩漏都會擠壓另一方。

本專題的設計原則為:**GPU 獨佔給 YOLO26n 的 TensorRT FP16 推理,所有其他感知模組走 CPU**。此原則的由來是 RTMPose(GPU 姿勢辨識)在 Jetson 上實測 GPU 佔用率 91-99%,無法與 YOLO26n 共存;SCRFD(GPU 人臉偵測)會使姿勢 FPS 額外下降 10%。因此選擇 YuNet(CPU)、MediaPipe(CPU)作為人臉與手勢/姿勢的主線方案。

| 模組 | RAM | CPU | GPU | 說明 |
|---|:---:|:---:|:---:|---|
| ROS2 + Go2 driver | 1.5-2.0 GB | ~20% | 0% | 基礎通訊層 |
| D435 camera | 0.6-1.0 GB | ~10% | 0% | RGB+Depth 串流 |
| 人臉(YuNet+SFace) | ~0.3 GB | ~40% | 0% | CPU-only |
| 手勢(MediaPipe Recognizer) | ~0.2 GB | ~45% | 0% | CPU-only |
| 姿勢(MediaPipe Pose Lite) | ~0.25 GB | ~50% | 0% | CPU-only |
| 物體(YOLO26n TRT) | ~1.0 GB | ~40% | 使用中 | TensorRT FP16 |
| 語音(edge-tts+Cloud ASR) | ~0.1 GB | ~10% | 0% | 雲端主線 |
| **總計** | **~4.0-5.5 GB** | **~215%** | 部分 | 餘 2.5-4 GB |

以上數據取自 Demo 部署主線(`start_full_demo_tmux.sh`)的覆寫配置。餘量 2.5-4 GB 看似充足,但 Linux kernel + systemd 服務約佔 1.5 GB,實際可用餘量約 1-2.5 GB。任何新增模組(導航堆疊、本地 LLM、記憶服務)都需嚴格評估 RAM 衝擊。

### 5-6-2 ARM 生態碎片化

Jetson 為 ARM64 架構,部分 Python 套件缺乏預編譯 ARM wheel 或與 JetPack 6 的 CUDA 版本不相容:

- `ultralytics` 官方套件會強制安裝 x86 版 PyTorch,破壞 JetPack 預裝的 ARM 最佳化版本(實際發生於 2026-04-04,環境救援耗時半天)
- `mediapipe` 無官方 ARM64 wheel,需使用社群編譯版本
- `onnxruntime-gpu==1.23.0` 需指定 Jetson AI Lab 的特殊 pip index URL(`https://pypi.jetson-ai-lab.io/jp6/cu126`)
- `rtmlib` 必須 `pip install --no-deps` 避免拉回 CPU 版 onnxruntime 覆蓋 GPU 版

## 5-7 WebRTC 通訊協議限制

### 5-7-1 延遲與可靠性

Go2 Pro 的所有外部通訊(動作控制、感測器資料、音訊播放)皆透過 WebRTC DataChannel,平均延遲約 **100-300 ms**,不適合對即時性要求極高的運動控制(Go2 EDU 透過 CycloneDDS 有線連線可達微秒級)。本專題的互動式應用(語音對話延遲約數秒、動作觸發延遲低於 1 秒即可接受)尚在可容忍範圍,但導航避障的緊急停止就不夠快(見 5-1-2)。

### 5-7-2 Megaphone API 的隱性格式要求

Go2 的 Megaphone 音訊上傳 API(api_id 4001-4003)有多個**未記載**的格式要求,團隊花費約兩週逆向工程才完整釐清:

- chunk_size 必須為 **4096** base64 字元(非 bytes)
- payload 必須包含 `current_block_size` 欄位(官方文件未載明)
- msg type 必須為 `"req"`(非直覺的 `"msg"`)
- EXIT(4002) 後必須 sleep 0.5 秒(cooldown),否則下次 ENTER(4001) silent fail
- mid-session 重啟 tts_node 會導致 Megaphone 狀態機卡死(見 5-1-5)

### 5-7-3 多 Driver Instance 殘留

`ros2 launch` 啟動 `go2_robot_sdk` 後,`killall python3` 只殺 launch parent,C++ 子 process(robot_state_publisher、pointcloud_to_laserscan、joy_node 等)會殘留。下次 launch 會再生一組,導致多 instance 搶 WebRTC 連線和 ROS2 topic,表現為指令有時生效有時無效。必須逐一 `pkill -9 go2_driver; pkill -9 robot_state; pkill -9 pointcloud; pkill -9 joy_node; pkill -9 teleop; pkill -9 twist_mux` 才能徹底清除。

## 5-8 多語言支援限制

本系統的語音辨識(SenseVoice / Whisper)、語音合成(edge-tts / Piper)與大型語言模型(Qwen2.5-7B)雖多數原生支援多語言,但各環節表現不一:SenseVoice 優勢在中文與亞洲語言;edge-tts 的中文與英文聲音風格不一致;LLM 的 system prompt、intent rules、RuleBrain 皆以繁體中文撰寫;台語、客語等方言缺乏高品質開源模型。

本次開發明確限定繁體中文,不做多語言擴充。

## 5-9 資料隱私與安全

- **人臉資料無加密**:SFace 128 維特徵向量與姓名以明文 pickle 儲存於 `/home/jetson/face_db/`,無靜態加密或存取權限控制
- **語音雲端傳輸**:使用者語音透過 SSH tunnel 傳至遠端 RTX 8000 伺服器,雖為私有網段但未實作端對端加密
- **無帳號與角色系統**:face_db 僅記錄姓名,無「管理者/一般成員/訪客」角色分級
- **無持久化稽核日誌**:系統事件僅短期保留於 ROS2 topic 緩衝區,無事後追溯能力

**緩解措施**:所有視覺感知在 Jetson 本地運行,不將影像上傳;人臉資料庫儲存於 Jetson 而非雲端;Demo 期間所有資料僅在區域網路內流動。正式產品化需補足加密、帳號、角色權限與稽核日誌。

## 5-10 遠端存取與推播通知

- PawAI Studio 僅支援區域網路存取,無雲端 relay 或 NAT 穿透
- 無帳號身份驗證機制(OAuth、JWT)
- 無原生行動端 App(iOS / Android)
- 跌倒、陌生人警戒等守護事件僅顯示於 Studio 面板,未推送至外部裝置(Apple Push Notification Service、Firebase Cloud Messaging)

**可行性評估**:技術上可透過 Tailscale VPN 從外部存取 Studio;推播通知可透過 LINE Notify 或 Telegram Bot API 以極低成本實現(10 行 Python)。但完整的遠端存取+身份驗證+行動端 App 工作量遠超本次範圍。

## 5-11 情緒辨識與主動關懷

本系統未整合面部情緒辨識(FER)或語音語調分析模型。情緒辨識面臨:

- 開源 FER 模型在東亞面孔的準確度有限,跨文化泛化能力差
- 「微表情」與「中性表情」的分界模糊,誤判率高
- 過度解讀使用者情緒可能引發隱私疑慮,尤其針對長輩或兒童
- 需融合面部、語音、姿勢等多模態資訊,系統整合複雜度高

目前透過對話關鍵字間接推測使用者狀態(例如提到「累」觸發關懷回應),屬淺層規則判斷。

## 5-13 開發時程與工程債務

### 5-13-1 文件與實作同步差距

PawAI Brain 三層架構(Safety → Policy → Expression)的完整設計規格(`docs/archive/2026-05-docs-reorg/superpowers-legacy/specs/2026-04-11-pawai-home-interaction-design.md`)已更新至最新版本,但實作仍由較輕量的 `interaction_executive` v0 狀態機承擔。設計規格中的 Skill Queue(以 ROS2 Action 為底層)、完整的 Safety Layer 仲裁邏輯、Expression Layer 的情緒表達管線等均尚未落地,interaction_contract.md 中標記為 `planned / not yet implemented` 的 topic 包括 `/state/executive/brain` 等。

### 5-13-2 整合測試覆蓋率

目前 CI pipeline 已有多組單元測試與介面契約檢查,涵蓋各核心模組(speech_processor、face_perception、vision_perception、interaction_executive、object_perception)的關鍵邏輯。然而端到端場景測試(五大感知模組同時運行→中控決策→Go2 動作執行)仍以手動 Demo 驗證為主,跨模組修改後需人工回歸驗證。

### 5-13-3 Plan B 固定台詞備案

為確保 Demo 穩定,團隊針對極端場景(GPU 伺服器斷線、雲端 API 限流、網路中斷)設計固定台詞備案(約 15 組問答),ASR 辨識出意圖後直接匹配預設回答,繞過 LLM 動態生成。回應速度小於 1 秒但無法提供完整的 AI 對話體驗。

### 5-13-4 供電穩定性為最大 Demo 風險

綜合各項風險,Jetson 外接供電的不穩定性(Sprint 期間 8+ 次斷電)是 Demo 當天最大的單點失敗風險。團隊採取的緩解措施包括:降低 Demo 時同時運行的模組數量(非展示模組暫時關閉)、配置備援電源、安排專人即時監控溫度與電壓。
